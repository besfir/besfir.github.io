Why is it recommended to run only one process in a container?

Besfir · May 31, 2023
Table of Contents
  • Benefits of running a single service per container
  • Disadvantages of running multi-service per container
  • Conclusion
  • Reference

It’s best practice to separate areas of concern by using one service per container.

Docker documentation

In the Docker document recommends the use of one service per container for separation of areas of interest as a best practice. In addition, many documents online mention this a lot, so why encourage the use of one service per container?

Benefits of running a single service per container

  • Scalability: It makes it much easier to scale specific services as needed.
    If multi-services are operated in one container, even the same service may require different settings for each container when horizontally expanding. In this case, it may be more complicated than when the docker is not used.
  • Reusability: The service becomes much easier to reuse in other projects that require it.
    If multi-services were run in one container, they would not be available in other projects because they are specialized in the project.
  • Portability and Predictability: Since the scope of work reduced, it becomes much easier to debug security patches or errors.
    If multi-services are operated in one container, other non-patch-related services in the container may also be affected when applying security patches due to the high degree of combination between services. Therefore, it may be necessary to perform tasks that would not have been necessary for a single service per container configuration. This means that portability is low, and predictability is also low because it is relatively difficult to predict the range of effects of errors.

Disadvantages of running multi-service per container

  • Log processing: If multi-services are run in one container, it is difficult to identify which service is output from. Therefore, additional settings should be made to enable identification, or logs should be redirected to a separate file.
  • Monitoring: Docker performs container health checks with PID 1. Therefore, if PID 1 is normal when driving multi-services, it may not be detected even if the rest of the process is terminated due to an error.

Conclusion

It was confirmed that it was better to configure a container with a single service due to the advantages of single service per container + the disadvantages of multi-service per container. One thing that I regret is that I wanted to check what would happen if there was an exception, but I couldn't record it because there were no significant examples. If I find a good example later, I will add it.
