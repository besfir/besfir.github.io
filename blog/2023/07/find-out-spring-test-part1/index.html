In this article, we will look at the annotations that you basically need to know when testing using spring boots, and the annotations that help you understand other annotations.

@ContextConfiguration

It is used to load and configure ApplicationContext for integration testing, and classes attribute is usually used to specify component classes.

@ExtendWith(SpringExtension.class)
// ApplicationContext will be loaded from FooConfig and BarConfig
@ContextConfiguration(classes = {FooConfig.class, BarConfig.class})
class MyTest { ... }

💡 Component classes
✅ A class annotated with @Configuration.
✅ A component (that is, a class annotated with @Component, @Service, @Repository, or other stereotype annotations).
✅ A JSR-330 compliant class that is annotated with jakarta.inject annotations.
✅ Any class that contains @Bean-methods.
✅ Any other class that is intended to be registered as a Spring component (i.e., a Spring bean in the ApplicationContext), potentially taking advantage of automatic autowiring of a single constructor without the use of Spring annotations.

If the classes attribute is not explicitly declared, look for nested classes with @Configuration annotations, and if not, look for @SpringBootConfiguration. Since this annotation is declared inside the @SpringBootApplication, it can be seen that if the classes attribute is not explicitly declared, all beans are used as fallback.

@TestPropertySource

A annotation that is used to configure the locations() of properties files and inlined properties() to be added to the Environment's set of PropertySources for an Application Context for integration tests.

locations

Attribute that specifies the location of the *.properties file.

declare a properties file
@ContextConfiguration
@TestPropertySource("/test.properties")
class MyPropertyTests { ... }

The example declares that the properties are imported from the test.properties file in the classpath root.

properties

Attributes in the form key=value that must be added to the spring environment before running the test.

declare inlined properties
@ContextConfiguration
@TestPropertySource(properties = { "timezone = GMT", "port: 4242" })
class MyPropertyTests { ... }

The example declares timezone and port properties.

@TestConfiguration

Use to declare additional bean or customization for testing.

Automatic registration

To prevent unintended conflicts, automatic registration is not possible, and it can be manually registered through @Import, @SpringBootTest, or @ContextConfiguration. However, when used in the nested class, automatic registration is performed.

Override

As with using @Configuration, an exception occurs when the Bean name overlaps. In order to avoid this, there are cases where the same name is given and overridden. In this case, an exception may occur and the test may fail.

In the example below, the entire AppConfig class file is overridden, so the bean named bar does not exist. An exception therefore arises.

@SpringBootTest(classes = TestConfig.class)
class UnitTestApplicationTests {

  @Autowired
  private String foo;

  @Autowired
  private Boolean bar;    // 🔥 NoSuchBeanDefinitionException

  @Test
  void someTest() {
    assertThat(foo).isEqualTo("foo test");
    assertThat(bar).isTrue();
  }

}
@Configuration("AppConfig")
public class AppConfig {

  @Bean
  public String foo() {
        return "foo";
  }
    
  @Bean
  public Boolean bar() {
        return true;
  }
}
@TestConfiguration("AppConfig")
public class TestConfig {

  @Bean
  public String foo() {
        return "foo test";
  }
}

If we specify environmental properties that allow Bean override, you can override normally. At this time, if you register bean using @Import without using the classes attribute, the bean will be ignored, so be careful.

@SpringBootTest(
  classes = TestConfig.class,
  properties = "spring.main.allow-bean-definition-overriding=true"
)
class UnitTestApplication2Tests {

  @Autowired
  private String foo;

  @Autowired
  private Boolean bar;

  @Test
  void someTest() {
    assertThat(foo).isEqualTo("foo test");
    assertThat(bar).isTrue();
  }

}
@Configuration
public class AppConfig {

  @Bean
  public String foo() {
    return "foo";
  }

  @Bean
  public Boolean bar() {
    return true;
  }
}
@TestConfiguration
public class TestConfig {

  @Bean
  public String foo() {
    return "foo test";
  }
}

@SpringBootTest

This is the most powerful test annotation because it loads the same Application Context as the actual running application, allowing developers to perform any tests they want. However, it has the disadvantage of being heavier at scale because it loads all the beans.

As in the example below, combining the above-described annotations can be made to work similarly.

//@ExtendWith(SpringExtension.class)
//@ContextConfiguration
//@TestPropertySource(locations = "/application.properties")
@SpringBootTest
class MyTest { ... }

Let's take a quick look at each attribute.

classes

It is the same as the classes attribute of @ContextConfiguration.

properties

It is the same as the properties attribute of @TestPropertySource.

webEnvironment

Used to specify how the embedded web server works.

  • MOCK

Default value for this attribute. It creates a Web Application Context and mocking the servlet environment without running the embedded tomcat, so you cannot make an actual request. Therefore, it uses MockMvc or WebTestClient for testing and is used in conjunction with @AutoConfigureMockMvc to automatically configure these clients.

💡 WebTestClient can be used in both live server and Mock environments, and only when Webflux exists in the dependency.

@SpringBootTest
@AutoConfigureMockMvc
class MyMockMvcTests {

  @Test
  void testWithMockMvc(@Autowired MockMvc mvc) throws Exception {
      mvc.perform(get("/")).andExpect(status().isOk()).andExpect(content().string("Hello World"));
  }

  // If Spring WebFlux is on the classpath, you can drive MVC tests with a WebTestClient
  @Test
  void testWithWebTestClient(@Autowired WebTestClient webClient) {
      webClient
              .get().uri("/")
              .exchange()
              .expectStatus().isOk()
              .expectBody(String.class).isEqualTo("Hello World");
  }
    
}
  • RANDOM_PORT

Create a Web Application Context and run the embedded tomcat on random port. We can check the injected port using @LocalServerPort and use TestRestTemplate or WebTestClient for request testing. Since the Spring runs on a thread different from the test thread, @Transactional may not work as expected.

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class UnitTestApplicationTests {

  @Autowired
  private TestRestTemplate restTemplate;

  @LocalServerPort
  private String port;

  @Test
  public void greetingShouldReturnDefaultMessage() throws Exception {
    String requestUrl = "http://localhost:" + port + "/";
    assertThat(restTemplate.getForObject(requestUrl, String.class)).contains("Hello World");
  }
  
}
  • DEFINED_PORT

It has the same characteristics as RANDOM_PORT except for driving the built-in tomcat with the specified port.

@SpringBootTest(
  webEnvironment = SpringBootTest.WebEnvironment.DEFINED_PORT,
  properties = "server.port=49719"
)
class UnitTestApplicationTests {
  
  @Autowired
  private TestRestTemplate restTemplate;

  @LocalServerPort
  private String port;

  @Test
  void envPropertyTest() {
    assertThat(port).isEqualTo("49719");
  }
  
  @Test
  public void greetingShouldReturnDefaultMessage() throws Exception {
    String requestUrl = "http://localhost:" + port + "/";
    assertThat(restTemplate.getForObject(requestUrl, String.class)).contains("Hello World");
  }
  
}
  • NONE

Create Application Context without configuring the web environment, and do not run the embedded tomcat.

useMainMethod

As shown in the example below, if any setting is used in the main method, it is used to select whether to use the same setting in the test.

@SpringBootApplication
public class UnitTestApplication {

    public static void main(String[] args) {
      SpringApplication application = new SpringApplication(UnitTestApplication.class);
      application.setBannerMode(Banner.Mode.OFF);
      application.setAdditionalProfiles("someProfile");
      application.run(args);
    }

}
  • ALWAYS

Always run the main method. If there is no class with the annotation @SpringBootConfiguration or if there is no main method in that class, the execution will fail.

  • NEVER

It never runs the main method and creates a test-specific Spring Application.

  • WHEN_AVAILABLE

If there is no class with @SpringBootConfiguration annotation or if there is no main method in the class, create a test-specific Spring Application.

args

Attribute that declares arguments that must be passed to the application being tested. To use the argument, you can inject ApplicationArguments.

@SpringBootTest(args = {"--app.foo=bar"})
class UnitTestApplicationTests {

  @Autowired
  private ApplicationArguments args;

  @Test
  void argumentTest() {
    assertThat(args.getOptionNames()).containsOnly("app.foo");
    assertThat(args.getOptionValues("app.foo")).containsOnly("bar");
  }
}

@Transactional

Data is rolled back when used in testing.

@SpringBootTest
@Transactional
public class TransactionalTest {
    @Autowired
    private MemberRepository repository;

    @Test
    void createMemberTest() {
        Member member = new Member();
        member.setAge(10);
        member.setUsername("foo");
        repository.save(member);
    }
}

@MockBean & @SpyBean

It is the same as the use of Mock and Spy mentioned in the previous article. The difference is that Mock and Spy replace the target bean and to be registered in the Application Context. Therefore, the mock object replaces all places where the bean should have been used.

@SpringBootTest
class UnitTestApplicationTests {

  @Autowired
  private ApplicationContext ac;

  @MockBean
  private FooRepository repository;

  @Autowired
  private FooService service;
  
  @BeforeEach
  void setUp() {
    when(repository.get()).thenReturn("mock foo");
  }

  @Test
  void mockBeanFromContextTest() {
    FooRepository componentFromContext = ac.getBean(FooRepository.class);

    assertThat(componentFromContext.get()).isEqualTo("mock foo");
    verify(repository).get();
  }

  @Test
  void mockBeanTest() {
    assertThat(service.show()).isEqualTo("mock foo");
    verify(repository).get();
  }

  @TestConfiguration
  @Import(FooService.class) // A FooService injected with mocked FooRepository
  static class Config { }
}
@Component
public class FooRepository {

  public String get() {
    return "foo";
  }
}
@Service
@RequiredArgsConstructor
public class FooService {

  private final FooRepository repository;

  public String show() {
    return repository.get();
  }
}

Mock or Spy may be registered by name or type. If there is one particular type of bean, it is registered as a type without specifying a name, but if there is more than one type of bean, it must be explicitly named using the name attribute.

@SpringBootTest
class UnitTestApplicationTests {

  @Autowired
  private ApplicationContext ac;

  @MockBean(name = "fooRepository")
  private FooRepository repository;

  @BeforeEach
  void setUp() {
    when(repository.get()).thenReturn("mock foo");
  }


  @Test
  void mockBeanFromContextTest() {

    FooRepository componentFromContext = ac.getBean("fooRepository", FooRepository.class);

    assertThat(componentFromContext.get()).isEqualTo("mock foo");
    verify(repository).get();
  }

}
@Configuration
public class AppConfig {

  ...

  @Bean
  public FooRepository foo2() {
    return new FooRepository();
  }
}

If the class to be Mocked is not registered as bean, it is newly registered.

@SpringBootTest
class UnitTestApplicationTests {

  @MockBean
  SomeComponent someComponent;

  @Test
  void someComponentTest() {
    when(someComponent.doSomething()).thenReturn("world");

    assertThat(someComponent.doSomething()).isEqualTo("world");
  }
}
public class SomeComponent {
  
  public String  doSomething() {
    return "hello";
  }
}

@Import

Used to register component classes. It is used in combination with annotations that do not support component class registration, such as @WebMvcTest and @DataJpaTest.

Do not use this with @SpringBootTest. Because @Import has a higher priority, it is first registered as bean, and then the component class loaded through @SpringBootTest is registered as bean. Accordingly, the component class specified in @Import may be ignored.

To help you understand, let's look at the example below. We have overridden the bean in anticipation of foo = foo test, but the actual result is foo = foo.

@SpringBootTest( properties = "spring.main.allow-bean-definition-overriding=true")
@Import({TestConfig.class})
class UnitTestApplication2Tests {

  @Autowired
  private String foo;

  @Autowired
  private Boolean bar;

  @Test
  void someTest() {
    assertThat(foo).isEqualTo("foo");
    assertThat(bar).isTrue();
  }
}

@Configuration
public class AppConfig {

  @Bean
  public String foo() {
    return "foo";
  }

  @Bean
  public Boolean bar() {
    return true;
  }
}

@TestConfiguration
public class TestConfig {

  @Bean
  public String foo() {
    return "foo test";
  }

}
