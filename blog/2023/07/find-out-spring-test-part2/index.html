In this article, we'll learn about @WebMvcTest, MockMvc, TestRestTemplate, and WebTestClient. We'll see how they differ from RestTemplate and WebClient, respectively, and briefly summarize how to use them.

@WebMvcTest

It is used to focus testing on the web layer and disables all automatic configuration and instead applies only the configuration relevant to MVC tests. Configurations relevant to MVC testing are @Controller, @ControllerAdvice, @JsonComponent, Converter/GenericConverter, Filter, WebMvcConfigurer, and HandlerMethodArgumentResolver beans that are not @Component, @Service, or @Repository.

controllers

This attribute specifies the controller to test against. If no controller is explicitly specified, all @Controllers are registered as beans. If you need to register component classes other than @Controller as beans, you can use @Import with it.

properties

This is the same as the @TestPropertySource attribute that we discussed in the previous article. It is a key=value attribute that needs to be added to the Spring environment before running the test.

useDefaultFilters

This attribute specifies whether the filters used in the @SpringBootApplication should also be applied to tests. The default value is true, and includes the @Controller, @ControllerAdvice, and WebMvcConfigurer beans.

includeFilters

When used, only component classes that meet certain conditions can be registered as beans. This means that you can filter the component classes to be registered as beans based on certain types, annotations, etc.

@WebMvcTest(
  includeFilters = @ComponentScan.Filter(
    type = FilterType.ANNOTATION,
    classes = {Service.class, Repository.class}
  )
)

excludeFilters

As opposed to the includeFilters attribute, this attribute is for filtering out component classes that are not to be registered as beans.

@WebMvcTest(
  excludeFilters = @ComponentScan.Filter(
    type = FilterType.ASSIGNABLE_TYPE,
    classes = {GenericController.class}
  )
)

excludeAutoConfiguration

This attribute is used when you want to exclude certain autoconfiguration classes. In this example, we're excluding Spring Security from the autoconfiguration.

@WebMvcTest(excludeAutoConfiguration = SecurityAutoConfiguration.class)
class WebLayerTest4 { ... }

MockMvc

It provides an environment to test the controller without running the actual server. To simulate a request, you call the Dispatcher Servlet directly using perform(), passing a MockMvcRequest containing the request information as a parameter. Let's take a look at the example below to get a rough idea of the flow of operations when testing the controller.

@RestController
public class FooController {

  @GetMapping("/")
  public String foo() {
    return "bar";
  }
}

class WebLayerTest {

  MockMvc mockMvc;

  @BeforeEach
  void setup() {
    this.mockMvc = MockMvcBuilders
      .standaloneSetup(new FooController())
      .build();
  }

  @Test
  void helloTest() throws Exception {
    ResultActions resultActions = mockMvc.perform(MockMvcRequestBuilders.get("/"));
    
    MvcResult mvcResult = resultActions.andReturn();
    assertThat(mvcResult.getResponse().getContentAsString()).isEqualTo("bar");
  }
}
  • Lines 7-9 : Configure the servlet environment for testing through MockMvcBuilders.
  • Line 14 : Call perform() to send a request to the controller.
    • As explained earlier, since we are not starting a live server, we call the DispatcherServlet directly and communicate with the mocked request and response objects.
    • In this case, the DispatcherServlet is a TestDispatcherServlet extended for testing.
  • Lines 16-17 : Validate the results with the mock response returned by MockMvc.

Initialization

To illustrate the flow of operations, the example above uses a manual initialization of MockMvc without using Spring Context. There are other ways to initialize it directly using Web Application Context or using Spring Boot auto-configuration. To get a better understanding of MockMvc, let's take a look at how to initialize it.

Below is a common configuration that will be used in this part.

@RestController
public class HelloController {

  @GetMapping("/")
  public String hello() {
    return "Hello!!";
  }

  @GetMapping("/redirect")
  public void redirect(HttpServletResponse response) throws IOException {
    response.sendRedirect("/");
  }

  @GetMapping("/exception")
  public void exception() throws Exception {
    throw new Exception();
  }
}

@ControllerAdvice
public class GlobalExceptionHandler {

  @ExceptionHandler(Exception.class)
  protected ResponseEntity<String> handleException(Exception e) {
      return new ResponseEntity<>("response", HttpStatus.INTERNAL_SERVER_ERROR);
  }
}
  • standaloneSetup()

This method is a static factory method of MockMvcBuilders that creates a MockMvc instance, and it does not use the Spring Context, but stubbing the Web Application Context. Therefore, it can not use the features of the Spring Framework.

import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.*;

class WebLayerTest {

  MockMvc mockMvc;

  @BeforeEach
  void setup() {
    this.mockMvc = MockMvcBuilders
      .standaloneSetup(new HelloController())
      .setControllerAdvice(new GlobalExceptionHandler())
      .alwaysDo(print())
      .build();
  }

  // below will omitted, cause it is redundant
  @Test
  void helloTest() throws Exception {
    ResultActions resultActions = mockMvc.perform(get("/"));
    MvcResult mvcResult = resultActions.andReturn();
    assertThat(mvcResult.getResponse().getContentAsString()).isEqualTo("Hello!!");
  }

  @Test
  void redirectTest() throws Exception {
    ResultActions resultActions = mockMvc.perform(get("/redirect"));
    MvcResult mvcResult = resultActions.andReturn();
    assertThat(mvcResult.getResponse().getStatus()).isEqualTo(302);
  }

  @Test
  void exceptionTest() throws Exception {
      mockMvc
        .perform(get("/exception"))
        .andExpect((result)-> assertThat(result.getResolvedException()).isExactlyInstanceOf(Exception.class));
  }
}
  • webAppContextSetup()

This is a static factory method in MockMvcBuilders that creates a MockMvc instance using the actual Web Application Context. Because we're using a Spring Context, we can use the features of the Spring Framework.

@WebAppConfiguration
@ContextConfiguration(classes = UnitTestApplication.class)
@ExtendWith(SpringExtension.class)
class WebLayer2Test {

  @Autowired
  WebApplicationContext context;

  MockMvc mockMvc;

  @BeforeEach
  void setup() {
    this.mockMvc = MockMvcBuilders.webAppContextSetup(context).alwaysDo(print()).build();
  }

  // below will omitted, cause it is redundant
  ...
}

Line 1: Serves to register an empty WebApplicationContext that should be injected on line 9.
Line 2 : Specify a class with the @SpringBootApplication annotation to create the actual Spring Context.
Line 13 : Initialize the MockMvc using the WebApplicationContext.

❓ Based on the characteristics of the @ContextConfiguration that I outlined in the previous post, it seems like I should be able to find a class with the @SpringApplication annotation without setting the classes attribute and register all of component classes as beans just like a real Spring Context. However, if you don't explicitly specify the classes attribute, no Spring Context is created.

  • @AutoConfigureMockMvc

It's declared at the class level, and spring boot will automatically initialize MockMvc and register it as a bean. We can inject and use it at our convenience.

@SpringBootTest
@AutoConfigureMockMvc
class WebLayer3Test {

  @Autowired
  MockMvc mockMvc;

  // below will omitted, cause it is redundant
  ...
}

For @WebMvcTest, there is no need to declare @AutoConfigureMockMvc because it is declared inside.

@WebMvcTest
class WebLayer4Test {

  @Autowired
  MockMvc mockMvc;

  // below will omitted, cause it is redundant
  ...
}

I'am not going to look at the builder API, as we expect to use auto-configuration in most cases, so I'll just leave a link to the javadoc.

MockHttpServletRequestBuilder

A builder for creating the mocked request object when sending a request through MockMvc. This builder can be created through static factory methods, of which there are the following types

MethodText
get()Create a builder for GET requests.
post()Create a builder for POST requests.
put()Create a builder for a put request.
patch()Create a builder for a patch request.
delete()Create a builder for delete requests.
options()Create a builder for options requests.
head()Create a builder for a head request.
request()Create a builder with the given HTTP method.
multipart()Create a builder for multipart requests.
asyncDispatch()Create a RequestBuilder that has started asynchronous processing.

To use it, you can pass a URI or a string as a parameter.
Note that if you pass a string as a parameter, it will be substituted in the order of the urlVariables when generating the builder.

String
MockHttpServletRequestBuilder request = MockHttpServletRequestBuilder
  .get("/1/{world}/{hello}, "hello", "world");
Output
Request URI = /1/hello/world
URI
    String string = UriComponentsBuilder.fromUriString("/1/{world}/{hello}")
      .uriVariables(Map.of("hello", "hello"))
      .uriVariables(Map.of("world", "world"))
      .build().toString();
    MockHttpServletRequestBuilder request = post(string);
Output
Request URI = /1/world/hello

For a more detailed explanation, see MockHttpServletRequestBuilder Javadoc.

Variant of param(String, String...) with a MultiValueMap.

For methods that append data, it's hard to deduce from the name. Therefore, I've highlighted the word "append" for methods that append data based on javadoc. I've only included examples that are likely to be used frequently because the name of the method explicitly indicates what it does.

@Test
void postTest() throws Exception {
  AddUserRequest requestDto = new AddUserRequest("test@mail.com", "123",15);
  String content = objectMapper.writeValueAsString(requestDto);
  MockHttpServletRequestBuilder requestBuilder = post("/post")
    .content(content)
    .contentType(MediaType.APPLICATION_JSON);
    
  MvcResult mvcResult = mockMvc.perform(requestBuilder)
    .andReturn();
    
  assertThat(mvcResult.getResponse().getStatus()).isEqualTo(200);
}

For a more detailed explanation, see MockHttpServletRequestBuilder Javadoc.

MockMvcResultHandlers

Static factory method for assigning some behavior to the response result. Used in conjunction with andDo() in ResultActions.

  • log() : Prints a log of the MvcResult details at the DEBUG level.
  • print() : Prints the MvcResult details to the standard output stream.
@Test
void helloTest() throws Exception {
  ResultActions resultActions = mockMvc.perform(get("/")).andDo(print());
  MvcResult mvcResult = resultActions.andReturn();
  assertThat(mvcResult.getResponse().getContentAsString()).isEqualTo("Hello world");
}

ResultActions

You can apply actions such as verification to the execution results of MockMvc.

  • andExpect()

The execution result can be verified, and the Json string can be verified using the jsonPath expression.

import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.*;

mockMvc.perform(get("/member/1"))
  .andExpect(status().isOk())
  .andExpect(content().contentType(MediaType.APPLICATION_JSON))
  .andExpect(jsonPath("$.member.name").value("Walter"));
  • andExpectAll()

Like JUnit's assertAll(), it performs multiple verify, and if one check fails, it checks them all without exiting.

mockMvc.perform(get("/member/1"))
  .andExpectAll(
      status().isOk(),
      content().contentType(MediaType.APPLICATION_JSON),
      jsonPath("$.member.name").value("Walter")
  );
  • andDo()

Perform a general action.

mockMvc.perform(get("/member/1")).andDo(print());
  • andReturn()

Returns the result of the executed MockMvc request, allowing you to access the result directly.

ResultActions resultActions = mockMvc.perform(get("/")).andDo(print());
MvcResult mvcResult = resultActions.andReturn();
assertThat(mvcResult.getResponse().getContentAsString()).isEqualTo("Hello world");

MvcResult

Provides access to the results of an executed MockMvc request. There are many APIs, but it is commonly used to access the response directly using getResponse().

AddUserRequest request = new AddUserRequest("nomail@email.com", "password", 15);
ObjectMapper objectMapper = new ObjectMapper();

MvcResult mvcResult = mockMvc.perform(
    post("/post")
      .content(objectMapper.writeValueAsBytes(request))
      .contentType(MediaType.APPLICATION_JSON)
  )
  .andReturn();

String content = mvcResult.getResponse().getContentAsString();
AddUserResponse response = objectMapper.readValue(content, AddUserResponse.class);
assertThat(response).isNotNull(); // assert what you need

For a more detailed explanation, see MvcResult Javadoc.

TestRestTemplate

This is a specialized RestTemplate for testing, used for integration testing with the WebEnvironment.RANDOM_PORT property of @SpringBootTest because it is used to make requests to the live server. It is an alternative to RestTemplate for testing, but it does not extend RestTemplate for fear of causing problems with injection.

For basic authentication, it provides a constructor to create a template with the specified credentials. All requests made using this instance will be authenticated using the supplied credential.

TestRestTemplate testRestTemplate = new TestRestTemplate("username", "password");
ResponseEntity<String> response = testRestTemplate.getForEntity("/member/info", String.class);

Assertions.assertEquals(response.getStatusCode(), HttpStatus.OK);

WebTestClient

This is a specialized WebClient for testing purposes, and it can use a live server, a mock environment, asynchronous requests, and synchronous requests. However, it does not support configuring a mock environment, so if you want to use it as a client in a mock environment, you need to initialize MockMvc and configure the mock environment.

You need to use the @AutoConfigureWebTestClient annotation to register WebTestClient as a bean, but @WebMvcTest is included in the autoconfiguration, so you can just inject it and use it.

Synchronous mock env
@SpringBootTest
@AutoConfigureMockMvc
@AutoConfigureWebTestClient
class WebClientTest {

  @Autowired
  WebTestClient webClient;

  @Test
  void synchronousTest() {
    String response = webClient.get()
      .uri("/")
      .exchange()
      .expectBody(String.class)
      .returnResult()
      .getResponseBody();
    assertThat(response).isEqualTo("Hello world");
  }
}
Synchronous live server
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureWebTestClient
class WebClientTest {
  // 중복 생략.
}
Asynchronous mock env
@WebMvcTest
class WebClientTest {

  @Autowired
  WebTestClient webClient;
  
  @Test
  void asynchronousTest() {
    Mono<String> asyncResponse = webClient.get()
      .uri("/")
      .exchange()
      .returnResult(String.class)
      .getResponseBody()
      .next();
  
    asyncResponse.subscribe(v -> assertThat(v).isEqualTo("Hello world"));
  }
}
