Database schema

A database schema, or schema for short, defines and represents the structure of a database based on a data model. The schema is determined early in the design of a database and is not often changed once it is established.

A schematized schema is called a schema diagram and is a powerful tool that helps you visually represent the relationships between tables, as shown in the figure below. Schema diagrams play an important role in the design and implementation of databases and help in the development and maintenance of database systems.

Database state

The actual data in a database can change quite often. The data in a database at any given time is called the database state or database snapshot. Another way to say it is a set of the current instances in the database.

Database state includes the contents and attributes of all tables in the database, relationships between tables, transaction information, log information, and more.

💡 A database instance is a state in which a DBMS is loaded into memory and running at any given time.

Three-schema architecture

It is one of the architectures for building database systems and has the purpose of separating the physical database from the user application. The model has three levels, each of which has a defined schema. This isolation is used to increase reliability by ensuring that changes at any level do not affect the higher levels.

However, most modern DBMSs do not completely or explicitly divide the three phases. This is because it is not practical to ensure that changes at each level do not affect the higher levels.

Even though we've divided the schema into three levels, the internal level is where the actual data resides.

💡 User applications refer to software used by general users or individuals or companies that handle tasks, and act as clients of database systems.

External schemas at external level (or User views at view level)

It expresses only the data that specific users need, and hides data that they don't need to know. This schema is expressed through a logical data model. It is also called external views, user views, and sub schema, among other names.

The above description is a bit confusing, but in a nutshell, it means a tabular representation of the structure of a database that a user or application sees in response to a query in the database.

Conceptual schemas at conceptual level

It describes the structure of the entire database, focusing on entities, data types, associations, user operations, and constraints. However, it hides information about the physical storage structure. An example of this schema is the ER diagram, which, like the external schema, is represented through a logical data model.

A conceptual schema is a database structure designed by a DBA or developer in the form of tables.

💡 A user operation is an action that a user or application performs to interact with a database. There can be many different types of user operations, but they mainly include the following tasks

  • Data retrieval (Querying)
  • Inserting data (Insertion)
  • Modifying data (Update)
  • Deleting data (Deletion)
  • Table Creation and Alteration
  • Database Backup and Restore
  • User Privilege Management

💡 An entity is a concept used to represent a conceptual object in database design, while a table is a structure that actually physically stores data. Entities and tables may not always correspond. (For example, inheritance structures)

Internal schemas at internal level

Represent how data is physically stored through a physical data model. It describes tangible things like data storage, data structures, and access paths. The main goal of this schema is to improve the performance and efficiency of the database by optimizing physical storage and processing.

Let's look at the difference between a logical and physical data model with an example code. The following example assumes a logical data model for storing student grades.

logical data model
Students (Table):
+------------+--------------+---------+
| Student_ID | Student_Name | Major   |
+------------+--------------+---------+
| 101        | John Doe     | CS      |
| 102        | Jane Smith   | History |
| 103        | Bob Johnson  | Physics |
| ...        | ...          | ...     |
+------------+--------------+---------+

Now, if we were to implement this logical data model as a physical data model, we might add a physical table structure, index structure, physical placement of data files, etc.

physical data model
Physical Table Structure:
- Students (Table):
  +--------------+-------------+---------------+
  | Student_ID   | Student_Name| Major         |
  +--------------+-------------+---------------+
  | 101 (Offset) | John Doe    | CS (Offset)   |
  | 102 (Offset) | Jane Smith  | History (Offset)|
  | 103 (Offset) | Bob Johnson | Physics (Offset)|
  | ...          | ...         | ...           |
  +--------------+-------------+---------------+

Index Structure:
- Index on Student_ID:
  +--------------+-------------+
  | Student_ID   | Offset      |
  +--------------+-------------+
  | 101          | Offset_1    |
  | 102          | Offset_2    |
  | 103          | Offset_3    |
  | ...          | ...         |
  +--------------+-------------+

Physical Storage on Disk:
- Students Data File:
  1. Offset_1: [101][John Doe][CS]
  2. Offset_2: [102][Jane Smith][History]
  3. Offset_3: [103][Bob Johnson][Physics]
  ...

Database Language

DDL (Data Definition Language)

It is a language used to define conceptual schemas, and in some cases even internal schemas. Examples include creating, modifying, and deleting tables, creating indexes, and creating databases.

SDL (Storage Definition Language)

A language used to define internal schemas. Nowadays, especially in relational DBMSs, SDL is rarely used and is replaced by settings such as parameters.

VDL (View Definition Language)

A language used to define an external schema. In most DBMSs, DDL also fulfills the role of VDL.

DML (Data Manipulation Language)

A language for utilizing data in a database. It provides the ability to add, delete, modify, and search data.

Unified language

Today's DBMSs exist as a unified language rather than DML, VDL, and DDL. A prime example of this is the relational data language - SQL.
