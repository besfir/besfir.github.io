Can i use PM2 inside Docker?

Besfir ·Updated at June 2, 2023
Table of Contents
  • Monitoring becomes difficult.
  • It goes against Docker's best practice.
  • Docker and PM2 are interchangeable.
  • Conclusion
  • Reference

I recently re-checked the project I created when I didn't know Docker at all and found that I was using PM2 inside Docker. It didn't fit the best practices mentioned in the Docker document, but I thought it might be an exception, so I investigated it.

Monitoring becomes difficult.

Since PM2 takes charge of the task that Docker was doing, it becomes difficult to monitor CPU/Memory metrics in Docker. In addition, even if the service dies, it will not be known from Docker because it will restart in PM2. Therefore, normal monitoring cannot be performed in the monitoring system (Cloudwatch, etc.).

It goes against Docker's best practice.

As a best practice mentioned in the official document, it is recommended to use one service per container for separation of concern. Therefore, using PM2 internally for clustering is not a very good example.

See Why is it recommended to run only one process in a container?

Docker and PM2 are interchangeable.

The main purposes of using PM2 include clustering, abnormally terminated process restart, and logging. This can be replaced to some extent in Docker, where clustering can be replaced by docker-compose's scale command, abnormally terminated process restart can be replaced by restart policy, and logging can be replaced by logs command. Therefore, it would be better to choose and use it according to the situation rather than using more computing resources by using both.

Conclusion

If you check the PM2 document Docker Integration, it will be possible to implement it. However, based on the above, I found that using PM2 inside Docker does more harm than good. Therefore, I will conclude that it is better not to use PM2 inside Docker and conclude the article.
