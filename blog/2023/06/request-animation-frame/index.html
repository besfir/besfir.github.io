Let's find out about requestAnimationFrame()

Besfir ·Updated at June 12, 2023

Before we start, we need to use some terms repeatedly, but it's long, so let's define it as a contraction.

  • setTimeout() and setInterval() => timer function
  • requestAnimationFrame() => rAF()

In order to implement animation that is difficult to express with CSS alone, JavaScript must be used, and using it without optimization can cause performance problems. Prior to the release of rAF() api, the timer function was used as an optimization technique, and since then, rAF() has been used.

In this article, we will check what problems the timer function has and compare it with rAF().

Frame loss

Since the timer function operates without considering the frame, it cannot guarantee when the callback function is called. Therefore, if it runs over two frames as shown in the figure below, the animation will not be smooth because it misses one frame.

💡 How does 16ms in the figure come from?
If you didn't specifically choose refresh rate and buy it, you would be using a monitor at 60Hz refresh rate, which means that the screen is refreshing itself 60 times every second. Therefore, the screen is updated at intervals of 1,000 ms/60 times ≈ 16 ms.

On the other hand, rAF() is synchronized with the screen scan rate and is executed when the browser is ready to draw the screen. To be more specific, the callback function written in rAF() is registered in the Animation Frame queue, and all callback functions in this queue are processed at once just before the next repaint. Accordingly, it is possible to prevent a problem in which a frame is lost because the callback function is started in the middle.

Using the timer function and rAF(), we made an animation in which each box moves 5 px to the right, a total of 1,000 px, as an example code. As mentioned earlier, the timer function is not guaranteed when the callback function will be executed, so frame loss may or may not occur enough to be visually confirmed. In this case, it can be confirmed by repeating it several times.

Waste of system resources

The timer function also works within an inactive window or hidden iframe, wasting system resources, resulting in battery life and performance degradation problems. However, using rAF() saves system resources by stopping work when the window becomes inactive.

To see this for yourself, let's switch the browser to another tab as soon as we run the animation of the example code mentioned above. If you wait for about 5 seconds and go back, the animation run with the timer function is already over, but the function run with the rAF() starts again from the spot.

rAF() performance optimization

window.addEventListener('scroll', function(e) {
  if (!ticking) {
    window.requestAnimationFrame(function() {
      doSomething();
      ticking = false;
    });
    ticking = true;
  }
});

I don't know where I saw it, but I saw a post saying that scroll events occur so much that I need to throttle them with the code above. Since then, I have applied the throttle unconditionally when using the scroll event, and when I happened to see the MDN docs, I realized that I had been mistaken.

Since the scroll event occurs when the scrolled screen is rendered, it is synchronized with the refresh rate. Therefore, because it runs at the same speed as rAF(), which is synchronized with the refresh rate, a throttle using rAF() is often unnecessary. So in what case is rAF() needed? It is necessary when using code that can cause layout thrashing as shown below.

function onScroll(){
  const box = element.getBoundingClientRect();
  if(box.top > pos){
    element.style.height = box.height * 2
  }
}
window.addEventListener('scroll', () => {
  onScroll()
});

Reflow, forced synchronous layout and layout thrashing

The above figure shows the workflow when rendering, and it is difficult to explain all the steps, so let's just talk about the reflow required for today's topic.

reflow refers to the process of re-performing from the layout step when the geometric value of the DOM element is changed. As can be seen from the re-performance from the layout stage, it is a costly task because not only the changed elements but also the calculation of other elements must be re-performed. Therefore, the browser collects tasks that require reflow in the queue to reduce this cost and schedules them for processing in one reflow in the next frame.

However, when trying to process reading and writing geometric values in one frame, the reflow is performed immediately. This is called forced synchronous layout, and a phenomenon in which forced synchronous layout occurs repeatedly, resulting in performance degradation is called layout thrashing.

Prevent layout thrashing

The code mentioned above reads the geometric value of top, resulting in a forced synchronous layout. Therefore, we will use the features of rAF() described above to separate the read logic that causes the reflow and hand over the remaining computational logic to rAF() as a callback function. This effectively defends layout thrashing because the read is processed in the current frame, and the rest of the operations are collected in the next frame and processed in one time.

function onScroll(height){
  element.style.height = height * 2
}
window.addEventListener('scroll', () => {
  const box = element.getBoundingClientRect();
  if(box.top > pos){
      requestAnimationFrame(()=>{
          onScroll(box.height)
      })
  }
});
