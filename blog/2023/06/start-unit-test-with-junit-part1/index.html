I've been aware of the importance of testing code for a long time, but I've been delaying them, saying that I'll learn later because I can finish my work faster if I write a code during the time I study for the testing code. However, whenever the requirements were added little by little, I suddenly felt ashamed of myself for testing them in a fist style. In this article, I will summarize the information that must be known, but can proceed with the test properly.

Unit test

A unit test is a procedure in computer programming to verify that certain modules of source code work exactly as intended. That is, it refers to the procedure of writing a test case for all functions and methods.

Korean Wikipedia

Advantages

  • Detect bug: The functionality is tested in small pieces, so you can quickly check where the problem occurred.
  • Refactoring: ensures better accuracy and stability in refectoring because it makes the code small and simple.
  • Documentation: Because there are libraries that use unit tests to document APIs, documentation and maintenance become convenient.

Disadvantages and Limitations

  • Ensuring results: because the method is isolated and simulated, the results of the unit test do not always represent the results of the system as a whole.
  • Development and Maintenance Costs: Test codes also need to be developed and maintained, increasing the scope to be managed.

JUnit 5

Unlike the previous JUnit, JUnit 5 consists of JUnit Platform + JUnit Jupiter + JUnit Vintage.

  • JUnit Platform: It is responsible for executing the test framework on the JVM. It also provides a test engine API to facilitate integration with JUnit.
  • JUnit Jupiter: It provides a test engine for Jupiter subprojects such as JUnit 5.
  • JUnit Vintage: It provides a test engine for running JUnit 3 and JUnit 4 based tests.

Basic annotations

Since it is frequently used, I am going to organize the annotations that I basically need to know.

  • @Test: Indicates that the method is a test method.
  • @BeforeAll: Indicates that this is the first method to be executed within the class.
  • @AfterAll: Indicates that this is the last method to be executed within the class.
  • @BeforeEach: Indicates that this is a method that must be executed before each test method in the class.
  • @AfterEach: Indicates that this is a method that must be executed after each test method in the class.

Lifecycle methods

@BeforeAll, @AfterAll, @BeforeEach, and @AfterEach are called life cycle methods because they run according to the life cycle of the test.

import org.junit.jupiter.api.AfterAll;
import org.junit.jupiter.api.AfterEach;
import org.junit.jupiter.api.BeforeAll;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Disabled;
import org.junit.jupiter.api.Test;

public class OrderTest {

    @BeforeAll
    static void beforeAll() {
        System.out.println("=-=-=-=-=-=");
        System.out.println("Before all");
    }

    @AfterAll
    static void afterAll() {
        System.out.println("After all");
        System.out.println("=-=-=-=-=-=");
    }

    @BeforeEach
    void setUp() {
        System.out.println("-----");
        System.out.println("Before each");
    }

    @AfterEach
    void tearDown() {
        System.out.println("After each");
        System.out.println("-----");
    }

    @Test
    void runFirst() {
        System.out.println("run first");
    }

    @Test
    void runLast() {
        System.out.println("run last");
    }

    @Test
    @Disabled
    void disableTest() {
        System.out.println("it does not run");
    }
}

-----
Before each
run last
After each

-----
Before each
run first
After each
-----
After all

@DisplayName

Let's take a look at the log of the Lifecycle Method unit again. I wrote runFirst() first, and even if I think about it in alphabetical order, it would be run first, but it did later.

This is because JUnit runs the tests in an unpredictable order, as unit tests should not normally depend on order. However, sometimes the order of testing may be important. In this case, annotate your test class or test interface with @TestMethodOrder and specify the desired MethodOrderer implementation.

OrderAnnotation

Sort the methods in ascending order based on the value specified in the @Order annotation. The default order value is Integer.MAX_VALUE / 2, and a method larger than this value runs later than a method without @Order annotation.

import org.junit.jupiter.api.MethodOrderer.OrderAnnotation;
import org.junit.jupiter.api.Order;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.TestMethodOrder;

@TestMethodOrder(OrderAnnotation.class)
public class TestMethodOrderTest {

    @Test
    @Order(1)
    void runFirst() {
        System.out.println("run first");
    }

    @Test
    @Order(2)
    void runSecond() {
        System.out.println("run second");
    }
    
    @Test
    void runThird() {
        System.out.println("run third");
    }

    @Test
    @Order(Integer.MAX_VALUE / 2 + 1)
    void run4th() {
        System.out.println("run 4th");
    }

    @Test
    @Order(Integer.MAX_VALUE)
    void runLast() {
        System.out.println("run last");
    }

}
Output
run first
run second
run third
run 4th
run last

MethodName

Sorts test methods alphanumerically based on their names and formal parameter lists.

import org.junit.jupiter.api.MethodOrderer.MethodName;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.TestMethodOrder;

@TestMethodOrder(MethodName.class)
public class TestMethodOrderTest {

    @Test
    void runLast() {
        System.out.println("run last");
    }

    @Test
    void runFirst() {
        System.out.println("run first");
    }
}
Output
run first
run last

DisplayName

Sorts test methods alphanumerically based on their display names.

import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.MethodOrderer;
import org.junit.jupiter.api.MethodOrderer.Random;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.TestMethodOrder;

@TestMethodOrder(MethodOrderer.DisplayName.class)
public class TestMethodOrderTest {

    @Test
    @DisplayName("run last")

        System.out.println("run last");
    }

    @Test
    @DisplayName("run first")

        System.out.println("run first");
    }

Random

It is randomly sorted using System.nanoTime() as a seed. It does not always guarantee the same order of execution.

import org.junit.jupiter.api.Test;

@TestMethodOrder(Random.class)

Assertions

The task of verifying the results of a test is called assertion, and Assertions, the title of this section, refers to a collection of asserting utility methods. So, all the asserting utility methods are in the org.junit.jupiter.api.Assertions. From now on, the method in this class will be called the assertion method.

Assertion methods optionally receive an error message in the last parameter. It can be handed over to a lambda that returns a string type or string, and when using lambda, it is called only when the test fails, so it seems that it can help optimize performance a little.

assertEquals() & assertNotEquals()

Used to verify that the values are the same / different.

@Test
void equalsTest() {
    assertEquals(1 + 1, 2);
}


@Test
void notEqualsTest() {
    assertNotEquals(1 + 1, 3);
}

assertNull() & assertNotNull()

Used to verify that the values is null / not null.

@Test
void nullTest() {
    assertNull(null);
}

@Test
void notNullTest() {
    assertNotNull(1);
}

assertTrue() & assertFalse()

Used to verify that the values is true / false.

@Test
void trueTest() {
    assertTrue(1 < 2);
}

@Test
void falseTest() {
    assertFalse(1 > 2);
}

assertSame() & assertNotSame()

Used to verify that the values are the same object / different object.

@Test
void sameTest() {
    String expected = "expected";
    assertSame(expected, "expected");
}

@Test
void notSameTest() {
    String expected = "expected";
    assertNotSame(expected, new String("expected"));
}

assertArrayEquals()

Used to verify that the values of the two arrays match.

@Test
void arrayEqualsTest() {
    char[] expected = {'e', 'x', 'p', 'e', 'c', 't', 'e', 'd'};
    char[] actual = "expected".toCharArray();
    assertArrayEquals(expected, actual);
}

assertIterableEquals()

It is used to verify that the values of the two Iterable implementations match. The values must be returned in the same order and do not necessarily have to be of the same type.

@Test
void iterableEqualsTest() {
    Iterable<String> expected = new ArrayList<>(Arrays.asList("Hello", "Junit", "Practice"));
    Iterable<String> actual = new LinkedHashSet<>(Arrays.asList("Hello", "Junit", "Practice"));

    assertIterableEquals(expected, actual);
}

assertLinesMatch()

It is used to verify that the strings of the two lists match, and each element performs the following steps from the beginning.

  1. Check that the values of each element are the same. - If the same, move on to the next element.
  2. If not, check the regular expression with String.matches(). - If the same, move on to the next element.
  3. If not, check if it is a fast-forward marker. - If the same, move on to the next element.
Case for step 2
@Test
void linesMatchTestCaseForStep2() {
    List<String> expected = Arrays.asList("Today's ", ".*", "\\d+");
    List<String> actual = Arrays.asList("Today's ", "lucky number is ", "72");
    assertLinesMatch(expected, actual);
}
Case for step 3
@Test
void linesMatchTestCaseForStep3() {
    List<String> expected = Arrays.asList(">>2>>", "blue");
    List<String> actual = Arrays.asList("The sky ", "is ", "blue");
    assertLinesMatch(expected, actual);
}

assertThrows()

Used to verify that the expected exception type is thrown.

@Test
void throwsTest() {

    Exception exception = assertThrows(Exception.class, () -> System.out.println(1/0));
    assertEquals("/ by zero", exception.getMessage());
}

assertThrowsExactly()

Used to verify that the expected exception is thrown.

@Test
void ThrowsExactlyTest() {

    Exception exception = assertThrowsExactly(ArithmeticException.class, () -> System.out.println(1/0));
    assertEquals("/ by zero", exception.getMessage());
}

assertTimeout()

Used to verify completion before the expected time value is exceeded. Even if the time is exceeded, the test runs until the end, and an error message shows how long it has passed.

success
import static java.time.Duration.ofMillis;

@Test
void timeoutTest() {
    String actualResult = assertTimeout(ofMillis(10), () -> "Hello world!");
    assertEquals("Hello world!", actualResult);
}
fail
import static java.time.Duration.ofMillis;
@Test
void timeoutTestExceeded() {
    assertTimeout(ofMillis(10), () -> {
        Thread.sleep(100);
    });
}
Failed output
execution exceeded timeout of 10 ms by 92 ms

assertTimeoutPreemptively()

Used to verify completion before the expected time value is exceeded. The test fails immediately when the time is exceeded.

@Test
void TimeoutPreemptivelyTest() {
    assertTimeoutPreemptively(ofMillis(10), () -> {
        Thread.sleep(100);
    });
}
Output
execution timed out after 10 ms

assertAll()

The assertion method does not execute subsequent code because an exception occurs when the test fails. However, there may be cases where you want to run all assertion methods regardless of failure. In that case, if you use this method to group the assertion methods, they do not cause an exception, so even if they fail, they do not stop immediately and aggregate the errors.

But, if an exception (e.g., Out of Memory) is generated in the blacklist, the execution is immediately stopped and overwritten with an unchecked exception.

@Test
void assertAllTest() {
    assertAll(
            "Group name",
            () -> assertTrue(1 < 2),
            () -> fail("Occur manually"),
            () -> assertTrue(1 > 2)
    );
}

    org.opentest4j.AssertionFailedError: expected: <true> but was: <false>
Expected :true
Actual   :false

@TestInstance

JUnit creates a new test instance for each test method to minimize the impact between tests. However, you may not want to create a new test instance every time as needed.

In this case, you can annotate the class @TestInstance to change the test instance creation pattern according to the mode you specify. The selectable modes are PER_METHOD and PER_CLASS.

  • PER_METHOD: It is the default value and is characterized by not sharing the state because it creates a new instance for each method.
    • @BeforeAll and @AfterAll must be declared in the static method.
  • PER_CLASS: It is possible to share the status in the method because each class creates a new instance.
    • @BeforeAll and @AfterAll do not necessarily have to be declared in the static method.

Why do we need to understand the life cycle of a test instance

If we don't know the life cycle of the test instance, we write the code on the premise that we can share the state because we think in PER_CLASS mode. Therefore, the following cases may occur.

@TestMethodOrder(MethodOrderer.OrderAnnotation.class)
class LifeCycleTest {

    int counter = 0;

    @Test
    @Order(1)
    void counterShouldBe1() {
        System.out.printf("Expected: %d, Actual: %d\n", 1, ++counter);
    }
    @Test
    @Order(2)
    void counterShouldBe2() {
        System.out.printf("Expected: %d, Actual: %d\n", 2, ++counter);
    }
}
Output
Expected: 1, Actual: 1
Expected: 2, Actual: 1

We expected to increase by 1 for each test, but because a new instance was created for each test method, the counter would start from zero every time.
