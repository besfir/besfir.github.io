In this article, we'll learn about the concept of a relational database, including the terms relation, primary key, foreign key, and constraints.
The concept of relational databases comes from mathematics, so to make it easier to understand, we'll start by looking at the concept of relational databases in math.

Set

A set is a collection of different elements, characterized by the fact that the order of the elements in a set does not matter.

Example
{1, 3, 11, 4, 7}
  • The set above satisfies the definition of a set because there are no duplicates.
  • It doesn't matter if we write it as {11, 3, 4, 1, 7} or {4, 11, 3, 1, 7}.
    • Because of the order-insensitive nature, the meaning doesn't change because of the reordering.

Relation in mathematics

The set of all sets that can be represented by selecting one element from each of the sets A and B in the figure above is called a cartesian product and can be symbolized as A × B. The reason for the sudden mention of the Cartesian product is that it is related to the relation in math.

When a relationship is formed by selecting one element from each of two sets, it is called a binary relation. Let's look at the figure above to understand binary relations.

  • Let's represent a binary relation with the symbol R.
  • R = null`
    • The subsets {1, p}, {2, q}, and {2, r} of R are each called an ordered pair.
  • A × B = {{1, p}, {1, q}, {1, r}, {2, p}, {2, q}, {2, r}}
  • This shows that a binary relation R is a subset of the Cartesian product of sets A and B.
    • It is symbolically represented by R ⊆ A × B.

If there are n sets, it is called an n-ary relation, which is a subset of the Cartesian product over n sets. Let's generalize and write an n-ary relation.

  • Consider the n-ary relation R = {{1, p, ..., ɑ}, {2, q, ..., β}, {2, r, ..., β}}.
  • `R' consists of three ordered pair sets.
  • The subsets of R', {1, p, ..., ɑ}, {2, q, ..., β}, and {2, r, ..., β}`, are each called a tuple.
    • In this case, the set is called an n-tuple because it is a set of n elements.
    • A tuple is a concept used to represent an association between two or more sets.
  • To summarize, an n-ary relation R is an n-tuple of three sets of order pairs.

Based on the above, a relation in mathematics can be expressed as follows

  • A subset of Cartesian products.
  • A set of tuples.

Relation in relational data models

Now that we've looked at relationships in math, let's apply them to the relational data model. Sets, set names, tuples, and relations in math correspond to domains, domain names, tuples, and relations in the relational data model, respectively. Here, a domain is a set of attribute values. To make it easier to understand, let's create a student relation example.

Define domains

Let's define the student domains as follows

  • students_ids : set of student numbers, 7-digit integer.
  • human_names: set of human names, string.
  • university_grades: set of university grades, {1,2,3,4}
  • major_names: a set of names of majors studied at the university.
  • phone_numbers : Set of cell phone numbers.

Earlier, we said that a domain is a set of attribute values. However, if you look at the domain definitions above, you'll notice that they specify the type of the values. For example, students_ids is a 7-digit integer, human_names is a string, and phone_numbers is a string or natural number depending on the definition. They also inherit the characteristics of the set in common, so no duplicate data can be used, which can also be seen as a type that does not allow duplication.

In this way, domains can also be seen as defining the types of values that each attribute can have.

Representing in graph

We'll use the domain we defined earlier to represent the student relation. Since we don't yet know how to represent the relation other than the mathematical representation we practiced earlier, we'll make it as mathematical as possible.

As I was creating it, something changed. In addition to the student's phone number, I realized that I needed an emergency contact number. So we used the phone number domain twice to represent this. However, domains alone don't tell you which domain is the student's phone number or the emergency contact number.

We mentioned earlier that a domain can also be seen as defining the type of attribute value. In this case, the type of value is specified through the domain name. Therefore, the domain name only specifies the type of the domain, not its purpose. This is where attributes come in. The attribute is used to express the name of the role that the domain plays in the relation. By using an attribute, we can distinguish between a student's phone number and an emergency contact in the image above, which use the same phone number domain.

As we saw in the previous section on relation in math, the association between each set can be represented as a tuple, which in the figure is a 6-tuple. As the degree of the tuple increases and the number of order pairs increases, the mathematical representation of the tuple becomes very complex. Therefore, in the relational data model, relation is represented by a table because it is the easiest way to represent tuples.

Sometimes a relation is an abstract concept that refers to a table, as above, but sometimes it refers to a relation state, as below. In this case, a relation is a set of tuples at a point in time, i.e., actual data at a point in time. However, the difference in semantics is minimal, and you shouldn't have any trouble understanding the context either way.

Let's summarize the main concepts

  • Domain: An atomic set of values that is no longer divisible, defining constraints on data types and attributes. It specifies the valid values that can be assigned to attributes.
  • domain name: The name assigned to a domain, used to define the range of values in the domain and to constrain the values of its attributes.
  • attribute: The name of the role the domain plays in the relationship. Each attribute represents a specific column in the table and can only have values defined for that domain.
  • tuple: A unit of data that represents each row in a table, and is an ordered pair of values containing the value of each attribute. Also known as a record, some values can be NULL.
    • I will refer to them as records when referring to rows in the relational data model, and tuples when bringing them into a programming language.
  • relation name: serves as the name of the table, indicating what subject or object the table contains information about.
  • relation: an abstract concept that combines a set of tuples (=relationship state) and attributes.

Relation schema

Represents the structure of a relation, represented as a list of relation names and attributes. It also includes constraints associated with the attributes.

STUDENT(id, name, grade, major, phone_num, emer_phone_num)

Degree of a relation

The degree of a relation is the number of attributes in the relation schema.

STUDENT(id, name, grade, major, phone_num, emer_phone_num) -> degree 6

Relational database

It is a structured database based on a relational data model. This relational database is composed of multiple relations.

Relational database schema

A relational database schema is a combination of a set of relational schemas and a set of integrity constraints. It is an important concept that defines the structure and integrity of a database.

Meaning of NULL

NULL has a very ambiguous meaning.

  • The value does not exist.
  • The value exists, but we don't know what it is yet.
  • Not relevant.

Let's look at an example.

  • For the TOEIC score attribute, NULL could mean the following
    • You haven't taken the test yet.
    • Submitted but not yet entered.
    • Submitted but missing.
  • The transfer attribute also has an ambiguous meaning for NULL, which could be
    • Not applicable because you haven't transferred.
    • You transferred but didn't update.

Keys

Super key

A set of attributes that uniquely identify the records in a relation.

  • For PLAYER(id, name, team_id, back_number, birth_date).
  • The super key is {id, name, team_id, back_number, birth_date}, {id, name}, {name, team_id, back_number}, etc.

Candidate key

A super key that cannot uniquely identify a record by removing any of its attributes. It is also called a key or minimal super key.

  • For PLAYER(id, name, team_id, back_number, birth_date).
  • The candidate keys are {id}, {team_id, back_number}.
    • {id}: The id value can identify the record because it is unique on its own, and removing that attribute would make it indistinguishable, so it is a candidate key.
    • {team_id, back_number} : Since there are multiple players on a team, the team_id attribute is not enough to distinguish them. The back_number attribute is also indistinguishable because there may be players with the same number on different teams. Therefore, removing either of them makes the record unidentifiable, making it a candidate key.

Primary key

The candidate key selected to uniquely identify a record in the relation.

  • For PLAYER(id, name, team_id, back_number, birth_date).
  • The primary key is {id} or {team_id, back_number}.
    • It is usually more convenient to set {id} as the primary key because it is more convenient to use a smaller number of attributes.
    • The attribute selected as the primary key is underlined.
  • It is abbreviated as PK.

Unique key

A candidate key that is not the primary key. Sometimes called an alternate key.

  • For PLAYER(id, name, team_id, back_number, birth_date).
  • The unique key is {team_id, back_number}.
    • Since the id attribute is underscored, {id} is the primary key, so {team_id, back_number} is the unique key.

Constraints

A constraint that a relation in a relational database must always honor.

Implicit constraints

This is a constraint imposed by the relational data model itself.

  • A relation cannot have duplicate records.
  • You can't have properties with the same name within a relation.

Schema-based constraints

Constraints that can be specified directly in the schema, usually via DDL. Also called explicit constraints.
