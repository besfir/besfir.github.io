SQL

Structured Query Language (SQL) is the standard language for relational database management systems (RDBMS) used in the industry. It can also be referred to as a comprehensive DBMS language that combines DDL, DML, and VDL.

SQL Key Terms

The key terms are listed below, but they are more often used interchangeably, so keep them in mind.

Relational data model SQL
relation table
attribute column
tuple row or record
domain domain

What is a relation in SQL?

In a relational data model, a relation does not allow duplicate tuples. However, in SQL, a relation is a multiset (=bag) of tuples, which means it allows duplicate tuples. Therefore, we can understand that a table in SQL can have duplicate records.

SQL & RDBMS

SQL is the standard language for RDBMSs, but since there is no mandatory implementation, the specifications provided by each RDBMS are slightly different. For this reason, the standard SQL is called ANSI SQL or ISO SQL, and the implementation of each RDBMS is called vendor SQL. A vendor is a supplier of a product or service, in this case, a specific RDBMS manufacturer.

Defining a DB with MySQL

Let's see how to define a DB by creating an RDB for an IT company. In this article, we'll create a relational database where we can define information about departments, employees, and projects. For the tutorial, we'll use the InnoDB engine, the most popular of MySQL's engines.

Database vs Schema

In MySQL, database and schema are synonymous. Therefore, the schema keyword can be used in place of the database keyword, such as create schema besfir_company. On the other hand, in other RDBMSs, they have different meanings. For example, in PostgreSQL, schema is used for the namespace of the database.

-- in MySQL
create database besfir_company = create schema besfir_company

This also leads to differences in where tables are defined. In MySQL, tables are defined in the database, whereas in PostgreSQL, tables are defined in the schema within the database.

Defining tables

We learned in the primary key section of the previous relation schema article that if an attribute has an underscore, it is a primary key. With that in mind, let's check out the relation schema.

  • DEPARTMENT(id, name, leader_id)
    • id : Primary key for unique identification.
    • name : The name of the department.
    • leader_id : Unique identification value of the department leader. Refer to the id value of EMPLOYEE.
  • EMPLOYEE(id, name, birth_date, gender, position, salary, department_id)
    • id: Primary key for unique identification.
    • name : Employee's name.
    • birth_date: Employee's birthday.
    • gender: Employee's gender.
    • salary: Employee's salary.
    • department_id: Department unique identification value. Refer to id value of DEPARTMENT.
  • PROJECT(id, name, leader_id, start_date, end_date)
    • id: Primary key for unique identification.
    • name: Project name.
    • leader_id : Unique identification value of project manager. Refer to id value of EMPLOYEE.
    • start_date: Project start date.
    • end_date: Project end date.
  • WORKS_ON(employee_id, project_id)
    • employee_id : Unique identification value of an employee who participated in the project. Refer to the id value of EMPLOYEE.
    • project_id : Unique ID identification value of the project. Refer to the id value of PROJECT.

Creating the table

This time, we'll create the table we designed earlier using SQL.

create table department (
    id        int primary key,
    name      varchar(20) not null unique,
    leader_id int
);
  • The id column is an integer (int) data type that serves as the primary key and uniquely identifies each row.
  • The name column is a string (varchar) data type of up to 20 characters, does not accept NULL, and has a unique value.
  • The leader_id column is an integer (int) data type.
  • See also
    • Data types of attributes
    • Key constraints : Primary key
    • Key constraints : UNIQUE
    • NOT NULL constraint
create table employee (
  id            int primary key,
  name          varchar(30) not null,
  birth_date    date,
  gender        char(1) check ( gender in ('M', 'F') ),
  position      varchar(10),
  salary        int,
  department_id int,
  foreign key (department_id) references department (id)
    on delete set null on update cascade,
  check ( salary >= 50000000 )
);
  • The gender column accepts a letter as a value, and only accepts 'M' or 'F' as a value through attribute-level validation.
  • The part beginning with foreign key ~ is for a foreign key that references another table. The department_id column references the id column of the department table.
    • When a specific record is deleted from the department table, the value of the department_id column in the employee table that references it is modified to NULL.
    • If the id value in the department table changes, the department_id value in the employee table that references it is also modified.
  • The salary column is validated at the table level to allow values greater than 50000000.
  • See also
    • check constraint
    • Referential Integrity Constraints: Foreign Keys
create table project (
  id         int primary key,
  name       varchar(20) not null unique,
  leader_id  int,
  start_date date,
  end_date   date,
  foreign key (leader_id) references employee (id)
    on delete set null on update cascade,
  check ( start_date < project.end_date )
);
create table works_on (
  employee_id int,
  project_id  int,
  primary key (employee_id, project_id),
  foreign key (employee_id) references employee (id)
    on delete cascade on update cascade,
  foreign key (project_id) references project (id)
    on delete cascade on update cascade
);
  • The project and works_on tables will not be described because they overlap with the previous ones.

Modifying tables

We've defined all the tables we need, but we've missed something. We didn't declare a foreign key on the leader_id of the department table. This is because the employee table didn't exist when we created the department table, so we couldn't declare a foreign key on it.

This time, let's apply the foreign key using the alter table keyword, which modifies the table.

alter table department add foreign key (leader_id)
  references employee(id)
  on update cascade 
  on delete set null;

We are now done defining your tables. In the next post, you'll learn how to manipulate data through these tables.

Key constraints : Primary key

A primary key is used to identify a record in a table and consists of one or more attributes. The primary key must be unique and cannot have NULL as a value.

  • Additional explanation
    • When specifying a single attribute as the primary key, either method is acceptable.
    • When specifying more than one attribute as the primary key, it must be specified in the form primary key(attribute1, attributes...).

Key constraints : UNIQUE

Attributes with unique constraints cannot have duplicate values. However, the treatment of NULL values can vary depending on the RDBMS. While MySQL and PostgreSQL allow duplicates of NULLs for columns with unique constraints, Microsoft SQL Server (MsSQL) does not allow duplicates of NULLs and allows only one NULL to exist per column.

    • In this case, the department_id column of the EMPLOYEE table can only refer to values that exist in the id column of the DEPARTMENT table.

Here's how to declare a foreign key.

SQLreference_optionDescription
create table employee (
 ...
 department_id int,
 foreign key (department_id)
  references department(id)
  on delete reference_option
  on update reference_option
);		CASCADEReflects deletion/update of referenced value as is
SET NULLReflects deletion/update of referenced value as NULL
RESTRICTPrevents deletion/update of referenced value
NO ACTIONSimilar to RESTRICT
SET DEFAULTChanges referenced value to default on deletion/update
  • reference_option additional explanation.
    • CASCADE : When a record is deleted from DEPARTMENT, the referencing EMPLOYEE records are also deleted. If the id value of a DEPARTMENT record is changed to 1, the department_id value of the referencing EMPLOYEE records is also changed to 1.
    • SET NULL : When a DEPARTMENT record is deleted or its id value is changed, the department_id value of the referencing EMPLOYEE records is set to NULL.
    • RESTRICT : If the record is referenced by other tables, deletion of the DEPARTMENT record or modification of specific columns is prohibited.
    • NO ACTION : In MySQL, it is the same as RESTRICT. In standard SQL, within a transaction, deletion/modification is allowed, but it's prohibited after the transaction ends.
    • SET DEFAULT : When the referenced value is deleted/changed, it is changed to the default value. This is not supported in MySQL.
    • In MySQL, only CASCADE, SET NULL, and RESTRICT are supported. In PostgreSQL, all five options are supported.

Specifying constraint names

By naming them, you can easily identify which constraints have been violated. Also, if you want to delete a constraint, you can do so by its name. For MySQL, the validation message with and without constraints is shown below.

create table test (
 age int constraint age_over_19 check(age > 19)
);
When using the constraint name 'age_over_19'When omitting the constraint name
Check constraint 'age_over_19' is violated.Check constraint 'test_chk_1' is violated.
  • Additional explanation.
    • If you don't specify a constraint name, it becomes difficult to identify which constraint has been violated. In other words, it leads to a less intuitive design.
    • In the case of MySQL, if you want to know what the 'test_chk_1' constraint is, you can find out by entering show create table test;.

attribute DEFAULT

Used to define the default value of a attribute. When saving a new record, if there is no input value for the attribute, it will be saved with the default value.

Declarations at the attribute level
create table orders (
 id int primary key,
 price int default 10000,
 name varchar(20)
);
  • Additional explanation.
    • When using the query insert into orders(id, name) values (1, 'coffee'),
    • Since the price was not specified, the "price" attribute of the record will be inserted with the default value of 10,000.
