Since the version Chrome 111, we was added an interesting feature called the "View Transition API". In this article, we'll take a look at the View Transition API.

What is the View Transition API?

The View Transition API provides the ability to easily create animated transitions between different DOM states while updating DOM content at once. Let's take a look at some examples of where it can be used.

If the order of elements changes due to user interaction, it may be difficult to identify which elements have changed. In this case, a reordering animation can help the user recognize that the order has changed. In the example above, the View Transition API is applied to the draggable elements to make the reordering easier to recognize.

SSRs that utilize backend frameworks are MPAs, which means that it takes a lot more work to implement page transition animation effects compared to SPAs. However, by utilizing the View Transition API, it is relatively easy to implement page transition effects. In the demo video above, we applied the View Transition API to Express + Handlebars to implement the page transition effect, and the video plays uninterrupted even when the page is switched.

Try it out

💡 As this is still an experimental feature, it only works with Chrome, Edge, and Opera, which are Chromium Engine-based browsers.

For practice, we've created a simple example that toggles the shape of a shape when a button is clicked. We'll build on this example and add some code to demonstrate how to use it.

Apply View Transition API

The basic usage is really simple. Just update the DOM inside startViewTransition() and the transition will be applied immediately.

const shape = document.getElementById('shape')
shape.addEventListener('click', () => document.startViewTransition(handleClick))

function handleClick() {
  shape.classList.toggle('active')
}

We'll change from calling updateDOM() directly on click to updating via the View Transition API. If you click the button, you can see that the default transition effect, cross-fade, is applied.

How does it work?

The View Transition API can be summarized in the following steps.

  1. When startViewTransition() is called, the API takes a screenshot of the current page. 2.
  2. The callback passed to startViewTransition() is called to change the DOM.
  3. Capture the new state of the changed page as a live representation.
  4. Construct a pseudo-element tree with the following structure.
::view-transition
└─ ::view-transition-group(root)
   └─ ::view-transition-image-pair(root)
      ├─ ::view-transition-old(root)
      └─ ::view-transition-new(root)
  • The ::view-transition is the top-level pseudo-element that contains all View Transition pseudo-element overlays.
    • Adding the CSS property view-transition-name will create a ::view-transition-group(some-name) overlay with that name.
  • The ::view-transition-old(root) is a screenshot of the old page we worked on in 1, and the ::view-transition-new(root) is a live representation captured in 3.
    • Both of these render as content similar to <img> or <video>, so they can be styled with attributes like object-fit or object-position.
  1. The default effect, a cross-fade, is applied to ::view-transition-old(root) and ::view-transition-new(root) to create a smooth transition.

Custom transition

This time, let's replace the default transition with a different effect. We want the pre-update elements to disappear as they fade out to the left, and the post-update elements to appear as they fade in from the right.

@keyframes fade-in {
  from { opacity: 0; }
}

@keyframes fade-out {
  to { opacity: 0; }
}

@keyframes slide-from-right {
  from { transform: translateX(50px); }
}

@keyframes slide-to-left {
  to { transform: translateX(-50px); }
}

::view-transition-old(root) {
  animation: 190ms cubic-bezier(0.4, 0, 1, 1) both fade-out,
    500ms cubic-bezier(0.4, 0, 0.2, 1) both slide-to-left;
}

::view-transition-new(root) {
  animation: 310ms cubic-bezier(0, 0, 0.2, 1) 190ms both fade-in,
    500ms cubic-bezier(0.4, 0, 0.2, 1) both slide-from-right;
}

I hadn't noticed it before, but when I checked the results, I realized that the transition effect was also applied to the header. It's a little awkward to have transitions applied to the header when you're navigating through a web page.

Different effects for different elements

The previous example pointed out some awkwardness, so let's create two requirements and modify them accordingly.

  • I don't want to apply a transition to the header before and after a DOM update because it doesn't change.
  • I want the button to move naturally as the size of the shape changes.
header{
  view-transition-name: header;
}

#activator{
  view-transition-name: activator;
}


@keyframes fade-in {
  from { opacity: 0; }
}

@keyframes fade-out {
  to { opacity: 0; }
}

@keyframes slide-from-right {
  from { transform: translateX(50px); }
}

@keyframes slide-to-left {
  to { transform: translateX(-50px); }
}

::view-transition-old(root) {
  animation: 190ms cubic-bezier(0.4, 0, 1, 1) both fade-out,
  500ms cubic-bezier(0.4, 0, 0.2, 1) both slide-to-left;
}

::view-transition-new(root) {
  animation: 310ms cubic-bezier(0, 0, 0.2, 1) 190ms both fade-in,
  500ms cubic-bezier(0.4, 0, 0.2, 1) both slide-from-right;
}

How does it work?

If you give it a view-transition-name value, it will smoothly transition to the same element before and after the DOM update.

  • Headers will remain as they are because there is no pre- or post-DOM update call to smoothly transition them.
  • The button will be repositioned up or down after the DOM update, depending on the size of the shape, so it will be moved to provide a smooth transition.

As mentioned earlier, specifying a view-transition-name value creates a ::view-transition-group(some-name) corresponding to that value. This makes it possible to implement different effects for different elements because they act as separate overlays.

::view-transition
├─ ::view-transition-group(root)
│  └─ ::view-transition-image-pair(root)
│     ├─ ::view-transition-old(root)
│     └─ ::view-transition-new(root)
├─ ::view-transition-group(header)
│  └─ ::view-transition-image-pair(header)
│     ├─ ::view-transition-old(header)
│     └─ ::view-transition-new(header)
└─ ::view-transition-group(activator)
   └─ ::view-transition-image-pair(activator)
      ├─ ::view-transition-old(activator)
      └─ ::view-transition-new(activator)

The view-transition-name value must be unique, and if there are duplicate values within a single page, the transition effect will be ignored. You can also give shapes a unique view-transition-name value and see what happens.
