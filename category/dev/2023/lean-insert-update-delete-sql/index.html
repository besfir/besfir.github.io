Insert data

The SQL statement that adds data is called an insert statement or insert query, and there are three ways to use it. There are three ways to use it: comma-separated all values, attribute list with value list, and batch insert.

Comma-separated all value

To use the way that uses all comma-separated values, write SQL like this

insert into table_name values (comma, separated, all, values);

The order of the values you input follows the order of the attributes when you created the table. Let's use the employee table we created in the previous post as an example.

create table employee (
  id            int primary key,
  name          varchar(30) not null,
  birth_date    date,
  gender        char(1) check ( gender in ('M', 'F') ),
  position      varchar(10),
  salary        int,
  department_id int,
  foreign key (department_id) references department (id)
    on delete set null on update cascade,
  check ( salary >= 50000000 )
);

The attributes of the employee table are declared in the following order: id, name, birth_date, gender, position, salary, department_id. Let's insert the data in this order.

insert into employee
values (1, 'John', '1987-02-01', 'M', 'DEV_BACK', 100000000, null);
output
Query OK, 1 row affected (0.00 sec)
select * from employee;
output
+----+------+------------+--------+----------+-----------+---------------+
| id | name | birth_date | gender | position | salary    | department_id |
+----+------+------------+--------+----------+-----------+---------------+
|  1 | John | 1987-02-01 | M      | DEV_BACK | 100000000 |          NULL |
+----+------+------------+--------+----------+-----------+---------------+
1 row in set (0.00 sec)
  • Consider the insert statements in lines 1 and 2. We've specified the values in the order of the aforementioned attributes and returned a Query OK result, so the data should have been successfully inserted.
    • One thing to note is that the value of department_id is null. This is because the foreign key must refer to an existing value, but since there is no data in the department table yet, it is temporarily set to null.
  • The select statement in line 5 is the SQL that reads the value, which we'll cover in the next post.

Earlier, we saw a case where data is inserted normally. This time, we'll create a case that violates a constraint, see the error message, and troubleshoot it.

insert into employee
values (1, 'Jane', '1996-05-05', 'F', 'DSGN', 90000000, null);
output
ERROR 1062 (23000): Duplicate entry '1' for key 'employee.PRIMARY'
  • An error occurred, and when I read the error message, I realized that the value of the primary key I was entering was a duplicate of a previously entered value, which violated the unique constraint.
  • So let's modify the id value to 2 and reinsert it.
insert into employee
values (2, 'Jane', '1996-05-05', 'F', 'DSGN', 40000000, null);
output
ERROR 3819 (HY000): Check constraint 'employee_chk_2' is violated.
  • Once again, an error occurred. Reading the error message, we see that a check constraint was violated.
  • Since we used check constraints on both attributes, gender and salary, let's assume that we don't know which constraint was violated.
  • So let's check the name of the constraint using show create table table_name, which we learned previous article.
show create table employee;
output

+----------+----------------------------------------------------------------------------------------------------------------------+

1 row in set (0.00 sec)
  • As it turns out, the error was caused by the value of salary being less than 50,000,000.
  • So let's change the value of salary to at least 50,000,000 and reinsert it.
insert into employee 
values (2, 'Jane', '1996-05-05', 'F', 'DSGN', 90000000, 1);
output
ERROR 1452 (23000): Cannot add or update a child row:
a foreign key constraint fails (`besfir_company`.`employee`, CONSTRAINT 
  `employee_ibfk_1` FOREIGN KEY (`department_id`) REFERENCES

  • This time, the query failed because it violated a foreign key constraint.
  • As mentioned earlier, the value of department_id must be a value that exists in the department table, but it doesn't yet, which violates the referential integrity constraint.
  • Since we don't have a value in the department table yet, we'll modify the value to null and reinsert it.
insert into employee

Attribute list with value list

To specify a list of attributes, use the following SQL statement

insert into table_name (comma, separated, attribute, list) 
values (comma, separated, values, in, attribute, order);

Whereas the previous way requires you to specify values based on the table's definition, this way gives you flexibility in how you specify values. Instead of having to enter all the values defined in the table, you can selectively insert only the properties you want, and you can reorder them.

insert into employee (name, birth_date, gender, position, id)
values ('Smith', '2000-10-12', 'F', 'DEV_BACK', 3);
output
Query OK, 1 row affected (0.00 sec)
select * from employee;
output
+----+-------+------------+--------+----------+-----------+---------------+
| id | name  | birth_date | gender | position | salary    | department_id |
+----+-------+------------+--------+----------+-----------+---------------+
|  1 | John  | 1987-02-01 | M      | DEV_BACK | 100000000 |          NULL |
|  2 | Jane  | 1996-05-05 | F      | DSGN     |  90000000 |          NULL |
|  3 | Smith | 2000-10-12 | F      | DEV_BACK |  50000000 |          NULL |
+----+-------+------------+--------+----------+-----------+---------------+
3 rows in set (0.00 sec)
  • We have entered only some of the attributes defined in the table, and we are not following the order defined in the table.
  • We are entering values in the order of the selected attributes.
  • We haven't entered a value for salary, but 50,000,000 is entered.
    • This is because we specified a default value of default 50000000 when we defined the table.

Bulk Insert

Use the following bulk insert SQL statement

insert into table_name (comma, separated, attribute, list)
    values
        (comma_1, separated_1, values_1, in_1, attribute_1, order_1),
        (comma_2, separated_2, values_2, in_2, attribute_2, order_2),
        ...;

Update data

When updating data, write SQL like this

update table_name(s) 
set attribute = value [, attribute = value, ...]
[ where condition(s) ];
  • You can specify one or more table names.
  • The attributes and values you want to modify act as a pair, and you can also specify more than one.
    • If you specify multiple tables, you may get an ambiguous error unless you explicitly specify the properties in the form table_name.attribute.
  • The where clause specifies the condition of the records to be targeted and can be omitted or multiple.
    • If you omit it, all records will be updated.

Now that we have stored the values in all tables, we can assign a value to the department_id attribute of the employee table. Let's use an update query to modify the value of the department_id attribute of the first record with an id value of 1.

update employee
set department_id = 1003
where id = 1;
output
Query OK, 1 row affected (0.01 sec)
Rows matched: 1  Changed: 1  Warnings: 0

The result was returned with a message that 1 row was affected. Let's verify that this is indeed the case.

select * from employee where id = 1;
output

Now let's create a hypothetical scenario and practice. The head of Company wants to increase the salary of all employees in the development department to double their current salary. Let's reflect this with an update query.

update employee
set salary = salary * 2
where department_id = 1003;
output
Query OK, 5 rows affected (0.01 sec)
Rows matched: 5  Changed: 5  Warnings: 0

The resulting message shows that five records have been modified. This time, we're going to double the salary of all employees working on a project with a project ID of 2003.

update employee, works_on
set salary = salary * 2
where id = employee_id
  and project_id = 2003;

Since the employee and works_on tables do not have overlapping attribute names, the above expression is fine. However, when using multiple tables, overlapping attribute names can lead to ambiguous errors, and it is often difficult to tell which table's attribute belongs to which table by looking at the table specification. Therefore, when using multiple tables, it is better to specify the table names and their attributes as shown below.

update employee, works_on
set employee.salary = employee.salary * 2
where employee.id = works_on.employee_id
  and works_on.project_id = 2003;
  • Line 1: To update the salary, we need to select the employee table, but we don't know the number of projects each employee is working on.
    • So we do an implicit join between the employee table, which contains the salary information, and the works_on table, which contains the employee and project numbers.
    • The resulting table is a Cartesian product, which is quite far from reality. If you're curious, you can try select * from employee, works_on;.
  • Line 3: Select only the information where employee.id = works_on.employee_id from the table represented by the Cartesian product to get the information that matches reality.
  • Line 4 : Select only the records where project_id is 2003.

This time, we're going to double the salaries of all employees.

update employee
set salary = salary * 2;

Since we're not restricting the condition with a WHERE clause, it will be reflected on all records in the table.

Delete data

When deleting data, write SQL like this

delete from table_name
[ where condition(s) ]

You should also be careful with delete queries, as they can delete all records if you don't specify a where clause.

Let's practice this by creating a hypothetical scenario. Nicole is leaving the company and you need to delete her information from all tables. Nicole's employee ID is 8 and she was part of project 2002.

delete from employee where id = 10;
output
Query OK, 1 row affected (0.01 sec)
select * from employee where id = 10;
output
Empty set (0.00 sec)
  • The tables we need to delete information from are employee and works_on.
  • We first deleted Nicole's information from the employee table.
  • But does the works_on table also need to be deleted?
select * from works_on where employee_id = 10;
  • The lookup shows that Nicole's information already deleted in the works_on table.
    • The reason is the reference option of the foreign key.
create table works_on (
  employee_id int,
  project_id  int,
  primary key (employee_id, project_id),
  foreign key (employee_id) references employee (id)
    on delete cascade on update cascade,
  foreign key (project_id) references project (id)
    on delete cascade on update cascade
);

This is because when we created the table, we specified the cascade option so that when material is deleted from the referenced table, it is also deleted from the works_on table.

Jane is taking a leave of absence and is dropping out of her current project. Jane's ID is 2, and she was a participant in projects 2001, 2002, and 2003.

delete from works_on where employee_id = 2;
  • Instead of having to specify each project_id, you can conditionally delete the employee_id and do it all at once.

Messi is currently working on three projects and has decided to drop out of projects 2002 and 2003 to focus on project 2001. Messi's ID is 5.

delete from works_on where employee_id = 5 and project_id = 2002
delete from works_on where employee_id = 5 and project_id = 2003
  • We could run a delete query for every project_id you want to delete.
    • However, this leaves a lot of room for error as the number of records we need to delete grows.
  • Use the not equal operator.
delete from works_on where employee_id = 5 and project_id <> 2001
-- or
delete from works_on where employee_id = 5 and project_id != 2001
  • not equal allows you to delete only records with project IDs other than 2001 at once.

The company has a big problem and closes all projects.

delete from works_on
  • If you don't use a where clause, you can delete the entire data in the table.
