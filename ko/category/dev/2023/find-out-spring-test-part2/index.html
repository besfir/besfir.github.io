이번 글에서는 @WebMvcTest, MockMvc, TestRestTemplate 그리고 WebTestClient에 대해서 알아보겠다. TestRestTemplate와 WebTestClient은 각각 RestTemplate 또는 WebClient와 어떻게 다른지 알아보고 사용 빙법에 대해서 간단히 알아볼 예정이다.

@WebMvcTest

웹 계층에 집중해서 테스트를 수행하기 위해 사용하며, 전체 자동 구성이 비활성화되고 대신 MVC 테스트와 관련된 구성만 적용된다. MVC 테스트와 관련된 구성이라 함은 @Controller, @ControllerAdvice, @JsonComponent, Converter/GenericConverter, Filter, WebMvcConfigurer 및 (@Component, @Service 또는 @Repository이 아닌) HandlerMethodArgumentResolver bean을 말한다.

controllers

테스트 대상 컨트롤러를 지정하는 속성이다. 명시적으로 컨트롤러가 지정되지 않으면 모든 @Controller를 bean으로 등록한다. @Controller 외에 다른 컴포넌트 클래스를 bean으로 등록해야 할 경우, @Import를 함께 사용할 수 있다.

properties

이전 글에서 다루었던 @TestPropertySource 의 속성과 동일하다. 테스트를 실행하기 전에 스프링 환경에 추가해야 하는 key=value 형식의 속성이다.

useDefaultFilters

@SpringBootApplication에 사용되는 필터를 테스트에도 적용할지 여부를 지정하는 속성이다. 기본값은 true이며, @Controller, @ControllerAdvice 그리고 WebMvcConfigurer bean 들이 포함된다.

includeFilters

사용시 특정 조건을 만족하는 컴포넌트 클래스들만 bean으로 등록할 수 있다. 즉, 특정 타입이나 어노테이션 등을 기준으로, bean으로 등록할 컴포넌트 클래스를 필터링할 수 있다.

@WebMvcTest(
  includeFilters = @ComponentScan.Filter(
    type = FilterType.ANNOTATION,
    classes = {Service.class, Repository.class}
  )
)

excludeFilters

includeFilters 속성과 반대로, bean으로 등록하지 않을 컴포넌트 클래스를 필터링하기 위한 속성이다.

@WebMvcTest(
  excludeFilters = @ComponentScan.Filter(
    type = FilterType.ASSIGNABLE_TYPE,
    classes = {GenericController.class}
  )
)

excludeAutoConfiguration

특정 자동 구성 클래스들을 제외하고 싶을 때 사용하는 속성입니다. 아래 예제의 경우, 자동 구성에서 스프링 시큐리티 제외한다.

@WebMvcTest(excludeAutoConfiguration = SecurityAutoConfiguration.class)
class WebLayerTest4 { ... }

MockMvc

실제 서버를 실행하지 않고, 컨트롤러를 테스트할 수 있는 환경을 제공한다. 요청을 시뮬레이션 하기 위해 perform()을 이용해서 Dispatcher Servlet을 직접 호출하며, 요청 정보를 담고 있는 MockMvcRequest을 파라미터로 넘긴다. 아래 예제와 함께 컨트롤러를 테스트할 때 대략적인 작동 흐름을 알아보자.

@RestController
public class FooController {

  @GetMapping("/")
  public String foo() {
    return "bar";
  }
}
class WebLayerTest {

  MockMvc mockMvc;

  @BeforeEach
  void setup() {
    this.mockMvc = MockMvcBuilders
      .standaloneSetup(new FooController())
      .build();
  }

  @Test
  void helloTest() throws Exception {
    ResultActions resultActions = mockMvc.perform(MockMvcRequestBuilders.get("/"));
    
    MvcResult mvcResult = resultActions.andReturn();
    assertThat(mvcResult.getResponse().getContentAsString()).isEqualTo("bar");
  }
}
  • 7-9행 : MockMvcBuilders을 통해서 테스트용 서블릿 환경을 구성한다.
  • 14행 : perform()을 호출해 컨트롤러로 요청을 보낸다.
    • 앞서 설명한 것처럼 실제 서버가 기동되는 것이 아니기 때문에 DispatcherServlet을 직접 호출하며, mocking한 요청 및 응답 객체로 통신한다.
    • 이때, DispatcherServlet은 테스트용으로 확장한 TestDispatcherServlet이다.
  • 16-17행 : MockMvc가 반환하는 mock 응답을 통해서 결과를 검증한다.

초기화

위의 예제는 작동 흐름을 설명하기 위해서 Spring Context를 이용하지 않고 MockMvc를 직접 초기화 하는 방법을 사용했다. 그 밖에 Web Application Context를 이용해서 직접 초기화하는 방법이나 스프링 부트 자동 구성을 이용하는 방법도 있다. MockMvc의 이해도를 높이기 위해 초기화 하는 방법들을 알아보자.

아래는 이번 단원에서 사용될 공통 구성이다.

@RestController
public class HelloController {

  @GetMapping("/")
  public String hello() {
    return "Hello!!";
  }

  @GetMapping("/redirect")
  public void redirect(HttpServletResponse response) throws IOException {
    response.sendRedirect("/");
  }

  @GetMapping("/exception")
  public void exception() throws Exception {
    throw new Exception();
  }
}
@ControllerAdvice
public class GlobalExceptionHandler {

  @ExceptionHandler(Exception.class)
  protected ResponseEntity<String> handleException(Exception e) {
      return new ResponseEntity<>("response", HttpStatus.INTERNAL_SERVER_ERROR);
  }
}
  • standaloneSetup()

MockMvc 인스턴스를 생성하는 MockMvcBuilders의 정적 팩토리 메소드이며, Spring Context를 사용하지 않고 Web Application Context를 Stubbing해서 사용한다. 따라서 스프링 프레임워크의 기능들을 사용하지 못한다.

import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.*;

class WebLayerTest {

  MockMvc mockMvc;

  @BeforeEach
  void setup() {
    this.mockMvc = MockMvcBuilders
      .standaloneSetup(new HelloController())
      .setControllerAdvice(new GlobalExceptionHandler())
      .alwaysDo(print())
      .build();
  }

  // 중복되기 때문에, 이후부터 아래는 생략한다.
  @Test
  void helloTest() throws Exception {
    ResultActions resultActions = mockMvc.perform(get("/"));
    MvcResult mvcResult = resultActions.andReturn();
    assertThat(mvcResult.getResponse().getContentAsString()).isEqualTo("Hello!!");
  }

  @Test
  void redirectTest() throws Exception {
    ResultActions resultActions = mockMvc.perform(get("/redirect"));
    MvcResult mvcResult = resultActions.andReturn();
    assertThat(mvcResult.getResponse().getStatus()).isEqualTo(302);
  }

  @Test
  void exceptionTest() throws Exception {
      mockMvc
        .perform(get("/exception"))
        .andExpect((result)-> assertThat(result.getResolvedException()).isExactlyInstanceOf(Exception.class));
  }
}
  • webAppContextSetup()

실제 Web Application Context를 이용해서 MockMvc 인스턴스를 생성하는, MockMvcBuilders의 정적 팩토리 메소드이다. Spring Context를 사용하기 때문에 스프링 프레임워크의 기능을 사용할 수 있다.

@WebAppConfiguration
@ContextConfiguration(classes = UnitTestApplication.class)
@ExtendWith(SpringExtension.class)
class WebLayer2Test {

  @Autowired
  WebApplicationContext context;

  MockMvc mockMvc;

  @BeforeEach
  void setup() {
    this.mockMvc = MockMvcBuilders.webAppContextSetup(context).alwaysDo(print()).build();
  }

  // 중복되기 때문에, 아래는 생략.
  ...
}

1행 : 9행에서 주입받아야 할 WebApplicationContext를 빈으로 등록하는 역할을 한다.
2행 : @SpringBootApplication 어노테이션이 달린 클래스를 지정해서 실제 Spring Context를 생성하도록 한다.
13행 : WebApplicationContext을 이용해서 MockMvc를 초기화 한다.

❓ 이전 장에서 정리했던 @ContextConfiguration의 특성대로라면 classes 속성을 설정하지 않아도 작동해야 될 것 같은데, 명시적으로 classes 속성을 지정하지 않으면 Spring Context가 생성되지 않는다.

  • @AutoConfigureMockMvc

클래스 수준에 선언하며, 스프링 부트에서 자동으로 MockMvc를 초기화 한 뒤 bean으로 등록해준다. 우리는 편하게 이를 주입해서 사용하면 된다.

@SpringBootTest
@AutoConfigureMockMvc
class WebLayer3Test {

  @Autowired
  MockMvc mockMvc;

  // 중복되기 때문에, 아래는 생략.
  ...
}

@WebMvcTest의 경우, 내부에 @AutoConfigureMockMvc이 선언되어 있으므로 따로 선언할 필요는 없다.

@WebMvcTest
class WebLayer4Test {

  @Autowired
  MockMvc mockMvc;

  // 중복되기 때문에, 아래는 생략.
  ...

MockHttpServletRequestBuilder

MockMvc를 통해 요청을 보낼 때, mocking한 요청 객체를 생성하기 위한 빌더이다. 이 빌더는 정적 팩토리 메소드를 통해서 생성할 수 있으며, 그 종류는 다음과 같다.

MethodText
get()get 요청에 대한 빌더를 생성한다.
post()post 요청에 대한 빌더를 생성한다.
put()put 요청에 대한 빌더를 생성한다.
patch()patch 요청에 대한 빌더를 생성한다.
delete()delete 요청에 대한 빌더를 생성한다.
options()options 요청에 대한 빌더를 생성한다.
head()head 요청에 대한 빌더를 생성한다.
request()주어진 HTTP method로 빌더를 생성한다.
multipart()multipart 요청에 대한 빌더를 생성한다.
asyncDispatch()비동기 처리를 시작한 RequestBuilder를 생성한다.

사용법은 URI 또는 문자열을 파라미터로 넘기면 된다. 문자열을 파라미터로 넘겨 빌더 생성시, url variable의 순서에 따라서 치환되니 이점 주의하도록 한다.

    MockHttpServletRequestBuilder request = post(string);

MethodText

accept()요청에 지정된 accept 헤더를 설정한다.
header()(key, values...) 형태로 헤더를 요청에 추가한다.

MultiValueMap을 이용한 queryParam(String, String...)의 변형이다.
cookie()요청에 쿠키를 추가한다.
locale()하나 또는 여러 로케일을 설정한다.

session()여러 요청에서 재사용 가능한 HTTP session 객체를 사용한다.
principal()Spring secutity에서 사용되는 principal을 설정한다.

데이터를 추가하는 메소드의 경우 이름을 통해서 추론하기는 어렵기 때문에, javadoc 기준으로 데이터를 추가하는 메소드는 단어에 강조 표시를 했다. 메소드명이 하는 일을 명시적으로 나타내기 때문에 빈번히 사용될 것 같은 예시만 작성해봤다.

@Test
void postTest() throws Exception {
  AddUserRequest requestDto = new AddUserRequest("test@mail.com", "123",15);
  String content = objectMapper.writeValueAsString(requestDto);
  MockHttpServletRequestBuilder requestBuilder = post("/post")
    .content(content)
    .contentType(MediaType.APPLICATION_JSON);
    
  MvcResult mvcResult = mockMvc.perform(requestBuilder)
    .andReturn();
    
  assertThat(mvcResult.getResponse().getStatus()).isEqualTo(200);
}

MockMvcResultHandlers

응답 결과에 어떤 동작을 부여하기 위한 정적 팩토리 메소드이다. ResultActions의 andDo()와 함께 쓰인다.

  • log() : DEBUG 레벨로 MvcResult 세부 정보 로그를 출력한다.
  • print() : 표준 출력 스트림에 MvcResult 세부 정보를 출력한다.
@Test
void helloTest() throws Exception {
  ResultActions resultActions = mockMvc.perform(get("/")).andDo(print());
  MvcResult mvcResult = resultActions.andReturn();
  assertThat(mvcResult.getResponse().getContentAsString()).isEqualTo("Hello world");
}

ResultActions

MockMvc의 실행 결과에 검증과 같은 작업을 적용할 수 있다.

  • andExpect()

실행 결과를 검증을 하며, jsonPath 표현식을 이용해서 Json 문자열을 검증할 수 있다.

import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.*;

mockMvc.perform(get("/member/1"))
  .andExpect(status().isOk())
  .andExpect(content().contentType(MediaType.APPLICATION_JSON))
  .andExpect(jsonPath("$.member.name").value("Walter"));
  • andExpectAll()

JUnit의 assertAll()처럼 여러 검증을 수행하며, 하나의 검증이 실패하더라도 종료하지 않고 모두 검증하는 기능이다.

mockMvc.perform(get("/member/1"))
  .andExpectAll(
      status().isOk(),
      content().contentType(MediaType.APPLICATION_JSON),
      jsonPath("$.member.name").value("Walter")
  );
  • andDo()

일반 작업을 수행한다.

mockMvc.perform(get("/member/1")).andDo(print());
  • andReturn()

실행된 MockMvc 요청의 결과를 반환하여 직접 결과에 엑세스 할 수 있다.

ResultActions resultActions = mockMvc.perform(get("/")).andDo(print());
MvcResult mvcResult = resultActions.andReturn();
assertThat(mvcResult.getResponse().getContentAsString()).isEqualTo("Hello world");

MvcResult

실행된 MockMvc 요청의 결과에 대한 접근을 제공한다. 많은 API가 있지만 일반적으로 getResponse()을 이용해서 응답에 직접 접근하는 용도로 사용한다.

AddUserRequest request = new AddUserRequest("nomail@email.com", "password", 15);
ObjectMapper objectMapper = new ObjectMapper();

MvcResult mvcResult = mockMvc.perform(
    post("/post")
      .content(objectMapper.writeValueAsBytes(request))
      .contentType(MediaType.APPLICATION_JSON)
  )
  .andReturn();

String content = mvcResult.getResponse().getContentAsString();
AddUserResponse response = objectMapper.readValue(content, AddUserResponse.class);
assertThat(response).isNotNull(); // assert what you need

자세한 설명이 필요하다면 MvcResult Javadoc을 참조하도록 한다.

TestRestTemplate

테스트용으로 특화된 RestTemplate으로, 라이브 서버에 요청하기 위해 사용되기 때문에 @SpringBootTest의 WebEnvironment.RANDOM_PORT 속성과 함께 통합 테스트에 사용된다. 테스트시 RestTemplate의 대안이지만, 주입시 문제가 생길 것을 우려해 RestTemplate을 확장하지는 않는다.

기본 인증을 위해, 지정된 자격증명을 이용해서 템플릿을 생성할 수 있는 생성자를 제공한다. 이 인스턴스를 사용하여 수행되는 모든 요청은 제공된 credential을 사용해서 인증된다.

TestRestTemplate testRestTemplate = new TestRestTemplate("username", "password");
ResponseEntity<String> response = testRestTemplate.getForEntity("/member/info", String.class);

Assertions.assertEquals(response.getStatusCode(), HttpStatus.OK);

WebTestClient

테스트용으로 특화된 WebClient로, 라이브 서버, mock 환경, 비동기 요청 그리고 동기 요청 모두 사용 가능하다. 다만 mock 환경을 구성하는 것은 지원하지 않기 때문에, mock 환경에서 클라이언트로 사용하기 위해선 MockMvc를 초기화해서 환경 구성을 해야한다.

@AutoConfigureWebTestClient 어노테이션을 사용해야 WebTestClient를 bean으로 등록되지만, @WebMvcTest는 자동 구성에 포함되어 있기 때문에 바로 주입해서 사용하면 된다.

Synchronous mock env
@SpringBootTest
@AutoConfigureMockMvc
@AutoConfigureWebTestClient
class WebClientTest {

  @Autowired
  WebTestClient webClient;

  @Test
  void synchronousTest() {
    String response = webClient.get()
      .uri("/")
      .exchange()
      .expectBody(String.class)
      .returnResult()
      .getResponseBody();
    assertThat(response).isEqualTo("Hello world");
  }
