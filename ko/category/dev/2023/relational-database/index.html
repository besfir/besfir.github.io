이번 글에서는 relation(관계), primary key(기본키), foreign key(외래키), constraints(제약사항) 등을 포함한 관계형 데이터베이스(relational database)의 개념에 대해 알아보겠다.
관계형 데이터베이스의 개념은 수학에서 유래한 것이므로 이해하기 쉽도록 수학에서 관계형 데이터베이스 개념을 먼저 살펴보는 것으로 시작하도록 하겠다.

집합 (Set)

집합은 서로 다른 원소를 가지는 모임을 뜻하는 말로, 하나의 집합에서 요소의 순서는 중요하지 않다는 특징이 있다.

Example
{1, 3, 11, 4, 7}
  • 위의 집합은 중복이 없기 때문에 집합의 정의에 만족한다.
  • {11, 3, 4, 1, 7}이나 {4, 11, 3, 1, 7} 등으로 표현해도 상관없다.
    • 순서가 중요하지 않다는 특징으로 인해서, 바뀐 순서 때문에 의미가 바뀌지는 않기 때문이다.

수학에서의 relation

위 그림의 집합 A와 B에서 각각 하나의 원소를 선택해 표현할 수 있는 모든 집합을 카르테시안 곱(cartesian product)라 하고, 기호로는 A × B로 표현할 수 있다. 갑자기 카르테시안 곱을 언급한 이유는 수학에서 말하는 relation과 관련이 있기 때문이다.

2개의 집합에서 각각 하나의 요소를 선택해 관계를 맺는 것을 이항관계(binary relation)라 한다. 위 그림을 통해 이항관계에 대해 알아보자.

  • 이항관계를 기호 R로 표현하겠다.
  • R = {{1, p}, {2, q}, {2, r}}
    • 이때, R의 부분집합 {1, p}, {2, q}, {2, r}은 각각을 순서쌍(ordered pair)이라 부른다.
  • A × B = {{1, p}, {1, q}, {1, r}, {2, p}, {2, q}, {2, r}}
  • 이를 통해, 이항관계 R은 집합 A, B의 카르테시안 곱의 부분집합인 것을 알 수 있다.
    • 기호로는 R ⊆ A × B로 표현한다.

집합이 n개라면 n항관계(n-ary relation)라 부르며, n개의 집합에 대한 카르테시안 곱의 부분집합이 된다. 이번엔 일반화하여 n항관계를 정리해보자.

  • n항관계 R = {{1, p, ..., ɑ}, {2, q, ..., β}, {2, r, ..., β}}이라고 가정해보자.
  • R은 3개의 순서쌍 집합(ordered pair set)으로 이루어졌다.
  • R의 부분집합인 {1, p, ..., ɑ}, {2, q, ..., β}, {2, r, ..., β}은 각각을 튜플(tuple)이라고 부른다.
    • 이 경우 집합이 n개, 즉 n개의 요소로 이루어진 집합이기 때문에 n-tuple이라고 부른다.
    • 튜플은 2개 이상의 집합들 간의 연관을 나타내는데 사용되는 개념이다.
  • 요약하자면, n항관계 R은 3개의 순서쌍 집합으로 이루어진 n-tuple이다.

위의 정리한 내용들을 근거로 수학에서의 relation은 다음과 같이 표현할 수 있다.

  • 카르테시안 곱의 부분집합이다.
  • 튜플의 집합이다.

관계형 데이터 모델에서의 relation

수학에서의 관계를 살펴봤으니, 이를 관계형 데이터 모델에 적용해보자. 수학에서의 집합, 집합의 이름, 튜플, relation은 각각 관계형 데이터 모델에서 도메인(domain), 도메인의 이름, 튜플, relation에 대응한다. 여기서 도메인은 속성(attribute) 값의 집합을 의미한다. 좀 더 이해하기 쉽도록 학생 relation 예제를 만들어 살펴보도록 하자.

도메인 정의하기

학생 도메인을 다음과 같이 정의해 보겠다.

  • students_ids : 학번 집합, 7자리 정수.
  • human_names : 사람 이름 집합, 문자열.
  • university_grades : 대학교 학년 집합, {1,2,3,4}
  • major_names : 대학교에서 배우는 전공 이름 집합.
  • phone_numbers : 핸드폰 번호 집합.

앞서, 도메인은 속성 값들의 집합이라고 이야기 했다. 그런데 위의 도메인 정의를 살펴보면, 값의 타입을 지정하고 있다는 것을 알 수 있다. students_ids는 7자리 정수, human_names는 문자열 그리고 phone_numbers는 정의하기에 따라 문자열 또는 자연수를 지정하는 등 값의 타입을 지정하고 있다. 또한 공통적으로 집합의 특성을 그대로 물려받기 때문에 중복된 데이터를 사용할 수 없는데, 이 역시도 중복을 허용하지 않는 타입으로 볼 수 있다.

이렇듯 도메인은 각 속성이 가질 수 있는 값들의 타입을 정의하는 것으로도 볼 수 있다.

도식으로 표현하기

앞서 정의한 도메인을 이용해서 학생 relation을 표현할 것이다. 아직 우리는 앞서 연습한 수학적으로 표현하는 것 외에는 relation을 표현하는 방법을 모르기 때문에 최대한 수학적으로 표현해 보겠다.

만들다보니 특이사항이 생겼다. 학생의 전화번호 외에도 비상 연락망이 필요함을 느꼈다. 따라서 이를 표현하기 위해 전화번호 도메인을 두 번 사용했다. 하지만 도메인만으로는 어떤 도메인이 학생의 전화번호 인지 혹은 비상 연락망인지 구분을 할 수 없다.

앞서 도메인은 속성 값의 타입을 정의하는 것이라고도 볼 수 있다고 했는데, 이 때 값의 타입은 도메인명을 통해 지정된다. 따라서 도메인명은 해당 도메인의 타입을 지정할 뿐, 사용 목적을 나타낼 수 없다. 여기서 속성(attribute)가 등장한다. 속성은 해당 도메인이 relation에서 맡은 역할의 이름을 표현하기 위해서 사용된다. 위 그림에서는 속성을 사용함으로써 동일한 전화번호 도메인을 사용하는 학생의 전화번호와 비상 연락처를 구분할 수 있게 됐다.

이전 수학에서의 relation에서 살펴봤던 것처럼 각 집합을 간의 연관을 튜플로 표현할 수 있고 그림에서는 6-tuple이 된다. 튜플의 차수가 올라가고 순서쌍의 개수가 많아질수록 수학적으로 튜플을 표현하는 것은 굉장히 복잡해진다. 따라서, 관계형 데이터 모델에서는 테이블로 relation을 표현하는데, 이는 튜플을 가장 쉽게 표현할 수 있는 도식이 바로 테이블이기 때문이다.

위처럼 관계가 테이블을 뜻하는 추상적인 개념인 경우도 있지만, 아래처럼 관계 상태(relation state)를 뜻하는 경우도 있다. 이 경우 relation은 임의의 시점에서의 튜플의 집합, 즉 임의의 시점에서의 실제 데이터를 의미한다. 다만, 의미의 차이가 미미하기 때문에 둘 중 어느 것으로 이해해도 문맥을 이해하는 데에는 어려움은 없을 것이다.

주요개념들을 정리해보자.

  • 도메인 (domain) : 더 이상 나눌 수 없는 원자적인 값들의 집합으로, 데이터 타입과 속성에 대한 제약 조건을 정의한다. 속성에 할당할 수 있는 유효한 값들을 명시한다.
  • 도메인명 (domain name) : 도메인에 할당된 이름으로, 도메인의 값 범위를 정의하고, 해당 속성의 값들을 제한하는데 사용된다.
  • 속성 (attribute) : 도메인이 관계(relation)에서 맡은 역할을 이름으로 나타낸다. 각 속성은 해당 테이블의 특정 열을 나타내며, 해당 도메인에 정의된 값들만을 가질 수 있다.
  • 튜플 (tuple) : 테이블의 각 행을 나타내는 데이터 단위이며, 각 속성의 값을 포함하는 순서쌍(ordered pair) 값이다. 레코드(record)라고도 불리며, 일부 값은 NULL 일 수 있다.
    • 필자는 이후 관계형 데이터 모델에서 행을 지칭할 때는 레코드라고 부를 것이며, 이를 프로그래밍 언어에 가져왔을 때는 튜플이라고 부르겠다.
  • relation 이름 (relation name) : 해당 테이블이 어떤 주제나 개체들에 대한 정보를 담고 있는지를 나타내는 역할로, 테이블의 이름으로 사용된다.
  • relation : 튜플의 집합(=관계 상태)과 속성을 합친 추상적인 개념이다.

Relation 스키마

Relation의 구조를 나타내며, relation 이름과 속성의 리스트로 표기된다. 속성과 관련된 제약조건도 포함된다.

STUDENT(id, name, grade, major, phone_num, emer_phone_num)

관계가 가지는 속성의 개수 (degree of a relation)

relation의 차수(degree of a relation)는 관계 스키마(relation schema)에서 속성의 수를 의미한다.

STUDENT(id, name, grade, major, phone_num, emer_phone_num) -> degree 6

관계형 데이터베이스

관계형 데이터 모델에 기반하여 구조화된 데이터베이스이다. 이 관계형 데이터베이스는 여러 개의 relation으로 구성된다.

관계형 데이터베이스 스키마

관계형 데이터베이스 스키마(relational database schema)는 관계형 스키마(relational schema)의 집합과 무결성 제약조건(integrity constraints)의 집합을 합한 것을 말한다. 따라서 데이터베이스의 구조와 무결성을 정의하는 중요한 개념이다.

관계의 특징

  • relation은 중복된 레코드를 가질 수 없다.
    relation은 레코드의 집합이기 때문에 집합의 특성을 물려받아, 중복된 레코드를 가질 수 없다.
idnamephone_numbergrademajor
sid_01Walter010-1234-56784컴콩
sid_01Walter010-1234-56784컴콩
sid_03Zoe...2...
  • relation의 레코드를 식별하기 위해 속성의 부분집합을 키(key)로 설정한다.
    아래 예제에서는 id 속성을 사용한다.
idnamephone_numbergrademajor
sid_01Walter...4...
sid_02Olivia...3...
sid_03Zoe...2...
  • relation에서 레코드의 순서는 중요하지 않다.
    레코드의 순서가 바뀌어도 레코드의 의미는 바뀌지 않는다. 따라서 레코드의 순서를 어떻게 표시해야 되는지 따로 정의하고 있지 않다.
  • 하나의 relation에서 속성의 이름은 중복될 수 없다.
  • 하나의 레코드에서 속성의 순서는 중요하지 않다.
    순서가 바뀐다고 해서 의미가 바뀌지 않는다.
  • 속성은 더 이상 나눌 수 없는 아토믹(atomic) 값이어야 한다.
    하나의 속성이 여러 개의 서브 속성으로 구성되는 composite 속성은 허용되지 않는다. 그리고 하나의 속성이 여러 개의 값을 가질 수 있으며, 이러한 값들이 독립적으로 존재할 수 있는 multivalued 속성 또한 허용되지 않는다.
    • 주소 속성의 경기도 성남시 수내동은 각각 도, 시, 동으로 하위 속성을 둘 수 있기 때문에 composite 속성이므로 허용되지 않는다.
    • 전공 속성의 컴공, 건축은 복수 전공 이므로 분리가 가능하다.
    • composite 속성과 multivalued 속성을 어떻게 분리하는지는 다른 글을 통해 알아보도록 하곘다.
idnameaddressgrademajor
sid_01Walter경기도 성남시 수내동4컴공, 건축
sid_02Olivia...3...
sid_03Zoe...2...

NULL의 의미

NULL은 의미가 굉장히 모호하다.

  • 값이 존재하지 않는다.
  • 값이 존재하나 아직 그 값이 무엇인지 알지 못한다.
  • 해당 사항과 관련이 없다.

예제를 통해 살펴보자.

  • 토익 점수 속성에 NULL의 의미는 다음과 같을 수 있다.
    • 시험을 아직 치르지 않은 경우.
    • 제출했지만 아직 입력하지 않은 경우.
    • 제출했지만 누락된 경우.
  • 편입 속성 또한 NULL의 의미가 모호해 다음과 같을 수 있다.
    • 편입을 하지 않았기 때문에 해당 되지 않는 경우.
    • 편입을 했지만 업데이트를 하지 않은 경우.
idnamephone_numbergrademajortoeic_scoretransfer_from
...............NULLbesfir University
...............912NULL
...............990NULL

이처럼 NULL의 의미는 굉장히 모호하기 때문에 되도록 사용하지 않는 것이 좋다.

Keys

수퍼키 (super key)

relation 내의 레코드들을 고유(unique)하게 식별할 수 있는 속성의 집합이다.

  • PLAYER(id, name, team_id, back_number, birth_date)의 경우.
  • 수퍼키는 {id, name, team_id, back_number, birth_date}, {id, name}, {name, team_id, back_number} 등 이다.

후보키 (candidate key)

어느 한 속성이라도 제거하면 고유하게 레코드를 식별할 수 없는 수퍼키를 말한다. key 또는 minimal superkey로 불리기도 한다.

  • PLAYER(id, name, team_id, back_number, birth_date)의 경우.
  • 후보키는 {id}, {team_id, back_number} 이다.
    • {id} : id 값은 그 자체만으로 고유하기 때문에 레코드를 식별할 수 있으며, 해당 속성을 제거하면 구분할 수 없게 되니 후보키가 된다.
    • {team_id, back_number} : 한 팀에 선수가 여럿이기 때문에 team_id 속성만으로는 구분할 수 없다. 또한 다른 팀에 동일한 등번호의 선수가 있을 수 있기 때문에 back_number 속성으로도 구분할 수 없다. 따라서 둘 중 하나의 제거하면 레코드를 식별할 수 없기 때문에 후보키가 된다.

기본키 (primary key)

relation에서 레코드를 고유하게 식별하기 위해 선택된 후보키이다.

  • PLAYER(id, name, team_id, back_number, birth_date)의 경우.
  • 기본키는 {id} 또는 {team_id, back_number} 이다.
    • 보통은 속성의 개수가 적은 것을 사용하는 것이 더 편리하기 때문에 {id}를 기본키로 설정하는 것이 더 편하다.
    • 기본키로 선택된 속성은 밑줄로 표시한다.
  • 약칭으로 PK라고 부른다.

유니크키 (unique key)

기본키가 아닌 후보키이다. 대체키(alternate key)로 부르기도 한다.

  • PLAYER(id, name, team_id, back_number, birth_date)의 경우.
  • 유니크키는 {team_id, back_number} 이다.
    • id 속성에 밑줄이 있으므로 {id}가 기본키이기 때문에 {team_id, back_number}가 유니크키가 된다.

외래키 (foreign key)

다른 relation의 PK(primary key의 약자)를 참조하는 속성의 집합으로, 약칭으로 FK라 부른다.

idnameteam_idback_numberbirth_date
player_01Walterteam_01......
player_02Adamteam_01......
idnamemanager
team_01Team Besfirbesfir
  • PLAYER(id, name, team_id, back_number, birth_date)와 TEAM(id, name, manager)의 경우.
  • 외래키는 PLAYER의 {team_id}

제약조건 (constraints)

관계형 데이터베이스의 relation들이 항상 지켜야 하는 제약조건을 말한다.

암묵적 제약조건 (implicit constraints)

관계형 데이터 모델 자체가 가지는 제약조건이다.

  • relation은 중복되는 레코드를 가질 수 없다.
  • relation 내에서는 같은 이름의 속성을 가질 수 없다.

스키마 기반 제약조건 (schema-based constraints)

주로 DDL을 통해 스키마에 직접 명시할 수 있는 제약조건이다. 명시적 제약조건(explicit constraints)라고도 부른다.

  • 도메인 제약조건(domain constraints)
    속성의 값은 해당 속성의 도메인에 속한 값이어야 한다. 아래 예제에서 56학년은 해당 도메인의 범위를 벗어나기 때문에 제약조건을 위배한다.
idnamephone_numbergrademajor
sid_01Walter...4...
sid_02Olivia...56...
sid_03Zoe...2...
  • 키 제약조건(key constraints)
    서로 다른 레코드는 같은 값의 키(key)를 가질 수 없다. 아래 예제에서 동일한 기본키를 사용하기 때문에 제약조건을 위배한다.
idnamephone_numbergrademajor
sid_01Walter...4...
sid_01Olivia...3...
sid_03Zoe...2...
  • NULL 값 제약조건(Null value constraint)
    속성이 not null로 명시됐다면 NULL을 값으로 가질 수 없다.
idname
(not null)		phone_numbergrademajor
sid_01Walter...4...
sid_02NULL...3...
sid_03Zoe...2...
  • 엔티티 무결성 제약조건(entity integrity constraint)
    기본키는 값에 NULL을 가질 수 없다.
idname
(not null)		phone_numbergrademajor
sid_01Walter...4...
NULLOlivia...3...
sid_03Zoe...2...

💡 엔티티는 데이터베이스에서 저장하고자 하는 객체 또는 개념을 나타내는 것으로 테이블 및 relation과 동의어이다.

  • 참조 무결성 제약조건(referential integiry constraint)
    FK(foreign key의 약자)와 PK와 도메인이 같아야 하고 PK에 없는 값을 FK가 값으로 가질 수 없다.
idnameteam_idback_numberbirth_date
player_01Walterteam_02......
player_02Adamteam_02......
idnamemanager
team_01Team Besfirbesfir
