데이터 추가하기

데이터를 추가하는 SQL 문(statement)을 insert 문 또는 insert 쿼리(query)라고 하는데, 이를 사용하는 방법은 3가지 있다. 콤마로 구분된 모든 값(comma-separated all values)을 이용하는 방법, 속성 목록과 값 목록(attribute list with value list)을 이용하는 방법, 그리고 일괄 삽입(bulk insert)이 있다.

콤마로 구분된 모든 값을 이용하는 방법

콤마로 구분된 모든 값을 이용하는 방법을 이용할 때에는 다음과 같이 SQL을 작성한다.

insert into table_name values (comma, separated, all, values);

이때 입력하는 값의 순서는 테이블을 생성할 때의 속성의 순서를 따른다. 이전 글에서 생성한 employee 테이블을 예로 들어보자.

create table employee (
  id            int primary key,
  name          varchar(30) not null,
  birth_date    date,
  gender        char(1) check ( gender in ('M', 'F') ),
  position      varchar(10),
  salary        int,
  department_id int,
  foreign key (department_id) references department (id)
    on delete set null on update cascade,
  check ( salary >= 50000000 )
);

employee 테이블의 속성은 id, name, birth_date, gender, position, salary, department_id 순서로 선언되었다. 이 순서에 맞춰 데이터를 삽입해보자.

insert into employee
values (1, 'John', '1987-02-01', 'M', 'DEV_BACK', 100000000, null);
output
Query OK, 1 row affected (0.00 sec)

select * from employee;
output
+----+------+------------+--------+----------+-----------+---------------+
| id | name | birth_date | gender | position | salary    | department_id |
+----+------+------------+--------+----------+-----------+---------------+
|  1 | John | 1987-02-01 | M      | DEV_BACK | 100000000 |          NULL |
+----+------+------------+--------+----------+-----------+---------------+
1 row in set (0.00 sec)
  • 1행과 2행의 insert 문을 보자. 앞서 언급했던 속성의 순서에 맞춰서 값을 지정했고, Query OK 결과를 반환한 것으로 보아 데이터가 정상적으로 삽입되었을 것이다.
    • 특이사항이 있는데, department_id의 값이 null이라는 점이다. 그 이유는 외래키의 특성상 존재하는 값을 참조해야 하는데, 아직 department 테이블에는 데이터가 없으므로 null로 임시 지정했다.
  • 5행의 select 문은 값을 읽는 SQL로, 다음 글에서 다룰 것이기에 설명을 생략한다.

앞서 우리는 정상적으로 데이터가 삽입되는 사례를 확인해 봤다. 이번에는 제약조건을 위반하는 사례를 만들어서 에러 메시지를 확인하고 이를 통해 문제를 해결해 보도록 하겠다.

insert into employee
values (1, 'Jane', '1996-05-05', 'F', 'DSGN', 90000000, null);
output
ERROR 1062 (23000): Duplicate entry '1' for key 'employee.PRIMARY'
  • 에러가 발생했는데, 에러 메시지를 읽어보니 입력하는 기본키의 값이 이전에 입력한 값과 중복되어 unique 제약조건을 위반했기 때문이다.
  • 따라서 id값을 2로 수정하고 다시 삽입해 보겠다.
insert into employee
values (2, 'Jane', '1996-05-05', 'F', 'DSGN', 40000000, null);
output
ERROR 3819 (HY000): Check constraint 'employee_chk_2' is violated.
  • 이번에도 에러가 발생했다. 에러 메시지를 읽어보니 check 제약조건을 위반했다.
  • 우리는 gender와 salary 두 속성에 check 제약조건을 사용했기 때문에, 둘 중 어떤 제약조건을 위반했는지 알 수 없다고 가정하자.
  • 따라서 이전에 배웠던 show create table table_name을 사용해서 제약조건의 이름을 확인해보겠다.
show create table employee;

속성 목록과 값 목록을 이용하는 방법

속성 목록을 지정하는 SQL 문은 다음과 같이 사용한다.

insert into table_name (comma, separated, attribute, list) 
values (comma, separated, values, in, attribute, order);

앞의 방식은 테이블의 정의에 맞춰 값을 지정해야 했다면, 이 방식은 값을 지정하는 방식에 유연성을 제공한다. 테이블에 정의된 모든 값을 입력할 필요 없이 원하는 속성만 선택적으로 삽입할 수 있고, 순서도 바꿀 수 있다.

insert into employee (name, birth_date, gender, position, id)
values ('Smith', '2000-10-12', 'F', 'DEV_BACK', 3);
output
Query OK, 1 row affected (0.00 sec)

select * from employee;
output
+----+-------+------------+--------+----------+-----------+---------------+
| id | name  | birth_date | gender | position | salary    | department_id |
+----+-------+------------+--------+----------+-----------+---------------+
|  1 | John  | 1987-02-01 | M      | DEV_BACK | 100000000 |          NULL |
|  2 | Jane  | 1996-05-05 | F      | DSGN     |  90000000 |          NULL |
|  3 | Smith | 2000-10-12 | F      | DEV_BACK |  50000000 |          NULL |
+----+-------+------------+--------+----------+-----------+---------------+
3 rows in set (0.00 sec)
  • 테이블에 정의된 속성 중 일부만 입력했으며, 테이블에 정의된 순서를 따르지 않고 있다.
  • 선택한 속성의 순서에 맞춰 값을 입력하고 있다.
  • 우리는 salary 값을 입력하지 않았지만 50,000,000이 입력되어있다.
    • 테이블을 정의할 때 default 50000000으로 기본값을 지정했기 때문이다.

일괄 삽입

일괄 삽입 SQL문은 다음과 같이 사용한다.

insert into table_name (comma, separated, attribute, list)
    values
        (comma_1, separated_1, values_1, in_1, attribute_1, order_1),
        (comma_2, separated_2, values_2, in_2, attribute_2, order_2),
        ...;

일괄 삽입에서는 각 레코드는 쉼표로 구분되고, 괄호로 묶인 값의 목록이 행 단위로 나열되는 형식이다. 이 형식으로 여러 개의 레코드를 한 번의 INSERT 쿼리로 삽입할 수 있다. 이를 이용해서 임직원 데이터를 추가로 삽입해 보겠다.

데이터 수정하기

데이터를 수정할 때에는 다음과 같이 SQL을 작성한다.

update table_name(s) 
set attribute = value [, attribute = value, ...]
[ where condition(s) ];
  • 테이블 이름은 하나 이상을 지정할 수 있다.
  • 수정하고자 하는 속성과 값은 하나의 쌍(pair)으로 동작하며, 이 또한 하나 이상 지정할 수 있다.
    • 여러개의 테이블을 지정했을 경우, 명시적으로 테이블_이름.속성 형식으로 속성을 지정하지 않으면 ambiguous 에러를 겪을 수 있다.
  • where 절(clause)은 목표로 하는 레코드의 조건을 지정하는 것으로 생략이 가능하며 여러개를 지정할 수도 있다.
    • 생략시 모든 레코드가 업데이트 되기 떄문에 주의가 필요하다.

모든 테이블에 값을 저장했으므로, employee 테이블의 department_id 속성에 값을 지정할 수 있게 되었다. update 쿼리를 통해서 id 값이 1인 첫번째 레코드의 department_id 속성의 값을 수정해 보겠다.

update employee
set department_id = 1003
where id = 1;
output
Query OK, 1 row affected (0.01 sec)
Rows matched: 1  Changed: 1  Warnings: 0

이제 가상의 시나리오를 만들고 실습해보겠다. besfir company의 대표가 개발 부서 한정으로 모든 직원의 연봉을 현재의 2배로 올려주려 한다. 이를 update 쿼리를 통해 반영해보자.

update employee
set salary = salary * 2
where department_id = 1003;
output
Query OK, 5 rows affected (0.01 sec)
Rows matched: 5  Changed: 5  Warnings: 0

결과 메시지를 통해서 5개의 레코드가 수정됐음을 알 수 있다. 이번에는 프로젝트 ID가 2003인 프로젝트에 참여한 모든 직원의 연봉을 2배로 올려보겠다.

update employee, works_on
set salary = salary * 2
where id = employee_id
  and project_id = 2003;

employee와 works_on 테이블에는 겹치는 속성명이 없기 때문에 위처럼 표현해도 괜찮다. 그러나 여러 테이블을 사용하면 겹치는 속성명이 있을 경우, ambiguous 에러가 발생할 수 있고 어떤 테이블의 속성인지 테이블 명세를 봐야 확인할 수 있는 경우가 많다. 따라서 여러 테이블을 사용시에는 아래와 같이 테이블 이름과 그 속성을 명시해주는 것이 더 좋은 표현 방법이다.

update employee, works_on
set employee.salary = employee.salary * 2
where employee.id = works_on.employee_id
  and works_on.project_id = 2003;
  • 1행 : 연봉을 수정하기 위해서는 employee 테이블을 선택해야 하는데, 이것만으로는 각 직원이 참여하는 있는 프로젝트 번호를 알 수 없다.
    • 따라서 연봉 정보를 갖고 있는 employee 테이블과 직원 및 프로젝트의 번호를 갖고 있는 works_on 테이블을 묵시적 조인(Implicit Join) 한다.
    • 이때 결과 테이블은 카르테시안 곱으로 현실과는 상당히 동떨어진 결과값을 갖게 된다. 궁금하다면 select * from employee, works_on;을 통해 확인해 보도록 한다.
  • 3행 : 카르테시안 곱으로 표현된 테이블에서 현실과 맞는 정보를 가져오기 위해 employee.id = works_on.employee_id인 정보만 선택한다.
  • 4행 : project_id가 2003인 레코드만 선택한다.

이번에는 모든 임직원의 연봉을 2배로 올려보겠다.

update employee
set salary = salary * 2;
  • where 절로 조건을 제한하지 않고 있기 때문에 테이블의 모든 레코드에 반영이 된다.

데이터 삭제하기

데이터를 삭제할 때에는 다음과 같이 SQL을 작성한다.

delete from table_name
[ where condition(s) ]

delete 쿼리 또한 where 절을 지정하지 않으면 모든 레코드가 삭제될 수 있으니, 사용에 주의해야 한다.

가상의 시나리오를 만들어서 이를 실습해보자. Nicole이 퇴사를 하게 되면서 모든 테이블에서 정보를 삭제해야 한다. Nicole의 employee ID는 8이며, project 2002에 참여했었다.

delete from employee where id = 10;
output
Query OK, 1 row affected (0.01 sec)

select * from employee where id = 10;
output
Empty set (0.00 sec)
  • 정보를 삭제해야 하는 테이블은 employee와 works_on 이다.
  • employee 테이블에서 먼저 Nicole의 정보를 삭제했다.
  • 그런데, works_on 테이블에서도 삭제가 필요할까?
select * from works_on where employee_id = 10;
  • 조회를 해보니 이미 works_on 테이블에 Nicole의 정보가 존재하지 않는다.
    • 그 이유는 외래키의 reference option 때문이다.
create table works_on (
  employee_id int,
  project_id  int,
  primary key (employee_id, project_id),
  foreign key (employee_id) references employee (id)
    on delete cascade on update cascade,
  foreign key (project_id) references project (id)
    on delete cascade on update cascade
);

우리가 테이블을 생성할 때, 참조하는 테이블에서 자료가 삭제되면 works_on 테이블에서도 삭제되도록 cascade 옵션을 지정했기 때문이다.

Jane이 휴직을 떠나게 되면서 현재 진행 중인 프로젝트에서 중도하차 하게 되었다. Jane의 ID는 2이며, 프로젝트 2001, 2002, 2003에 참여 중이었다.

delete from works_on where employee_id = 2;
  • project_id를 일일히 지정할 필요 없이, employee_id를 조건으로 삭제하면 한 번에 처리할 수 있다.

Messi는 현재 3개의 프로젝트에 참여하고 있는데, 프로젝트 2001에 집중하기 위해서 프로젝트 2002, 2003에서 빠지기로 했다. Messi의 ID는 5이다.

delete from works_on where employee_id = 5 and project_id = 2002
delete from works_on where employee_id = 5 and project_id = 2003
  • 삭제할 project_id마다 delete 쿼리를 실행하는 방법이 있다.
    • 그러나 이 방법은 지워야하는 레코드의 개수가 많아지면 실수할 여지가 생긴다.
  • not equal 연산자를 사용한다.
delete from works_on where employee_id = 5 and project_id <> 2001
-- or
delete from works_on where employee_id = 5 and project_id != 2001
  • not equal를 통해서 프로젝트 ID가 2001이 아닌 레코드만 한번에 삭제할 수 있다.

회사에 큰 문제가 생겨서 모든 프로젝트를 종료한다.

delete from works_on
  • where 절을 사용하지 않으면 태이블의 전체 데이터를 삭제할 수 있다.
