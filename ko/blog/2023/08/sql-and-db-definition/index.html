SQL

Structured Query Language(SQL)은 현업에서 사용되는 관계형 데이터베이스 관리 시스템(RDBMS)의 표준 언어로 DDL, DML 그리고 VDL을 합친 종합적인 DBMS 언어라고도 할 수 있다.

SQL 주요 용어

주요 용어는 아래와 같지만, 혼용해서 사용하는 경우가 더 많으므로 염두해 두도록하자.

관계형 데이터 모델 SQL
relation 테이블(table)
속성(attribute) 열(column)
튜플(tuple) 행(row) 또는 레코드(record)
도메인(domain) 도메인(domain)

SQL에서 relation이란?

관계형 데이터 모델에서의 relation은 튜플의 중복을 허용하지 않는다. 하지만 SQL에서 relation은 multiset(=bag) of tuples, 즉 튜플의 다중 집합이라는 의미로 중복된 튜플을 허용한다. 따라서 SQL에서 테이블은 중복된 레코드를 가질 수 있다고 이해할 수 있다.

SQL & RDBMS

SQL은 RDBMS의 표준 언어이지만 구현의 강제는 없기 때문에 RDBMS마다 제공하는 스펙이 조금씩 다르다. 따라서 구분을 위해 표준 SQL은 ANSI SQL 또는 ISO SQL이라고 하며, 각 RDBMS 별로 구현한 SQL은 밴더 SQL이라고 한다. 벤더(Vendor)는 제품이나 서비스를 공급하는 공급업체를 의미하며, 여기서는 특정 RDBMS 제조사를 말한다.

MySQL로 DB 정의하기

IT 회사 관련 RDB 만들기를 통해서 DB를 정의하는 방법을 알아보도록 하겠다. 이 글에서 우리는 부서, 사원, 프로젝트 관련 정보를 정의할 수 있는 관계형 데이터베이스를 만들어 볼 것이다. 예제에는 MySQL의 엔진 중 가장 대중적인 InnoDB 엔진을 사용하도록 하겠다.

Database 정의하기

먼저, 현재 어떤 데이터베이스가 존재하는지 보여주는 show databases 명령어를 통해서 현재 존재하는 데이터베이스를 확인해보자. 아직 아무런 데이터베이스도 생성하지 않았기 때문에 기본 생성된 데이터베이스를 확인할 수 있을 것이다.

show databases;
output
+--------------------+
| Database           |
+--------------------+
| information_schema |
| mysql              |
| performance_schema |
| sys                |
+--------------------+
4 rows in set (0.01 sec)

이번에는 create database <name> 명령어를 통해 우리의 목표인 IT 회사 데이터베이스를 생성해보도록 하겠다. 필자는 데이터베이스의 이름을 besfir_company라 하겠다.

create database besfir_company;
output
Query OK, 1 row affected (0.01 sec)
show databases;
output
+--------------------+
| Database           |
+--------------------+
| besfir_company     |
| information_schema |
| mysql              |
| performance_schema |
| sys                |
+--------------------+
5 rows in set (0.00 sec)

다음으로 현재 선택된 데이터베이스를 확인하는 명령어 select database()를 입력해보곘다.

select database();
output
+------------+
| database() |
+------------+
| NULL       |
+------------+
1 row in set (0.00 sec)

값이 NULL인데, 이는 아직 사용하기로 지정한 데이터베이스가 없다는 의미이다. 이제 앞서 생성한 데이터베이스를 사용할 데이터베이스로 지정해보자.

use besfir_company;
output
Database changed
select database();
output
+----------------+
| database()     |
+----------------+
| besfir_company |
+----------------+
1 row in set (0.00 sec)

사용할 데이터베이스로 besfir_company가 지정된 것을 확인했다. 이제 테이블 정의할 차례인데, 그에 앞서 데이터베이스를 삭제하는 방법도 알아보자. 삭제를 테스트 할 용도의 데이터베이스를 별도로 생성한 뒤 삭제해 보겠다.

create database delete_test;
show databases;
output
+--------------------+
| Database           |
+--------------------+
| besfir_company     |
| delete_test        |
| information_schema |
| mysql              |
| performance_schema |
| sys                |
+--------------------+
6 rows in set (0.00 sec)
drop database delete_test;
mysql> show databases;
+--------------------+
| Database           |
+--------------------+
| besfir_company     |
| information_schema |
| mysql              |
| performance_schema |

Database vs Schema

MySQL에서는 데이터베이스와 스키마가 같은 의미를 갖는다. 따라서 create schema besfir_company와 같이 schema 키워드가 database 키워드를 대체해서 사용될 수 있다. 반면에 다른 RDBMS에서는 둘의 의미가 다르게 사용된다. 가령 PostgreSQL의 경우, 스키마는 데이터베이스의 namespace 용도로 사용된다.

-- in MySQL
create database besfir_company = create schema besfir_company

이로 인해서 테이블이 정의되는 위치에도 차이를 보인다. MySQL의 경우 데이터베이스 안에 테이블의 정의되는데 반해 PostgreSQL의 경우 데이터베이스 내의 스키마 안에 테이블이 정의된다.

테이블 정의하기

우리는 이전 relation 스키마 글의 기본키 항목에서 속성에 밑줄이 있으면 기본키라는 것을 익혔다. 이점을 인지하고 relation 스키마를 확인해보자.

  • DEPARTMENT(id, name, leader_id)
    • id : 고유 식별을 위한 기본키.
    • name : 부서명.
    • leader_id : 부서장의 고유 식별 값. EMPLOYEE의 id 값을 참조한다.
  • EMPLOYEE(id, name, birth_date, gender, position, salary, department_id)
    • id : 고유 식별을 위한 기본키.
    • name : 직원 이름.
    • birth_date : 직원 생일.
    • gender : 직원 성별.
    • salary : 직원 연봉.
    • department_id : 부서 고유 식별 값. DEPARTMENT의 id 값을 참조한다.
  • PROJECT(id, name, leader_id, start_date, end_date)
    • id : 고유 식별을 위한 기본키.
    • name : 프로젝트 명.
    • leader_id : 프로젝트 관리자 고유 식별 값. EMPLOYEE의 id 값을 참조한다.
    • start_date : 프로젝트 시작 일자.
    • end_date : 프로젝트 종료 일자.
  • WORKS_ON(employee_id, project_id)
    • employee_id : 프로젝트에 참여한 직원 고유 식별 값. EMPLOYEE의 id 값을 참조한다.
    • project_id : 프로젝트의 고유 id 식별 값. PROJECT의 id 값을 참조한다.

테이블 생성하기

이번에는 앞서 설계한 테이블을 SQL을 통해서 생성해 보겠다.

create table department (
    id        int primary key,
    name      varchar(20) not null unique,
    leader_id int
);
  • id 열은 정수(int) 데이터 타입으로 기본키 역할을 하며, 각 행을 고유하게 식별한다.
  • name 열은 최대 20자의 문자열(varchar) 데이터 타입으로 NULL을 허용하지 않으며 고유한(unique) 값을 갖는다.
  • leader_id 열은 정수(int) 데이터 타입이다.
  • 참고하기
    • 데이터 타입
    • 키 제약조건 : 기본키
    • 키 제약조건 : 유니크
    • Not null 제약조건
create table employee (
  id            int primary key,
  name          varchar(30) not null,
  birth_date    date,
  gender        char(1) check ( gender in ('M', 'F') ),
  position      varchar(10),
  salary        int,
  department_id int,
  foreign key (department_id) references department (id)
    on delete set null on update cascade,
  check ( salary >= 50000000 )
);
  • gender 열은 문자를 값으로 받으며, 속성 수준 유효성 감사를 통해서 'M' 또는 'F'만을 값으로 받는다.
  • foreign key ~로 시작하는 부분은 다른 테이블을 참조하는 외래키에 대한 부분으로, department_id 열은 department 테이블의 id 열을 참조한다.
    • department 테이블에서 특정 레코드 삭제시 이를 참조하는 employee 테이블의 department_id 열의 값은 NULL로 수정된다.
    • department 테이블의 id 값이 변경되면 이를 참조하는 employee 테이블의 department_id 값도 수정된다.
  • salary 열은 테이블 수준에서 유효성 검사를 해서 50000000 이상인 값을 허용한다.
  • 참고하기
    • Check 제약조건
    • 참조 무결성 제약조건
create table project (
  id         int primary key,
  name       varchar(20) not null unique,
  leader_id  int,
  start_date date,
  end_date   date,
  foreign key (leader_id) references employee (id)
    on delete set null on update cascade,
  check ( start_date < project.end_date )
);
create table works_on (
  employee_id int,
  project_id  int,
  primary key (employee_id, project_id),
  foreign key (employee_id) references employee (id)
    on delete cascade on update cascade,
  foreign key (project_id) references project (id)
    on delete cascade on update cascade
);
  • project 테이블과 works_on 테이블은 앞에서 살펴본 내용이 많기 때문에 설명을 생략한다.

테이블 수정하기

필요한 테이블을 모두 정의했지만 하나 빠뜨린 것이 있다. 바로 department 테이블의 leader_id에 외래키를 선언하지 않은 것이다. department 테이블을 생성하는 시점에는 employee 테이블이 존재하지 않았기 때문에 외래키를 선언할 수 없었기 때문이다.

이번에는 테이블을 수정하는 키워드 alter table을 사용해서 외래키를 적용해보자.

alter table department add foreign key (leader_id)
  references employee(id)
  on update cascade 
  on delete set null;

이제 테이블을 정의하는 작업이 모두 완료가 되었다. 다음 글에서 이 테이블들을 통해서 데이터를 다루는 방법을 익히도록 하곘다.

    • MySQL의 경우 SQL 표준을 따라 모든 정수형 타입을 구현하고 있다.

    • MySQL의 경우 DECIMAL과 NUMERIC 타입의 차이가 없이 엄격하게 동작하기 때문에 자리수가 일치하지 않으면 저장하지 않는다.

    • 보통 암호화 키를 저장할 때 사용한다.

키 제약조건

키 제약조건 : 기본키

기본키는 테이블의 레코드를 식별하기 위해 사용되며, 하나 이상의 속성으로 구성되어 있다. 기본키는 고유(unique)해야하며 NULL도 값으로 가질 수 없다.

team_idback_numberbirth_datenamephone_number
team_1327.........
team_1327.........
team_371NULL.........

기본키를 선언하는 방법은 아래와 같다.

하나의 속성으로 구성될 때하나 이상의 속성으로 구성될 때
create table player (
 id int primary key,
 ...
); 		create table player (
 team_id varchar(12),

 ...
 primary key(team_id, back_number)

    • 하나의 속성을 기본키로 지정할 때는 두 방법 중 어느것을 써도 무방하다.
    • 둘 이상의 속성을 기본키로 지정할 때는 primary key(속성1, 추가 속성...) 형태로 지정해야 한다.

키 제약조건 : 유니크(unique)

Unique 제약조건이 적용된 속성은 중복된 값을 가질 수 없다. 단, NULL은 중복을 허용할 수도 있는데, 이는 RDBMS마다 다를 수 있다. MySQL이나 PostgreSQL의 경우 NULL의 중복을 허용하지만, MsSQL(SQL Server)의 경우 NULL의 중복을 허용하지 않아 열당 하나의 NULL만 존재할 수 있다.

team_idback_number
(UNIQUE)		birth_datenamephone_number
team_1327.........
team_2847.........
team_37112.........

Unique를 선언하는 방법은 아래와 같다.

하나의 속성으로 구성될 때하나 이상의 속성으로 구성될 때
create table player (
 id int unique,
 ...
); 		create table player (
 team_id varchar(12),
 back_number int,
 ...
 unique(team_id, back_number)
);

check 제약조건

참조 무결성 제약조건 : 외래키

속성이 다른 테이블의 기본키나 유니크 키를 참조할 때 사용한다.

EMPLOYEE

id...department_id
......256
......1024
......14

DEPARTMENT

id......
128......
256......
1024......
  • 부가 설명.
    • EMPLOYEE 테이블의 department_id 열이 DEPARTMENT 테이블의 id 열을 참조하고 있다.
    • 이때, EMPLOYEE 테이블의 department_id 열은 반드시 DEPARTMENT 테이블의 id 열에 존재하는 값만을 참조할 수 있다.
    • 따라서 department_id 열에 14는 존재하지 않는 DEPARTMENT id 값을 입력했으므로 참조 무결성 제약조건을 위배하게 된다.

외래키를 선언하는 방법은 아래와 같다.

SQLreference_option설명
create table employee (
 ...
 department_id int,
 foreign key (department_id)
  references department(id)
  on delete reference_option
  on update reference_option
);		CASCADE참조값의 삭제/변경을 그대로 반영
SET NULL참조값이 삭제/변경 시 NULL로 반영
RESTRICT참조값이 삭제/변경되는 것을 금지
NO ACTIONRESTRICT와 유사
SET DEFAULT참조값이 삭제/변경 시 default 값으로 변경
  • reference_option 부가 설명
    • CASCADE : DEPARTMENT에서 레코드가 삭제되면, 이를 참조하는 EMPLOYEE 레코드가 같이 삭제된다.
      DEPARTMENT 레코드의 id 값이 1로 변경되면, 이를 참조하는 EMPLOYEE 레코드의 department_id 값이 1로 변경된다.
    • SET NULL : DEPARTMENT 레코드가 삭제되거나 id 값이 변경되면 이를 참조하는 EMPLOYEE 레코드의 department_id 값이 NULL로 변경된다.
    • RESTRICT : 다른 테이블에서 해당 레코드를 참조하고 있다면, DEPARTMENT 레코드를 삭제하거나 특정 열을 수정하는 작업이 금지된다.
    • NO ACTION : MySQL의 경우 RESTRICT와 동일하다. 표준 SQL의 경우, 트랜젝션 안에서는 삭제/변경을 허용하지만 트랜젝션이 끝난 후에는 금지된다.
    • SET DEFAULT : 값이 삭제/변경되는 경우 default 값으로 변경된다. MySQL에서는 지원되지 않는다.
    • MySQL에서는 CASECADE, SET NULL, RESTRICT 세 가지만 지원되며, PostgreSQL에서는 다섯 가지 모두 지원한다.

제약조건 이름 명시하기

이름을 붙이면 어떤 제약조건을 위반했는지 쉽게 파악할 수 있다. 또한 제약조건을 삭제하고 싶을 때 해당 이름으로 삭제도 가능하다. MySQL의 경우, 제약조건을 명시했을때와 그렇지 않았을 때 유효성 검사 메시지는 아래와 같다.

create table test (
 age int constraint age_over_19 check(age > 19)
);
constraint age_over_19 이름을 붙였을 때constraint age_over_19 이름을 생략했을 때
Check constraint 'age_over_19' is violated.Check constraint 'test_chk_1' is violated.
  • 부가 설명
    • 제약조건 이름을 명시하지 않으면, 어떤 제약조건을 위반했는지 파악하기 어려워진다. 다시말해, 직관적이지 않은 설계가 된다.
    • MySQL의 경우 'test_chk_1' 제약조건이 무엇인지 알고 싶다면, show create table test;라고 입력하면 확인이 가능하다.
