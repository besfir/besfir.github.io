requestAnimationFrame()에 대해 알아보자

시작하기에 앞서, 몇몇 용어를 반복 사용해야 하는데 길어서 축약 정의하도록 하겠다.

  • setTimeout()와 setInterval() => timer 함수
  • requestAnimationFrame() => rAF()

CSS만으로 표현하기 어려운 애니메이션을 구현하기 위해서는 자바스크립트를 사용해야 하는데, 최적화 없이 이를 사용하면 성능상 문제가 발생할 수 있다. rAF()가 나오기 전에는 최적화 기법으로 timer 함수가 사용됐고, 그 이후로는 rAF()을 사용되고 있다.

timer 함수 vs rAF()

이 섹션에서는 timer 함수에 어떤 문제가 있어서 rAF()가 나왔는지 비교해 보며 알아보도록 하겠다.

프레임 유실

timer 함수는 프레임을 고려하지 않고 작동하기 때문에 언제 콜백 함수가 호출되는지 보장할 수 없다. 따라서 아래 그림처럼 프레임에 걸쳐서 실행되면 해당 프레임을 놓치기 때문에 애니메이션이 버벅거리게 된다.

💡 그림의 16ms는 어떻게 나온 값일까?
특별히 화면 주사율을 선택해서 구매하지 않았다면 60Hz의 모니터를 사용중 일 텐데, 이는 1초에 60회 화면을 갱신한다는 의미이다. 따라서 1,000ms / 60회 ≈ 16ms 간격으로 화면을 갱신하게 된다.

반면, rAF()는 화면 주사율에 동기화 되며, 브라우저가 화면을 그릴 준비가 되었을 때 실행된다. 조금 더 자세히 말하자면, rAF()에 작성한 콜백함수는 Animation Frame queue에 등록되며, 다음 repaint 직전에 이 큐에 있는 모든 콜백함수가 한번에 처리된다. 따라서 위 그림처럼 콜백 함수가 어중간하게 시작되어서 프레임이 유실되는 문제를 방지할 수 있다.

timer 함수와 rAF()를 이용해서 각 박스가 오른쪽으로 5px씩, 총 1,000px 이동하는 애니메이션을 예제 코드로 만들었다. 앞서 언급했듯이 언제 콜백 함수가 실행될지 보장되지 않기 때문에 눈으로 확인될 정도로 프레임 유실이 일어날 수도 있고 그렇지 않을 수도 있다. 이 경우 몇 차례 반복하다 보면 확인할 수 있다.

시스템 리소스 낭비

timer 함수는 창이 비활성화 상태이거나 숨겨진 iframe 내에서도 동작해서 시스템 리소스가 낭비되었고, 이에 따라 배터리 수명과 성능 저하 문제를 야기했다. 그러나 rAF()를 이용하면 창이 비활성화 상태가 됐을 때 작업을 중지하기 때문에 시스템 자원을 절약할 수 있다.

이 내용도 직접 확인해 보기 위해 앞서 언급한 예제 코드의 애니메이션을 실행하자마자 브라우저를 다른 탭으로 전환해 보자. 넉넉히 5초 정도 기다렸다가 돌아가 보면 timer 함수로 실행한 애니메이션은 이미 끝나있지만, rAF()로 실행한 함수는 아까 그 자리에서부터 다시 시작한다.

rAF() 성능 최적화

window.addEventListener('scroll', function(e) {
  if (!ticking) {
    window.requestAnimationFrame(function() {
      doSomething();
      ticking = false;
    });
    ticking = true;
  }
});

어디서 봤는지 모르겠는데, 스크롤 이벤트는 너무 많이 발생하기 때문에 위의 코드로 스로틀 처리를 해야된다는 글을 봤다. 그 이후로 스크롤 이벤트를 사용할 때 무조건적으로 스로틀을 적용했는데 우연히 MDN 문서를 보니, 그동안 잘못 알고 있었다는 것을 깨닳았다.

스크롤 이벤트는 스크롤 된 화면을 랜더링 했을 때 발생하므로 화면 주사율과 동기화 된다. 따라서 화면 주사율과 동기화 되어있는 rAF()와 같은 속도로 실행되기 때문에 rAF()를 이용한 스로틀은 종종 불필요하다. 그렇다면 어떤 경우에 rAF()가 필요할까? 바로, 아래처럼 레이아웃 스레싱을 일으킬 수 있는 코드를 사용할 때 필요하다.

function onScroll(){
  const box = element.getBoundingClientRect();
  if(box.top > pos){
    element.style.height = box.height * 2
  }
}
window.addEventListener('scroll', () => {
  onScroll()
});

Reflow, 강제 동기식 레이아웃 그리고 레이아웃 스레싱

위 그림은 랜더링 할때 작업 흐름을 나타내고 있는데, 각 단계를 모두 설명하기에는 피로도가 높으므로 오늘 주제에 필요한 reflow만 짚고 넘어가겠다.

reflow는 DOM 요소의 기하학적 값이 변경됐을 때, layout 단계부터 재수행하는 과정을 말한다. layout 단계부터 재수행하는 것을 통해 알 수 있듯이 변경한 요소뿐만 아니라, 다른 요소들의 계산도 재수행해야 하기 때문에 비용이 크게 들어가는 작업이다. 따라서 브라우저는 이 비용을 줄이기 위해서 reflow가 필요한 작업들을 큐에 모아서, 다음 프레임에 한번의 reflow로 처리할 수 있도록 예약을 한다.

그러나 기하학적 값을 읽고 쓰는 것을 한 프레임에서 처리하려고 할 때에는 즉시 reflow를 수행하게 된다. 이를 강제 동기식 레이아웃이라고 하며, 강제 동기식 레이아웃이 반복적으로 일어나 성능 저하를 초래하는 현상을 레이아웃 스레싱이라고 한다.

레이아웃 스레싱 방지

위에서 언급했던 코드는 top이라는 기하학적인 값을 한 프레임에서 읽고 썼기 때문에 강제 동기식 레이아웃이 발생한다. 따라서 우리는 앞서 설명한 rAF()의 특징을 이용해서, reflow를 일으키는 읽기 로직을 분리하고 나머지 연산 로직은 rAF()에 콜백함수로 넘기도록 할 것이다. 이렇게하면 읽기는 현재 프레임에서 처리하고, 나머지 연산은 다음 프레임에 모아서 한 번에 처리되기 때문에 레이아웃 스레싱을 효과적으로 방어할 수 있다.

function onScroll(height){
  element.style.height = height * 2
}
window.addEventListener('scroll', () => {
  const box = element.getBoundingClientRect();
  if(box.top > pos){
    requestAnimationFrame(()=>{
      onScroll(box.height)
    })
  }
});

감사하게도 이 내용으로 누군가 이미 테스트 가능한 예제를 만들어 놨기에 개발자툴로 테스트 해봤다. 이게 맞는건가 싶을 정도로 한번에 처리됐다.
