테스트 코드의 중요성은 이전부터 인지하고 있었지만, 그걸 익힐 시간에 코드를 짜면 더 빠르게 마무리 할 수 있으니 나중에 공부해야겠다며 계속 미뤄왔다. 하지만, 요구사항이 조금씩 추가될 때마다 주먹구구식으로 테스트 하고 있는 스스로에게 문득 부끄러움을 느꼈다. 따라서 개인 공부 목적으로 테스트 코드에 익숙해지기 시리즈를 써보려 한다.

이번 글에서는 반드시 알아야만, 테스트를 올바르게 진행할 수 있는 정보들을 정리해 볼 예정이다.

단위 테스트

단위 테스트는 컴퓨터 프로그래밍에서 소스 코드의 특정 모듈이 의도된 대로 정확히 작동하는지 검증하는 절차다. 즉, 모든 함수와 메소드에 대한 테스트 케이스를 작성하는 절차를 말한다.

위키피디아 발췌

단위 테스트의 장점

  • 문제점 파악: 기능을 작게 쪼개서 테스트하기 때문에 어떤 부분에서 문제가 생겼는지 빠르게 파악할 수 있다.
  • 리팩토링: 코드를 작고 단순하게 만들기 때문에 리펙토링 시에 더 나은 정확성과 안정성을 보장한다.
  • 문서화: 단위 테스트를 이용해서 API 문서화를 해주는 라이브러리들이 있기 때문에, 문서의 작성 및 유지보수가 편리해진다.

단위 테스트의 한계점 및 단점

  • 보장: 메소드를 고립시켜서 시뮬레이션하기 때문에, 단위 테스트의 결과가 항상 시스템 전체의 결과를 대변하지는 못한다.
  • 개발 및 유지보수 비용: 테스트 코드도 개발 및 유지보수 해야하기 때문에 관리해야할 범위가 늘어난다.

JUnit 5

JUnit 5는 이전 JUnit과는 다르게 JUnit Platform + JUnit Jupiter + JUnit Vintage 구성돼있다.

  • JUnit Platform: JVM에서 테스트 프레임워크를 실행하는 역할을 한다. 또한 테스트 엔진 API를 제공하여, JUnit과 쉽게 통합될 수 있도록 지원하기도 한다.
  • JUnit Jupiter: JUnit 5와 같은 Jupiter 하위 프로젝트에 대한 테스트 엔진을 제공한다.
  • JUnit Vintage: JUnit 3, JUnit 4 기반 테스트를 실행하기 위한 테스트 엔진을 제공한다.

기본 어노테이션

  • @Test: 해당 메소드가 테스트 메소드임을 나타낸다.
  • @BeforeAll: 클래스 내에서 가장 먼저 실행돼야 하는 메소드임을 나타낸다.
  • @AfterAll: 클래스 내에서 가장 나중에 실행돼야 하는 메소드임을 나타낸다.
  • @BeforeEach: 클래스 내의 각 테스트 메소드보다 먼저 실행되어야 하는 메소드임을 나타낸다.
  • @AfterEach: 클래스 내의 각 테스트 메소드보다 나중에 실행되어야 하는 메소드임을 나타낸다.

생명주기 메소드

@BeforeAll, @AfterAll, @BeforeEach, @AfterEach는 테스트의 생명주기에 따라 실행되기 때문에 생명주기 메소드라고 부른다.

import org.junit.jupiter.api.AfterAll;
import org.junit.jupiter.api.AfterEach;
import org.junit.jupiter.api.BeforeAll;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Disabled;
import org.junit.jupiter.api.Test;
public class OrderTest {
    @BeforeAll
    static void beforeAll() {
        System.out.println("=-=-=-=-=-=");
        System.out.println("Before all");
    }
    @AfterAll
    static void afterAll() {
        System.out.println("After all");
        System.out.println("=-=-=-=-=-=");
    }
    @BeforeEach
    void setUp() {
        System.out.println("-----");
        System.out.println("Before each");
    }
    @AfterEach
    void tearDown() {
        System.out.println("After each");
        System.out.println("-----");
    }
    @Test
    void runFirst() {
        System.out.println("run first");
    }
    @Test
    void runLast() {
        System.out.println("run last");
    }
    @Test
    @Disabled
    void disableTest() {
        System.out.println("it does not run");
    }
}
Output
=-=-=-=-=-=
Before all
-----
Before each
run last
After each
-----
-----
Before each
run first
After each
-----
After all
=-=-=-=-=-=

@DisplayName

IDE의 JUnit 패널이나 Gradle 테스트 보고서를 보면 우리가 테스트한 결과를 확인할 수 있다. 이때, 각 테스트는 메소드 이름으로 식별이 가능하다.

class DisplayNameTest {
    @Test
    void displayNameContainingSpacesAndYourLanguageExample() { }
    @Test
    void displayNameContainingAngryCharactersExample() { }
    @Test
    void displayNameContainingEmojiExample() { }
}

하지만, 간혹 메소드 이름은 그대로두고 테스트 이름만 바꾸고 싶을 수 있는데, 이때 사용하는 것이 @DisplayName이다.

@DisplayName("Custom test class name")
class DisplayNameTest {
    @Test
    @DisplayName("한글도 잘 표시된다")
    void displayNameContainingSpacesAndYourLanguageExample() { }
    @Test
    @DisplayName("୧(๑•̀ᗝ•́)૭")
    void displayNameContainingAngryCharactersExample() { }
    @Test
    @DisplayName("🌈")
    void displayNameContainingEmojiExample() { }
}

생명주기 메소드 단원의 로그를 다시 한번 보자. runFirst()를 먼저 작성했고 알파벳 순서로 생각해봐도 먼저 실행될 것 같은데, 나중에 실행됐다.

그 이유는, 일반적으로 단위 테스트는 순서에 의존해서는 안되기 때문에 JUnit이 예측할 수 없는 순서로 테스트를 실행하기 때문이다. 하지만 간혹 테스트 순서가 중요한 경우가 있을 수 있다. 이런 경우, 클래스에 @TestMethodOrder를 선언하고 원하는 MethodOrderer 구현체를 지정하면 된다.

OrderAnnotation

@Order에 지정한 값을 기준으로 메서드를 오름차순으로 정렬합니다. 기본 순서 값은 Integer.MAX_VALUE / 2이며, 이 값보다 큰 메소드는 어노테이션이 없는 메소드 보다 나중에 실행된다.

import org.junit.jupiter.api.MethodOrderer.OrderAnnotation;
import org.junit.jupiter.api.Order;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.TestMethodOrder;
@TestMethodOrder(OrderAnnotation.class)
public class TestMethodOrderTest {
    @Test
    @Order(1)
    void runFirst() {
        System.out.println("run first");
    }
    @Test
    @Order(2)
    void runSecond() {
        System.out.println("run second");
    }
    
    @Test
    void runThird() {
        System.out.println("run third");
    }
    @Test
    @Order(Integer.MAX_VALUE / 2 + 1)
    void run4th() {
        System.out.println("run 4th");
    }
    @Test
    @Order(Integer.MAX_VALUE)
    void runLast() {
        System.out.println("run last");
    }
}
Output
run first
run second
run third
run 4th
run last

메소드 이름을 기준으로 Alphanumeric 순서로 실행한다.

import org.junit.jupiter.api.MethodOrderer.MethodName;

DisplayName

MethodName과 비슷하지만, display name을 기준으로 alphanumeric 순서로 실행한다.

import org.junit.jupiter.api.DisplayName;
import org.junit.jupiter.api.MethodOrderer;
import org.junit.jupiter.api.MethodOrderer.Random;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.TestMethodOrder;
@TestMethodOrder(MethodOrderer.DisplayName.class)
public class TestMethodOrderTest {
    @Test
    @DisplayName("run last")
    void runLast() {
        System.out.println("run last");
    }
    @Test
    @DisplayName("run first")
    void runFirst() {
        System.out.println("run first");
    }
}

run first
run last

Random

System.nanoTime()을 시드로 사용해서 무작위로 정렬된다. 항상 동일한 실행 순서를 보장하지 않는다.

import org.junit.jupiter.api.MethodOrderer.Random;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.TestMethodOrder;
@TestMethodOrder(Random.class)
public class TestMethodOrderTest {
    @Test
    void runFirst() {
        System.out.println("run first");
    }
    @Test
    void runSecond() {
        System.out.println("run second");
    }
    @Test
    void runThird() {
        System.out.println("run third");
    }
    @Test
    void run4th() {
        System.out.println("run 4th");
    }
    @Test
    void runLast() {
        System.out.println("run last");
    }
}

run 4th
run third

run second

Assertions

테스트에서 결과를 검증하는 작업을 Assertion이라 하며, 이번 단원의 제목인 Assertions는 assertion하는 유틸리티 메소드의 집합을 말한다. 그래서 org.junit.jupiter.api.Assertions 에 assertion하는 유틸리티 메소드가 모두 들어있다. 이후로 이 클래스 안에 있는 메소드는 assertion 메소드라고 부르도록 하겠다.

assertion 메소드들은 선택적으로 마지막 매개변수에 에러 메시지를 받을 수 있다. 문자열 타입이나 문자열을 반환하는 람다로 넘길 수 있으며, 람다를 사용시에는 테스트가 실패할 때만 호출되므로 성능 최적화에 조금은 도움이 될 수 있을 것으로 보인다.

assertEquals() & assertNotEquals()

값이 같은지 / 다른지 검증할 때 사용한다.

@Test
void equalsTest() {
    assertEquals(1 + 1, 2);
}
@Test
void notEqualsTest() {
    assertNotEquals(1 + 1, 3);
}

assertNull() & assertNotNull()

값이 null인지 / null이 아닌지 검증할 때 사용한다.

@Test
void nullTest() {
    assertNull(null);
}
@Test
void notNullTest() {
    assertNotNull(1);
}

assertTrue() & assertFalse()

값이 참인지 / 거짓인지 검증할 때 사용한다.

@Test
void trueTest() {
    assertTrue(1 < 2);
}
@Test
void falseTest() {
    assertFalse(1 > 2);
}

assertSame() & assertNotSame()

두 객체가 같은 객체인지 / 다른 객체인지 검증할 때 사용한다.

@Test
void sameTest() {
    String expected = "expected";
    assertSame(expected, "expected");
}
@Test
void notSameTest() {
    String expected = "expected";
    assertNotSame(expected, new String("expected"));
}

assertArrayEquals()

두 배열의 값이 일치하는지 검증할 때 사용한다.

@Test
void arrayEqualsTest() {
    char[] expected = {'e', 'x', 'p', 'e', 'c', 't', 'e', 'd'};
    char[] actual = "expected".toCharArray();
    assertArrayEquals(expected, actual);
}

assertIterableEquals()

Iterable 구현체의 값이 같은지 검증할 때 사용한다. 같은 순서로 값을 반환해야 하며, 꼭 동일한 타입일 필요는 없다.

@Test
void iterableEqualsTest() {
    Iterable<String> expected = new ArrayList<>(Arrays.asList("Hello", "Junit", "Practice"));
    Iterable<String> actual = new LinkedHashSet<>(Arrays.asList("Hello", "Junit", "Practice"));
    assertIterableEquals(expected, actual);
}

assertLinesMatch()

리스트의 문자열이 일치하는지 검증할 때 사용하며, 각 요소마다 아래의 단계를 처음부터 수행한다.

  1. 각 요소 값이 같은지 체크한다. - 같다면, 다음 요소로 넘어간다.
  2. 같지 않다면, String.matches()로 정규 표현식 체크를 한다. - 같다면, 다음 요소로 넘어간다.
  3. 같지 않다면, fast-forward marker 인지 체크한다. - 같다면, 다음 요소로 넘어간다.
Case for step 2
@Test
void linesMatchTestCaseForStep2() {
    List<String> expected = Arrays.asList("Today's ", ".*", "\\d+");
    List<String> actual = Arrays.asList("Today's ", "lucky number is ", "72");
    assertLinesMatch(expected, actual);
}
Case for step 3
@Test
void linesMatchTestCaseForStep3() {
    List<String> expected = Arrays.asList(">>2>>", "blue");
    List<String> actual = Arrays.asList("The sky ", "is ", "blue");
    assertLinesMatch(expected, actual);
}

assertThrows()

예상된 exception의 타입을 던지는지 검증할 때 사용한다.

@Test
void throwsTest() {
    Exception exception = assertThrows(Exception.class, () -> System.out.println(1/0));
    assertEquals("/ by zero", exception.getMessage());
}

assertThrowsExactly()

예상된 exception을 던지는지 검증할 때 사용한다.

@Test
void ThrowsExactlyTest() {
    Exception exception = assertThrowsExactly(ArithmeticException.class, () -> System.out.println(1/0));
    assertEquals("/ by zero", exception.getMessage());
}

assertTimeout()

예상된 시간 값을 초과하기 전에 완료되는지 검증할 때 사용한다. 시간초과가 되더라도 테스트는 끝까지 실행하며, 얼마나 시간을 넘겼는지 에러 메시지로 보여준다.

success
import static java.time.Duration.ofMillis;
@Test
void timeoutTest() {
    String actualResult = assertTimeout(ofMillis(10), () -> "Hello world!");
    assertEquals("Hello world!", actualResult);
}
fail
import static java.time.Duration.ofMillis;
@Test
void timeoutTestExceeded() {
    assertTimeout(ofMillis(10), () -> {
        Thread.sleep(100);
    });
}
Failed output
execution exceeded timeout of 10 ms by 92 ms

assertTimeoutPreemptively()

예상된 시간 값을 초과하기 전에 완료되는지 검증할 때 사용한다. 시간초과가 되면 테스트는 즉시 실패한다.

@Test
void TimeoutPreemptivelyTest() {
    assertTimeoutPreemptively(ofMillis(10), () -> {
        Thread.sleep(100);
    });
}
Output
execution timed out after 10 ms

assertAll()

assertion 메소드는 테스트가 실패하면 예외가 발생해서 그 뒤의 코드를 실행하지 않는다. 그런데 실패와 상관없이 모든 assertion 메소드를 실행하고 싶은 경우가 있을 수 있다. 그럴 경우 이 메소드를 사용해서 assertion 메소드를 그룹화하면, 예외를 발생시키지 않아서 실패해도 즉시 중지되지 않고 오류를 집계해서 보여준다.

다만, 블랙리스트에 있는 예외 (예: Out of Memory)를 발생시키면 실행이 즉시 중지되고 unchecked exception으로 덮어 씌워진다.

@Test
void assertAllTest() {
    assertAll(
            "Group name",
            () -> assertTrue(1 < 2),
            () -> fail("Occur manually"),
            () -> assertTrue(1 > 2)
    );
}

Expected :true
Actual   :false

테스트를 강제로 실패시킨다.

failed
@Test
void assertEachTest() {
    fail("Occur manually");
}

@TestInstance

JUnit은 테스트끼리 영향을 최소화하기 위해서 각 테스트 메소드마다 새로운 테스트 인스턴스를 만든다. 하지만 필요에 따라서 매번 새로운 테스트 인스턴스를 만들고 싶지 않을 수 있다.

이 경우 클래스에 @TestInstance을 선언해서 지정한 모드에 따라 테스트 인스턴스 생성 패턴을 변경할 수 있다. 모드는 PER_METHOD 또는 PER_CLASS를 선택할 수 있다.

  • PER_METHOD: 기본 값이며 메소드마다 새로운 인스턴스를 생성하기 때문에 상태를 공유하지 않는다는 특징이 있다. @BeforeAll과 @AfterAll은 반드시 static 메소드에 선언되어야 한다.
  • PER_CLASS: 클래스마다 새로운 인스턴스를 생성하기 때문에 메소드에서 상태를 공유하는 것이 가능하다. @BeforeAll과 @AfterAll은 꼭 static 메소드에 선언될 필요는 없다.

테스트 인스턴스의 생명 주기를 이해해야 하는 이유

테스트 인스턴스의 생명주기를 모른다면 우리는 PER_CLASS 모드로 생각하기 때문에 상태를 공유할 수 있다는 전제로 코드를 작성하게 된다. 따라서 아래와 같은 경우가 발생할 수 있다.

@TestMethodOrder(MethodOrderer.OrderAnnotation.class)
class LifeCycleTest {
    int counter = 0;
    @Test
    @Order(1)
    void counterShouldBe1() {
        System.out.printf("Expected: %d, Actual: %d\n", 1, ++counter);
    }
    @Test
    @Order(2)
    void counterShouldBe2() {
        System.out.printf("Expected: %d, Actual: %d\n", 2, ++counter);
    }
}
Output
Expected: 1, Actual: 1
Expected: 2, Actual: 1

매 테스트마다 1씩 증가할 것을 기대했지만, 테스트 메소드마다 새로운 인스턴스가 생성됐기 때문에 counter는 매번 0부터 시작하게 된다.
