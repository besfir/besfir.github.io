이번 글에서는 스프링 부트를 이용해서 테스트할 때 기본적으로 알아야 할 어노테이션과, 알아두면 다른 어노테이션을 이해하는데 도움이 되는 어노테이션을 알아볼 것이다.

@ContextConfiguration

통합 테스트를 위해 ApplicationContext를 로드하고 구성하는데 사용되며, 보통 컴포넌트 클래스를 지정하기 위해 classes 속성이 주로 사용된다.

@ExtendWith(SpringExtension.class)
// ApplicationContext will be loaded from FooConfig and BarConfig
@ContextConfiguration(classes = {FooConfig.class, BarConfig.class})
class MyTest { ... }

💡 컴포넌트 클래스
✅ @Configuration 어노테이션이 달린 클래스.
✅ @Component, @Service, @Repository 또는 다른 스테레오 타입 어노테이션이 달린 클래스.
✅ jakarta.inject 어노테이션이 달린 JSR-330 호환 클래스.
✅ @Bean 메소드를 포함하고 있는 모든 클래스.
✅ 스프링 컴포넌트로 등록하려는 의도로 스프링 어노테이션 사용 없이 단일 생성자에 자동 주입되는 모든 클래스.

classes 속성을 명시적으로 선언하지 않으면, @Configuration 어노테이션이 달린 중첩 클래스를 찾고 그마저 없으면 @SpringBootConfiguration을 찾는다. 그 어노테이션은 @SpringBootApplication 내부에 선언되어 있으므로, classes 속성을 명시적으로 선언하지 않는다면 fallback으로 모든 bean을 사용한다는 것을 알 수 있다.

@TestPropertySource

통합 테스트를 위해 로드된 Application Context 환경의 PropertySource 집합에 추가할 properties 파일의 위치나 인라인 properties를 구성하는데 사용하는 어노테이션이다.

locations

*.properties 파일의 위치를 지정하는 속성이다.

declare a properties file
@ContextConfiguration
@TestPropertySource("/test.properties")
class MyPropertyTests { ... }

예제에서는 classpath 루트에 있는 test.properties 파일에서 속성들을 가져오도록 선언하고 있다.

properties

테스트를 실행하기 전에 스프링 환경에 추가해야 하는 key=value 형식의 속성이다.

declare inlined properties
@ContextConfiguration
@TestPropertySource(properties = { "timezone = GMT", "port = 4242" })
class MyPropertyTests { ... }

예제에서는 timezone과 port 속성들을 선언하고 있다.

@TestConfiguration

테스트용 추가 bean이나 커스터마이징을 선언할 때 사용한다.

자동 등록

의도치 않은 충돌을 방지하기 위해서인지 자동 등록이 되지 않으며 @Import, @SpringBootTest 또는 @ContextConfiguration 등을 통해서 수동으로 등록할 수 있다. 다만, 중첩 클래스에 사용됐을 때에는 자동 등록이 된다.

오버라이드

@Configuration을 사용할 때와 마찬가지로 Bean 이름이 겹치면 예외가 발생한다. 이를 피하기 위해서 같은 이름을 부여해서 오버라이드 하는 경우가 있다. 이 경우, 예외가 발생해 테스트가 실패할 수 있다.

아래 예제의 경우, AppConfig 클래스 파일 전체가 오버라이드 되기 때문에, bar라는 이름의 bean은 존재하지 않게 된다. 따라서 예외가 발생한다.

@SpringBootTest(classes = TestConfig.class)
class UnitTestApplicationTests {

  @Autowired
  private String foo;

  @Autowired
  private Boolean bar;    // 🔥 NoSuchBeanDefinitionException

  @Test
  void someTest() {
    assertThat(foo).isEqualTo("foo test");
    assertThat(bar).isTrue();
  }

}

@Configuration("AppConfig")
public class AppConfig {

  @Bean
  public String foo() {
        return "foo";
  }
    
  @Bean
  public Boolean bar() {
        return true;
  }
}

@TestConfiguration("AppConfig")
public class TestConfig {

  @Bean
  public String foo() {
        return "foo test";
  }
}

Bean 오버라이드를 허용하는 환경 속성을 지정하면 정상적으로 오버라이드 가능하다. 이때, classes 속성을 이용하지 않고, @Import를 이용해서 bean을 등록하면 해당 bean이 무시되니 주의하도록 한다.

@SpringBootTest(
  classes = TestConfig.class,
  properties = "spring.main.allow-bean-definition-overriding=true"
)
class UnitTestApplication2Tests {

  @Autowired
  private String foo;

  @Autowired
  private Boolean bar;

  @Test
  void someTest() {
    assertThat(foo).isEqualTo("foo test");
    assertThat(bar).isTrue();
  }

}

@Configuration
public class AppConfig {

  @Bean
  public String foo() {
    return "foo";
  }

  @Bean
  public Boolean bar() {
    return true;
  }
}

@TestConfiguration
public class TestConfig {

  @Bean
  public String foo() {
    return "foo test";
  }
}

@SpringBootTest

실제 구동되는 에플리케이션과 동일하게 Application Context를 로드하기 때문에, 개발자가 원하는 어떠한 테스트라도 수행할 수 있기 때문에 가장 강력한 테스트 어노테이션이다. 하지만 모든 bean을 로드하기 때문에 규모가 커질수록 무거워진다는 단점이 있다.

아래 예시와 같이, 앞서 살펴본 어노테이션들을 조합하면 비슷하게 작동하도록 만들 수 있다.

//@ExtendWith(SpringExtension.class)
//@ContextConfiguration
//@TestPropertySource(locations = "/application.properties")
@SpringBootTest
class MyTest { ... }

각 속성에 대해서 간단히 알아보자.

classes

@ContextConfiguration의 classes 속성과 같다.

properties

@TestPropertySource의 properties 속성과 같다.

webEnvironment

내장 웹 서버 동작 방식을 지정할 때 사용된다.

  • MOCK

기본 값이며, Web Application Context를 생성한다. 내장 톰캣을 구동하지 않고 서블릿 환경을 mocking하기 때문에, 실제 요청을 할 수 없다. 따라서 테스트를 위해 MockMvc 또는 WebTestClient를 사용하며, 이 클라이언트들을 자동으로 구성하기 위해 @AutoConfigureMockMvc와 함께 활용된다.

💡 WebTestClient는 라이브 서버와 Mock 환경 모두 사용 가능하며, 의존성에 webflux가 존재해야 사용 가능하다.

@SpringBootTest
@AutoConfigureMockMvc
class MyMockMvcTests {

  @Test
  void testWithMockMvc(@Autowired MockMvc mvc) throws Exception {
      mvc.perform(get("/")).andExpect(status().isOk()).andExpect(content().string("Hello World"));
  }

  // If Spring WebFlux is on the classpath, you can drive MVC tests with a WebTestClient
  @Test
  void testWithWebTestClient(@Autowired WebTestClient webClient) {
      webClient
              .get().uri("/")
              .exchange()
              .expectStatus().isOk()
              .expectBody(String.class).isEqualTo("Hello World");
  }
    
}
  • RANDOM_PORT

Web Application Context를 생성하며, 임의 포트로 내장 톰캣을 구동한다. @LocalServerPort를 이용해서 주입된 포트를 확인할 수 있으며, TestRestTemplate 또는 WebTestClient를 사용해서 테스트한다. 테스트 스레드와는 다른 스레드에서 스프링이 구동되기 때문에, @Transactional이 예상한 대로 동작하지 않을 수 있다.

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class UnitTestApplicationTests {

  @Autowired
  private TestRestTemplate restTemplate;

  @LocalServerPort
  private String port;

  @Test
  public void greetingShouldReturnDefaultMessage() throws Exception {
    String requestUrl = "http://localhost:" + port + "/";
    assertThat(restTemplate.getForObject(requestUrl, String.class)).contains("Hello World");
  }
  
}
  • DEFINED_PORT

지정한 포트로 내장 톰캣을 구동하는 것을 제외하면 RANDOM_PORT와 특징이 같다.

@SpringBootTest(
  webEnvironment = SpringBootTest.WebEnvironment.DEFINED_PORT,
  properties = "server.port=49719"
)
class UnitTestApplicationTests {
  
  @Autowired
  private TestRestTemplate restTemplate;

  @LocalServerPort
  private String port;

  @Test
  void envPropertyTest() {
    assertThat(port).isEqualTo("49719");
  }
  
  @Test
  public void greetingShouldReturnDefaultMessage() throws Exception {
    String requestUrl = "http://localhost:" + port + "/";
    assertThat(restTemplate.getForObject(requestUrl, String.class)).contains("Hello World");
  }
  
}
  • NONE

웹 환경을 구성하지 않고 Application Context를 생성하며, 내장 톰캣을 구동하지 않는다.

useMainMethod

아래와 같이 메인 메소드에서 어떤 설정을 사용할 경우, 테스트에서도 해당 설정을 사용할지 선택할 때 사용된다.

@SpringBootApplication
public class UnitTestApplication {

    public static void main(String[] args) {
      SpringApplication application = new SpringApplication(UnitTestApplication.class);
      application.setBannerMode(Banner.Mode.OFF);
      application.setAdditionalProfiles("someProfile");
      application.run(args);
    }

}
  • ALWAYS

항상 메인 메소드를 실행한다. @SpringBootConfiguration 어노테이션이 달린 클래스가 없거나 해당 클래스에 메인 메소드가 없다면 실행에 실패한다.

  • NEVER

절대 메인 메소드를 실행하지 않으며, 테스트 특화 SpringApplication을 생성한다.

  • WHEN_AVAILABLE

@SpringBootConfiguration 어노테이션이 달린 클래스가 없거나 해당 클래스에 메인 메소드가 없다면, 테스트 특화 SpringApplication을 생성한다.

args

테스트 중인 응용 프로그램에 전달해야 하는 인수를 선언하는 속성이다. 인수를 사용하려면 ApplicationArguments를 주입받으면 된다.

@SpringBootTest(args = {"--app.foo=bar"})
class UnitTestApplicationTests {

  @Autowired
  private ApplicationArguments args;

  @Test
  void argumentTest() {
    assertThat(args.getOptionNames()).containsOnly("app.foo");
    assertThat(args.getOptionValues("app.foo")).containsOnly("bar");
  }
}

@MockBean & @SpyBean

이전 글에서 언급한 Mock과 Spy의 쓰임새와 동일하다. 다른점은 Mock과 Spy가 bean을 대체해서 Application Context에 등록된다는 점이다. 따라서, 해당 bean이 사용되었어야 할 모든 자리에, mock 객체가 대신하게 된다.

@SpringBootTest
class UnitTestApplicationTests {

  @Autowired
  private ApplicationContext ac;

  @MockBean
  private FooRepository repository;

  @Autowired
  private FooService service;
  
  @BeforeEach
  void setUp() {
    when(repository.get()).thenReturn("mock foo");
  }

  @Test
  void mockBeanFromContextTest() {
    FooRepository componentFromContext = ac.getBean(FooRepository.class);

    assertThat(componentFromContext.get()).isEqualTo("mock foo");
    verify(repository).get();
  }

  @Test
  void mockBeanTest() {
    assertThat(service.show()).isEqualTo("mock foo");
    verify(repository).get();
  }

  @TestConfiguration
  @Import(FooService.class) // A FooService injected with mocked FooRepository
  static class Config { }
}

@Component
public class FooRepository {

  public String get() {
    return "foo";
  }
}

@Service
@RequiredArgsConstructor
public class FooService {

  private final FooRepository repository;

  public String show() {
    return repository.get();
  }
}

Mock 또는 Spy는 이름이나 타입별로 등록될 수 있다. 특정 타입의 bean이 하나라면 이름을 지정할 필요없이 타입으로 등록되지만, 둘 이상이라면 name 속성을 이용해서 명시적으로 이름을 지정해야한댜.

@SpringBootTest
class UnitTestApplicationTests {

  @Autowired
  private ApplicationContext ac;

  @MockBean(name = "fooRepository")
  private FooRepository repository;

  @BeforeEach
  void setUp() {
    when(repository.get()).thenReturn("mock foo");
  }


  @Test
  void mockBeanFromContextTest() {

    FooRepository componentFromContext = ac.getBean("fooRepository", FooRepository.class);

    assertThat(componentFromContext.get()).isEqualTo("mock foo");
    verify(repository).get();
  }

}

@Configuration
public class AppConfig {

  ...

  @Bean
  public FooRepository foo2() {
    return new FooRepository();
  }
}

Mocking 하려는 클래스가 bean으로 등록되지 않은 경우, 알아서 등록을 해준다.

@SpringBootTest
class UnitTestApplicationTests {

  @MockBean
  SomeComponent someComponent;

  @Test
  void someComponentTest() {
    when(someComponent.doSomething()).thenReturn("world");

    assertThat(someComponent.doSomething()).isEqualTo("world");
  }
}

public class SomeComponent {
  
  public String  doSomething() {
    return "hello";
  }
}

@Import

컴포넌트 클래스를 등록하기 위해 사용한다. @WebMvcTest, @DataJpaTest 등과 같이 컴포넌트 클래스 등록을 지원하지 않는 어노테이션과 조합해서 사용한다.

굳이 그럴 일은 없겠지만 @SpringBootTest와 같이 사용하면 안된다.@Import의 우선 순위가 더 높기 때문에 먼저 bean으로 등록되고, 그 이후 @SpringBootTest를 통해 로드된 컴포넌트 클래스가 bean으로 등록된다. 따라서 @Import에 지정한 컴포넌트 클래스가 무시될 수 있다.

이해를 돕기 위해 아래 예제를 보자. foo = foo test를 기대하며 bean 오버라이드를 했지만 실제 결과는 foo = foo이다.

@SpringBootTest( properties = "spring.main.allow-bean-definition-overriding=true")
@Import({TestConfig.class})
class UnitTestApplication2Tests {

  @Autowired
  private String foo;

  @Autowired
  private Boolean bar;

  @Test
  void someTest() {
    assertThat(foo).isEqualTo("foo");
    assertThat(bar).isTrue();
  }
}

@Configuration
public class AppConfig {

  @Bean
  public String foo() {
    return "foo";
  }

  @Bean
  public Boolean bar() {
    return true;
  }
}

@TestConfiguration
public class TestConfig {

  @Bean
  public String foo() {
    return "foo test";
  }

}
