데이터베이스 설계가 너무도 어려워 정리를 하려고 보니, 필자 스스로 데이터베이스 기초를 전혀 모른다는 것을 깨닳았다. 따라서 데이터베이스 설계 시리즈를 작성하기에 앞서, 데이터베이스 시작하기 시리즈에 대해서 정리해보려 한다. 이번 글에서는 데이터베이스에 입문하기 위해 필요한 정보를 정리해 보겠다.

Database

일반적으로, 관련있는 데이터를 전자적으로 저장하고 사용하는 조직화 된 집합을 말한다.

관련 있는 데이터

소셜 네트워크 서비스로 예를 들어 보자. 사용자들은 사진을 올리고 글을 작성하며, 다른 사용자들의 콘텐츠에 댓글을 다는 등 다양한 정보를 생성한다. 이렇듯 같은 서비스 내에서 동일한 출처와 목적을 가진 데이터들을 관련 있는 데이터라고 한다.

조직화 된 집합

책이 바닥에 어질러져 있다고 가정해 보자. 이러한 상태에서는 책을 빠르게 찾기 어렵고, 동일한 책을 중복으로 소유하고 있는지 확인하기도 어려울 것이다. 하지만 이 책들을 정리해서 책장에 꽂아 놓는다면, 원하는 책을 빠르게 찾을 수 있고, 불필요한 중복 소유 여부도 쉽게 알 수 있다. 이와 같이 데이터를 체계화하여 효율성과 데이터 접근성을 높인 집합을 조직화 된 집합이라 한다.

전자적으로 저장하고 사용

데이터를 효율적으로 저장하고, 빠르게 검색 가능한 컴퓨터에 보관하는 것을 의미한다. 이를 통해, 작은 공간을 차지하면서도 빠른 연산과 검색이 가능해진다.

DBMS

사용자에게 데이터베이스를 정의하고 만들고 관리하는 기능을 제공하는 소프트웨어로서, 주로 PostgreSQL, MySQL, Oracle, MsSQL 등이 대표적이다. 데이터베이스를 정의하는 과정에서 데이터의 구조를 결정하게 되는데, 이 과정에서 부가적인 데이터가 발생하며 이러한 부가 데이터를 metadata라고 한다.

Metadata

해당 용어는 다양한 분야에서 널리 사용되며, 원래의 의미는 "data about data"로, 다시 말해 데이터를 설명하거나 특성을 기술하는 데이터를 가리킨다. 예를 들어, 사진을 찍으면 해당 사진이 촬영된 날짜, 파일 형식, 해상도 등과 같이 사진을 설명하는 추가 정보를 볼 수 있는데, 이러한 추가 정보를 metadata라고 한다.

여기서는 데이터베이스를 정의하거나 설명하는 데이터를 지칭하는데, 다른 용어로 카탈로그(catalog)라고 부르기도 한다. 카탈로그는 메타데이터 자체를 의미하기도 하지만 메타데이터가 저장되는 장소를 가리키기도 하며, DBMS를 통해 저장, 관리 된다.

메타데이터의 예로는 데이터 유형, 구조, 제약 조건, 보안, 저장, 인덱스, 사용자 그룹 등이 있다.

Database system

Database + DBMS + 연관된 applications을 가리킨다. 그런데, 여러 문서를 읽다보면 database system을 줄여서 그냥 database라 부르는 경우도 있다. 따라서 해당 문서의 database가 순수한 데이터를 의미하는 것인지 database system을 의미하는 것인지 문맥을 잘 파악해서 의미를 이해해야 한다.

작동 흐름

  • Users / Programmers : 사용자 또는 프로그래머가 어떤 애플리케이션을 이용한다.
  • Application Programs / Queries : 해당 애플리케이션은 사용자가 요청한 어떤 기능을 수행하기 위해 DBMS에 쿼리를 보낸다. 여기서 말하는 쿼리는 데이터베이스로부터 원하는 데이터를 가져오거나 원하는 형태로 수정 또는 삭제하도록 하는 요청을 말한다.
  • Software to Process Queries / Programs : 해당 쿼리가 무엇을 의미하는지 분석을 한다.
  • Software to Access Stored Data : 무엇을 요청한 것인지 파악이 되면 해당 요청을 처리한다.
    • Stored Database Definition : 메타데이터를 알아야 데이터를 처리할 수 있기 때문에, 먼저 메타데이터를 찾아서 확인한다.
    • Stored Database : 메타데이터를 바탕으로 원하는 데이터를 처리한 후 결과를 애플리케이션에 반환한다.

Data Models

데이터베이스의 구조를 설명하는데 사용될 수 있는 개념들의 집합이다. 그 개념들을 바탕으로 데이터베이스의 구조를 추상화하여 원하는 형태로 표현할 수 있는 방법을 제공한다. 데이터베이스에서 읽고 쓰기 위한 기본적인 동작들도 데이터 모델에 포함된다. 언급된 데이터베이스 구조는 데이터 유형, 데이터 사이의 관계, 제약 사항 등을 포함하는 개념을 말한다.

데이터 모델은 여러 종류가 있으며, 각각은 추상화 수준과 데이터베이스 구조화 방식이 약간씩 다르다.

Conceptual (or high-level) data models

일반 사용자들도 쉽게 이해할 수 있는 개념들로 이뤄진 모델로, 추상화 수준이 가장 높다. 이 모델은 비지니스 요구 사항을 추상화하여 설명할 때 사용한다. 대표적인 데이터 모델로 Entity-Relationship Model이 있으며, 이를 시각적으로 표현한 것이 Entity-Relationship Diagram 줄여서 ER Diagram이다.

아래의 그림은 대학교 도서관 시스템을 구축한다고 가정했을 때, 학생과 도서의 관계를 ER Diagram으로 나타낸 것이다.

logical (or representational) data models

이해하기 어렵지 않으면서도 데이터베이스를 상세하게 구조화할 수 있는 개념들을 제공한다. 데이터베이스를 상세하게 구조화 한다는 것은 논리 데이터 모델(logical data model)이 컴퓨터에 저장된 실제 데이터 모델의 구조와 크게 다르지 않도록 구조화 하는 것을 의미한다. 다만, 이 모델은 실제 저장장치에 데이터가 저장되는 방식까지 나타내는 것이 아닌, 특정 DBMS나 스토리지에 종속되지 않는 수준에서 데이터베이스를 구조화 할 수 있는 모델이다.

대표적인 데이터 모델로는 관계형 데이터 모델(relational data model)이 있으며, 그밖에 객체 데이터 모델(object data model)과 객체-관계형 데이터 모델(object-relational data model)이 있다. 앞서 언급한 주요 DBMS 중에서 Oracle, MySQL, MsSQL은 관계형 데이터 모델에 기반하고 있다. 그리고 PostgreSQL은 객체-관계형 데이터 모델에 기반하고 있지만 관계형 데이터 모델을 포함하므로, 주요 DBMS들은 관계형 데이터 모델에 기반하고 있다 할 수 있겠다.

데이터베이스 시작하기 시리즈에서는 이 관계형 데이터 모델을 집중적으로 알아볼 것이다.

위 그림에서 볼 수 있듯 관계(relation)는 쉽게 말해 테이블을 의미한다. 테이블에서 행(row)은 하나의 데이터를 의미하며 튜플(tuple) 또는 레코드(record)라고 부른다. 또한 열(column)은 속성(attribute)이라 부르며 튜플 또는 레코드의 특성을 의미한다.

physical (or low-level) data models

컴퓨터에 데이터가 어떻게 파일 형태로 저장되는지를 설명할 수 있는 방법을 제공한다. 바꿔 말해, data format, data orderings, access path 등의 요소를 표현하여, 데이터베이스 구조를 실제 저장 장치에 데이터가 저장되는 방식에 최대한 가깝게 표현하는 데이터 모델입니다. 이 중 access path는 데이터 검색을 빠르게 하기 위한 구조체로, 대표적으로 인덱스가 이에 해당된다.
