💡 테스트 대역에 관한 용어가 익숙치 않다면 Spy, Stub, Fake 그리고 Mock에 대해서 알아보자를 읽어보길 바란다.

Mockito는 mocking framework로, 개발자가 제어할 수 없는 상황을 통제하기 위해서 Mock 객체를 생성하는 용도로 사용한다. 다만, 코드 설계를 바꾸는 등의 노력을 통해서 Mock 객체를 사용하지 않을 수 있다면, 그렇게 하는것이 좋은 코드를 작성하는데 도움이 될 수 있다.

Mock 객체 생성

Mockito.mock()을 사용하면 Mock 객체를 생성할 수 있으며, 별다른 설정을 하지 않는다면 각 타입에 따라 기본 값을 반환한다. 기본값을 정리해보자면 다음과 같다.

  • 리턴 값이 void라면 아무 일도 발생하지 않는다.
  • 리턴 값이 원시 타입이라면 원시 타입 기본값을 반환한다.
  • 리턴 값이 객체 타입이라면 null을 반환한다.
  • 리턴 값이 Optional 타입이라면 Optional.empty()를 반환한다.
  • 리턴 값이 컬렉션 타입이라면 빈 컬렉션을 반환한다.
class MockBasicTest {
  @Test
  void createMockTest() {
    var mock = Mockito.mock(DefaultValueTest.class);

    assertDoesNotThrow(mock::invokeVoidMethod);
    assertEquals(0, mock.getPrimitiveValue());
    assertEquals(Collections.emptyList(), mock.getCollectionValue());
    assertEquals(Optional.empty(), mock.getOptionalValue());
    assertNull(mock.getObjectValue());
  }
}

class DefaultValueTest {
  public void voidMethod() {

  }

  public int primitiveReturn() {
    return 10;
  }

  public String objectReturn() {
    return "string";
  }

  public Optional<Double> optionalReturn() {
    return Optional.of(3.14159);
  }

  public List<Character> collectionReturn() {
    return List.of('1','2');
  }

}

어노테이션 방식이 편하다면, 이를 이용할 수도 있다.

@ExtendWith(MockitoExtension.class)
class MockitoBasicTest {
  
  @Test
  void createMockTest(@Mock DefaultValueTest mock) {

    assertDoesNotThrow(mock::invokeVoidMethod);
    assertEquals(0, mock.getPrimitiveValue());
    assertEquals(Collections.emptyList(), mock.getCollectionValue());
    assertEquals(Optional.empty(), mock.getOptionalValue());
    assertNull(mock.getObjectValue());
  }
}

테스트 간 공유하도록 필드에 선언해서 중복을 최소화 할 수도 있다.

@ExtendWith(MockitoExtension.class)
class MockitoBasicTest {

  @Mock
  DefaultValueTest mock;
  
  @Test
  void createMockTest() {

    assertDoesNotThrow(mock::invokeVoidMethod);
    assertEquals(0, mock.getPrimitiveValue());
    assertEquals(Collections.emptyList(), mock.getCollectionValue());
    assertEquals(Optional.empty(), mock.getOptionalValue());
    assertNull(mock.getObjectValue());
  }
}

Spy 객체 생성

Mock 객체 생성했던 것과 비슷한 방법으로 Spy 객체도 생성할 수 있다. 테스트 대역 관련 글에서 언급했던 것처럼, 별다른 설정을 하지 않는다면 실제 객체처럼 동작한다.

@ExtendWith(MockitoExtension.class)
class MockitoBasicTest {

  @Test
  void createSpyTest() {
    DefaultValueTest spy = Mockito.spy(DefaultValueTest.class);

    assertDoesNotThrow(spy::invokeVoidMethod);
    assertEquals(10, spy.getPrimitiveValue());
    assertEquals(List.of('1','2'), spy.getCollectionValue());
    assertEquals(Optional.of(3.14159), spy.getOptionalValue());
    assertNotNull(spy.getObjectValue());
  }
}

어노테이션 이용시 필드에만 선언이 가능하다.

@ExtendWith(MockitoExtension.class)
class MockitoBasicTest {

  @Spy
  DefaultValueTest spy;
  
  @Test
  void createSpyTest() {
    assertDoesNotThrow(spy::invokeVoidMethod);
    assertEquals(10, spy.getPrimitiveValue());
    assertEquals(List.of('1','2'), spy.getCollectionValue());
    assertEquals(Optional.of(3.14159), spy.getOptionalValue());
    assertNotNull(spy.getObjectValue());
  }
}

Stubbing

Mock 객체가 지정한 값을 반환할 수 있도록 stubbing 하는 방법을 알아보겠다.

Without argument

조건을 지정하는 when()과 동작을 지정하는 thenXX()를 조합해서 stubbing 할 수 있다.

💡 객체 생성 단원에서 사용하던 예제를 그대로 사용하기 때문에 중복되는 부분은 제외하도록 하겠다.

  • thenReturn() : 조건에 들어가는 메소드의 반환 타입에 맞춰 리터럴 값을 지정할 수 있다.
@Test
void thenReturnTest() {
  Mockito.when(mock.getObjectValue()).thenReturn("Hello world");

  assertEquals(mock.getObjectValue(), "Hello world");
}
  • thenThrow() : 조건에 맞는 메소드를 실행하면 예외를 반환한다.
@Test
void thenThrowTest(){
  Mockito.when(mock.getOptionalValue()).thenThrow(new IllegalStateException());

  assertThrows(IllegalStateException.class, mock::getOptionalValue);
}
  • then() & thenAnswer() : 조건에 맞는 메소드를 실행하면 콜백을 실행한다.
@Test
void thenAnswerTest(){
  Mockito.when(mock.getCollectionValue()).thenAnswer((invocation)->{
    Object[] args = invocation.getArguments();
    Object mock = invocation.getMock();
    System.out.println("called with arguments: " + Arrays.toString(args));
    return List.of('t','e','s','t');
  });

  assertEquals(mock.getCollectionValue(), List.of('t','e','s','t'));
}
  • thenCallRealMethod() : 조건에 맞는 메소드를 실행하면 실제 메소드의 실행값을 반환한다.
@Test
void thenCallRealMethodTest(){
  Mockito.when(mock.getCollectionValue()).thenCallRealMethod();

  assertEquals(mock.getCollectionValue(), List.of('1','2'));
}

With argument

인수를 포함하지 않을 때와 stubbing 하는 방법은 동일하나, 지정한 리터럴 값 인수나 인수 타입만 stubbing 되는 차이가 있다.

  • 리터럴 값 사용시
    해당 리터럴 값 인수만 stubbing 되며, 다른 인수를 이용하면 위에서 언급했던 것처럼 기본값을 반환한다.
@Test
void stubWithLiteralArgumentTest() {
  List mock = Mockito.mock(List.class);
  when(mock.get(0)).thenReturn(10);

  assertEquals(10, mock.get(0));
  assertNull(mock.get(1));
}
  • ArgumentMatchers 사용시
    인수로 넘길 타입을 지정할 수 있다. Matchers 이름이 직관적이라서 굳이 설명이 필요하지 않기 때문에 사용 예제만 첨부한다.
...
import static org.mockito.ArgumentMatchers.*;
import static org.mockito.Mockito.when;


@Test
void stubWithArgumentMatchersTest() {
  Map mock = Mockito.mock(Map.class);

  when(mock.get(isNull())).thenReturn(0);
  when(mock.get(anyBoolean())).thenReturn(1);
  when(mock.get(anyInt())).thenReturn(10);
  when(mock.get(anyDouble())).thenReturn(20);
  when(mock.get(anyString())).thenReturn(30);
  when(mock.get(anyMap())).thenReturn(100);
  when(mock.get(anyCollection())).thenReturn(200);
  when(mock.get(any(Character[].class))).thenReturn(300);
  when(mock.get(any(DefaultValueTest.class))).thenReturn(1000);

  assertEquals(0, mock.get(null));
  assertEquals(1, mock.get(false));
  assertEquals(10, mock.get(1234));
  assertEquals(20, mock.get(3.14));
  assertEquals(30, mock.get("string"));
  assertEquals(100, mock.get(Map.of()));
  assertEquals(200, mock.get(List.of()));
  assertEquals(300, mock.get(new Character[]{'A'}));
  assertEquals(1000, mock.get(new DefaultValueTest()));
}

@Test
void stubWithAnyArgumentTest() {
  Map mock = Mockito.mock(Map.class);

  when(mock.get(any())).thenReturn(0);

  assertEquals(0, mock.get(null));
  assertEquals(0, mock.get(false));
  assertEquals(0, mock.get(1234));
  assertEquals(0, mock.get(3.14));
  assertEquals(0, mock.get("string"));
  assertEquals(0, mock.get(Map.of()));
  assertEquals(0, mock.get(List.of()));
  assertEquals(0, mock.get(new Character[]{'A'}));
  assertEquals(0, mock.get(new DefaultValueTest()));
}
  • 리터럴 값과 ArgumentMatchers를 같이 사용시
    Matchers와 리터럴 값을 같이 적용하면 matchers가 우선순위를 가진다.
  @Test
  void stubWithArgumentTest() {
    List mock = Mockito.mock(List.class);
    lenient().when(mock.get(0)).thenReturn(10);
    lenient().when(mock.get(1)).thenReturn(100);
    when(mock.get(anyInt())).thenReturn(1000);

    assertEquals(1000, mock.get(0));
    assertEquals(1000, mock.get(1));
    assertEquals(1000, mock.get(2));
  }

💡 lenient()는 느슨한 규칙을 적용하는 설정으로, 지정된 Stub이 사용되지 않아도 에러를 발생시키지 않는다.

위에서 언급한 matchers 외에도 다양한 matchers가 있다. 이에 대해 알고 싶다면 ArgumentMatchers javadoc을 참고하길 바란다.

반환 타입이 void인 경우

예외를 발생시키는 stubbing만 가능하다. 반환 값을 stubbing 가능하지만 메소드 반환 타입이 void이기 때문에 컴파일 에러가 발생하기 때문이다.

@ExtendWith(MockitoExtension.class)
class MockitoBasicTest {

  @Mock
  DefaultValueTest mock;

  @Test
  void stubVoidThrowRuntimeExceptionTest() {
    Mockito.doThrow(new IllegalArgumentException()).when(mock).invokeVoidMethod();

    assertThrows(IllegalArgumentException.class, () -> mock.invokeVoidMethod());
  }
}

Checked Exception을 발생시키려면 지정한 메소드에 throws가 선언되어 있어야한다.

class DefaultValueTest {
  public void invokeVoidMethodThrow() throws IllegalAccessException { }

  ...
}

@ExtendWith(MockitoExtension.class)
class MockitoBasicTest {

  @Mock
  DefaultValueTest mock;
  
  @Test
  void stubVoidThrowCheckedExceptionTest() throws IllegalAccessException {
    Mockito.doThrow(new IllegalAccessException()).when(mock).invokeVoidMethodThrow();

    assertThrows(IllegalAccessException.class, () -> mock.invokeVoidMethodThrow());
  }
  ...
}

Consecutive calls

동일 메소드를 같은 인수로 호출할 때 stubbing 하는 방법을 알아보자.

  • 가변인자 사용
    같은 동작을 하고 리터럴 값만 다르다면 가변인자를 사용해서 stubbing 할 수 있다.
@Test
void ConsecutiveCallsWithVarargTest() {
  when(mock.getPrimitiveValue()).thenReturn(0, 10, 200);

  assertEquals(0,mock.getPrimitiveValue());
  assertEquals(10,mock.getPrimitiveValue());
  assertEquals(200,mock.getPrimitiveValue());
}
  • 체이닝 사용
    여러 동작을 stubbing 할 때에는 체이닝을 하면 된다.
@Test
void ConsecutiveCallsWithChainingTest() {
  when(mock.getPrimitiveValue())
    .thenReturn(0, 5)
    .thenThrow(new IllegalArgumentException())
    .thenCallRealMethod();

  assertEquals(0, mock.getPrimitiveValue());
  assertEquals(5, mock.getPrimitiveValue());
  assertThrows(IllegalArgumentException.class, () -> mock.getPrimitiveValue());
  assertEquals(10, mock.getPrimitiveValue());
}
  • ⛔️ 주의
    동일한 matchers 또는 인수를 가진 메소드를 여러번 stubbing 하는 경우, 각 stubbing은 이전 stubbing을 덮어쓴다.
@Test
void ConsecutiveCallsWithMultipleStubTest() {
  when(mock.getPrimitiveValue()).thenReturn(0);
  when(mock.getPrimitiveValue()).thenReturn(10);
  when(mock.getPrimitiveValue()).thenReturn(20);

  assertEquals(20, mock.getPrimitiveValue());
  assertEquals(20, mock.getPrimitiveValue());
  assertEquals(20, mock.getPrimitiveValue());
}

thenXXX() vs doXXX()

위에서 살펴봤던 stubbing 메소드 thenReturn(), thenThrow(), thenAnswer() 그리고 thenCallRealMethod()는 doReturn()처럼 do로 시작하는 이름의 메소드가 있다. 이후 thenXXX()와 doXXX()라고 부르도록 하겠다.

doXXX()는 인수로 Object 타입을 넘기므로 런타임에 타입 체크를 하기 때문에 타입 안전성이 떨어진다. 이 때문에 가능하면 thenXXX()를 사용하는 것이 좋다. 그렇다면 doXXX()는 언제 사용하기 위해 만들어졌을까?

  • void 리턴 타입을 stubbing 할 때
    사실 바로 윗 단원에서 void를 stubbing 할때 doThrow()를 사용했었다. 반환 타입이 void인 메소드는 thenXXX()를 사용하면 컴파일 에러가 발생하므로 doXXX()를 사용한다.
  • Spy 객체 사용시
    Stubbing 할 때 실제 기능이 호출되면서 예상치 못한 동작, 즉 사이드 이펙트가 발생하기 때문에 이를 예방하기 위해 사용한다.
@Spy
ArrayList spy;

@Test
void spyCauseIndexOutOfBoundsException() {
  when(spy.get(0)).thenReturn("foo");    // IndexOutOfBoundsException

  assertEquals("foo", spy.get(0));
}

@Test
void spyReturnString() {
  doReturn("foo").when(spy).get(0);
  
  assertEquals("foo", spy.get(0));
}

검증하기

지정한 메소드가 실행 됐는지 확인하기 위해 verify()를 사용하며, VerificationMode와 조합해서 사용한다.

  • times(): 지정한 메소드가 몇 번 호출됐는지 확인한다. 인자를 넘기지 않으면 기본값인 times(1)을 사용한다.
@Test
void verifyTest() {
  mock.getPrimitiveValue();

  verify(mock, times(1)).getPrimitiveValue();
}
  • atLeastOnce(),atMostOnce(): 적어도/최대 한 번 호출됐는지 확인한다.
  • atLeast(), atMost(): 최소/최대로 지정한 수만큼만 호출됐는지 확인한다.
  • never(): 지정한 메소드가 호출되지 않았는지 확인한다.
@Test
void verifyWithNeverTest() {
  mock.getPrimitiveValue();

  verify(mock, never()).invokeVoidMethod();
}
  • verifyNoInteractions(): 지정한 Mock 객체가 어떠한 상호작용도 하지 않았는지 확인한다.
@Test
void verifyWithVerifyNoInteractionsTest() {
  ArrayList arrayListMock = mock(ArrayList.class);
  arrayListMock.get(0);
  arrayListMock.get(1);

  HashMap hashMapMock = mock(HashMap.class);
  verifyNoInteractions(hashMapMock);
}
  • inOrder(): 순서대로 호출 되었는지 확인한다.
@Test
void verifyInOrderTest() {
  mock.getPrimitiveValue();
  mock.getOptionalValue();

  InOrder inOrder = inOrder(mock);
  inOrder.verify(mock, times(1)).getOptionalValue();
  inOrder.verify(mock, times(1)).getPrimitiveValue();
}

💡 같은 메소드를 호출했더라도, 반드시 순서에 맞춰서 확인해야 한다.

@Test
void verifyInOrderTest() {
  mock.getPrimitiveValue();
  mock.getPrimitiveValue();
  mock.getOptionalValue();
  mock.getPrimitiveValue();

  InOrder inOrder = inOrder(mock);
  inOrder.verify(mock, times(2)).getPrimitiveValue();
  inOrder.verify(mock, times(1)).getOptionalValue();
  inOrder.verify(mock, times(1)).getPrimitiveValue();
}

Verification mode에 대해 정보가 더 필요하다면 Verification mode javadoc을 참고하길 바란다.

@InjectMocks

Mock 객체와 spy 객체를 모두 주입하도록 만드는 어노테이션이다.

@ExtendWith(MockitoExtension.class)
class MemberTest {

  @Mock
  MemberRepository mockMemberRepository;

  @Spy
  Notification spyNotification;

  @InjectMocks
  MemberService service;

  @Test
  void registerMemberTest() {
    // Given
    Member member = generateMember();
    when(mockMemberRepository.findByEmail("test@test.com")).thenReturn(Optional.empty());
    when(mockMemberRepository.save(any(Member.class))).thenReturn(member);

    // When
    service.registerMember("test@test.com", "password");

    // Then
    verify(mockNotification).trigger(any(Member.class));
  }

  @Test
  void registerMemberThrowTest() {
    // Given
    Member member = generateMember();
    when(mockMemberRepository.findByEmail("test@test.com")).thenReturn(Optional.of(member));

    // When
    IllegalArgumentException exception = assertThrows(IllegalArgumentException.class, () -> service.registerMember("test@test.com", "password"));

    // Then
    assertEquals("This member is already registered.", exception.getMessage());
  }

  private Member generateMember() {
    return new Member("test@test.com", "password");
  }
}
public class MemberService {
  private final MemberRepository repository;
  private final Notification notification;

  public MemberService(MemberRepository repository, Notification notification) {
    this.repository = repository;
    this.notification = notification;
  }

  public Member registerMember(String email, String password) {
    Optional<Member> existMember = repository.findByEmail(email);
    if (existMember.isPresent()) {
      throw new IllegalArgumentException("This member is already registered.");
    }

    Member savedMember = repository.save(new Member(email, password));
    notification.trigger(savedMember);

    return savedMember;
  }
}

public class Member {
  private String email;

  private String password;

  public Member(String email, String password) {
    this.email = email;
    this.password = password;
  }

  public String getPassword() {
    return password;
  }

  public String getEmail() {
    return email;
  }
}

public interface MemberRepository {
  Optional<Member> findByEmail(String email);

  Member save(Member member);
}

public interface Notification {
  void trigger(Member savedMember);
}
